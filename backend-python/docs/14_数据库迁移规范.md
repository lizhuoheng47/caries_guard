# 数据库迁移规范 (Database Migration Standard)

> CariesGuard Python 端数据库 Schema 变更的唯一合法路径是 **Alembic 迁移**。
> 任何直接操作数据库的 DDL 或修改 `ensure_schema()` 均被视为违规操作。

---

## 1. 总体原则

| 原则 | 说明 |
|------|------|
| **版本可追溯** | 每次 schema 变更对应一个 Alembic migration 文件，有完整的 upgrade/downgrade |
| **代码即真相** | ORM 模型 (`app/models/*.py`) 是 schema 的唯一权威定义，迁移文件必须与之对齐 |
| **环境一致** | 本地/Docker/CI/Staging/Prod 通过同一套迁移脚本收敛到一致的 schema 状态 |
| **向前兼容** | 新迁移不得破坏已有数据；删列/改类型等破坏性变更需额外审批 |

---

## 2. 新增字段/表/索引的标准流程 (6 步)

### Step 1 — 更新基线文档

在 `docs/13_caries_ai_baseline_schema.md` 中补充新字段/表/索引的定义，保持与实际数据库同步。

### Step 2 — 修改 ORM 模型

在 `app/models/` 对应的域文件中新增/修改字段声明：

```python
# app/models/rag.py
class KnowledgeDocument(Base, TimestampMixin, SoftDeleteMixin, ...):
    # 新增字段示例
    tags: Mapped[str | None] = mapped_column(String(500), nullable=True, comment="文档标签")
```

### Step 3 — 生成迁移文件

```bash
cd backend-python
alembic revision -m "add_kb_document_tags"
```

> ⚠️ **不使用 `--autogenerate`**（Phase 4 文档 §6.3 明确禁止）。手写迁移内容以确保精确控制。

编辑生成的迁移文件：

```python
def upgrade() -> None:
    op.add_column("kb_document", sa.Column("tags", sa.String(length=500), nullable=True, comment="文档标签"))

def downgrade() -> None:
    op.drop_column("kb_document", "tags")
```

### Step 4 — 本地验证

```bash
alembic upgrade head
```

确保迁移可以在本地数据库上无错执行。

### Step 5 — 运行测试

```bash
pytest tests/ -v
```

所有单测必须通过，包括 `test_models_schema.py` 的反射对账测试。

### Step 6 — 提交

将以下文件一并提交 PR：
- `docs/13_caries_ai_baseline_schema.md`（基线文档更新）
- `app/models/*.py`（ORM 模型修改）
- `migrations/versions/NNNN_*.py`（迁移文件）

## 3. 迁移文件命名规范

### 格式

```
NNNN_动词_对象_细节.py
```

### 示例

| 文件名 | 说明 |
|--------|------|
| `0001_baseline_caries_ai.py` | 基线迁移 (已存在) |
| `0002_add_kb_document_tags.py` | 给 kb_document 表新增 tags 字段 |
| `0003_create_audit_trail.py` | 新建审计轨迹表 |
| `0004_add_index_rag_session_patient.py` | 给 rag_session 表新增索引 |
| `0005_alter_error_message_to_text.py` | 修改字段类型 |

### 编号

- 使用 4 位数字序号，从 `0001` 开始递增
- 在 `alembic revision` 生成后手动重命名文件以符合规范

---

## 4. 禁止事项

| 禁止操作 | 原因 |
|----------|------|
| 直接在 MySQL 中执行 DDL (ALTER TABLE、CREATE INDEX 等) | 绕过版本控制，导致环境不一致 |
| 修改 ORM 模型却不生成对应的迁移文件 | 代码与数据库 schema 断裂 |
| 修改 `ensure_schema()` 中的 DDL 语句 | 该方法已废弃，仅作应急保留 |
| 使用 `alembic revision --autogenerate` | 自动生成的迁移可能不够精确 |
| 迁移文件中 `downgrade()` 留空或 `pass` | 必须实现可执行的回滚逻辑 |

---

## 5. downgrade 要求

- 每个迁移文件的 `downgrade()` **必须可执行**
- 在开发环境中实际验证回滚：

```bash
# 回滚到上一个版本
alembic downgrade -1

# 再升级回来
alembic upgrade head
```

- 含有破坏性变更的迁移（删表、删列、改类型）需在 PR 描述中特别注明

---

## 6. 环境策略

| 环境 | Schema 管理方式 |
|------|----------------|
| **本地开发 (裸跑)** | `CG_DB_SCHEMA_BOOTSTRAP_ENABLED=true`（默认），`ensure_schema()` 仍可用作应急建表 |
| **Docker / CI** | `CG_DB_SCHEMA_BOOTSTRAP_ENABLED=false`，`CG_DB_MIGRATION_ENABLED=true`。entrypoint 脚本自动执行 `alembic upgrade head` |
| **Staging / Prod** | 同 Docker。首次部署已有库使用 `alembic stamp head` 注入基线标记 |

### 已有库的基线注入

对于已存在 20 张业务表但没有 `alembic_version` 表的环境：

```bash
# 只打标记，不实际执行 DDL
alembic stamp 0001
```

## 7. 迁移文件编写规范

### 7.1 表引擎与字符集

所有新表必须使用统一的表级参数：

```python
TABLE_KWARGS = {
    "mysql_engine": "InnoDB",
    "mysql_charset": "utf8mb4",
    "mysql_collate": "utf8mb4_unicode_ci",
}
```

### 7.2 数据类型对照

| Python ORM | MySQL 实际类型 | 说明 |
|------------|---------------|------|
| `BigInteger` | `BIGINT` | 主键、外键 |
| `String(N)` | `VARCHAR(N)` | 定长文本 |
| `CHAR(1)` | `CHAR(1)` | 标志位 (`deleted_flag`, `active_flag`) |
| `Text` | `TEXT` | 中等文本 |
| `mysql.LONGTEXT` | `LONGTEXT` | 大文本 (`content_text`, `answer_text`) |
| `mysql.JSON` | `JSON` | JSON 字段 |
| `DateTime` | `DATETIME` | 不带时区 |
| `Numeric(p,s)` | `DECIMAL(p,s)` | 精确小数 |

### 7.3 默认值

- 使用 `server_default` 而非 `default`（确保数据库层面有默认值）
- 字符串默认值用引号包裹：`server_default="ACTIVE"`

## 8. 常见场景示例

### 8.1 新增一张表

```python
def upgrade() -> None:
    op.create_table(
        "new_table_name",
        sa.Column("id", sa.BigInteger(), primary_key=True, autoincrement=True),
        sa.Column("name", sa.String(length=128), nullable=False),
        sa.Column("org_id", sa.BigInteger(), nullable=True),
        sa.Column("status", sa.String(length=32), nullable=False, server_default="ACTIVE"),
        sa.Column("deleted_flag", sa.CHAR(length=1), nullable=False, server_default="0"),
        sa.Column("created_at", sa.DateTime(), nullable=False),
        sa.Column("updated_at", sa.DateTime(), nullable=False),
        mysql_engine="InnoDB",
        mysql_charset="utf8mb4",
        mysql_collate="utf8mb4_unicode_ci",
    )

def downgrade() -> None:
    op.drop_table("new_table_name")
```

### 8.2 新增索引

```python
def upgrade() -> None:
    op.create_index("idx_table_col1_col2", "table_name", ["col1", "col2"])

def downgrade() -> None:
    op.drop_index("idx_table_col1_col2", table_name="table_name")
```

### 8.3 修改列类型

```python
def upgrade() -> None:
    op.alter_column(
        "table_name", "column_name",
        existing_type=sa.String(length=500),
        type_=sa.Text(),
        existing_nullable=True,
    )

def downgrade() -> None:
    op.alter_column(
        "table_name", "column_name",
        existing_type=sa.Text(),
        type_=sa.String(length=500),
        existing_nullable=True,
    )
```

## 9. 工具与命令速查

```bash
# 查看当前数据库版本
alembic current

# 查看迁移历史
alembic history --verbose

# 升级到最新
alembic upgrade head

# 回滚一个版本
alembic downgrade -1

# 回滚到基线
alembic downgrade base

# 重新升级（验证幂等性）
alembic upgrade head

# 打标记（已有库）
alembic stamp 0001

# 生成新迁移骨架
alembic revision -m "describe_your_change"
```
