# 14 数据库迁移规范

## 1. 数据库归属

| 数据库 | 迁移工具 | 负责人 |
| --- | --- | --- |
| `caries_biz` | Flyway | Java 后端 |
| `caries_ai` | Alembic | Python AI/RAG |

任何服务不得跨库执行 schema 迁移。

## 2. Python Alembic 规范

Python 侧只允许修改 `caries_ai`：

```bash
cd backend-python
alembic revision -m "describe_change"
alembic upgrade head
```

Docker 环境通过：

```env
CG_DB_SCHEMA_BOOTSTRAP_ENABLED=false
CG_DB_MIGRATION_ENABLED=true
```

启动时执行迁移。

## 3. Java Flyway 规范

Java 侧只允许修改 `caries_biz`。migration 命名必须稳定、可排序、不可重写已发布脚本。

## 4. JSON 扩展优先原则

AI/RAG 输出经常迭代。若新增字段不影响查询索引和强约束，优先写入：

- `ai_infer_job.result_json`
- `ai_infer_job_image.result_json`
- `ana_result_summary.raw_result_json`
- `llm_call_log` / `rag_retrieval_log` 的 JSON 扩展字段

只有当字段需要稳定查询、统计、索引或业务状态推进时，才新增列。

## 5. Phase 5C 迁移要求

Phase 5C 不要求 Java 表结构重构。必须保证：

- callback 顶层结构兼容；
- `raw_result_json` 保存 `gradingMode / gradingImplType / uncertaintyMode / uncertaintyImplType / uncertaintyScore / needsReview`；
- `ai_infer_job_image.grading_label` 和 `uncertainty_score` 可留痕；
- high uncertainty 可被 Java review 语义消费。

## 6. RAG 与通用大模型迁移要求

RAG 相关表的迁移重点是知识和日志，不是微调训练表：

- 知识库版本表；
- 文档和 chunk 表；
- 检索日志；
- 通用大模型调用日志；
- 人工评估和安全审核记录。

若后续需要训练或评估数据，只作为离线评估资产，不作为在线主链依赖。

## 7. 发布流程

1. 修改设计文档和数据字典；
2. 新增 migration；
3. 本地执行迁移；
4. 运行单元测试；
5. Docker E2E 验证；
6. 归档 SQL、日志、callback payload；
7. 合并发布。
