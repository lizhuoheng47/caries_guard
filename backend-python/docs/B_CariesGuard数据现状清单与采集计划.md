## 1. 本文档要解决什么问题

本文档回答以下 5 个关键问题：

1. 我们到底有多少数据？分别是什么模态？
2. 我们想做的 TPC-Net（TPE + CVC）需要多少配对数据？
3. EDL 分级需要多少带分级标签的数据？
4. 风险融合需要多少带结构化特征的病例？
5. 如果真实数据撑不起 Plan A，Plan B / Plan C 怎么走？

如果这 5 个问题没有明确答案，所有 Python 端开发都是"沙上盖楼"。

## 2. 为什么必须先做数据摸底

国一级别的作品在答辩环节会被问这些问题：

- "你们的数据来自哪里？有伦理审批吗？"
- "训练集多大？有没有做患者级划分？"
- "跨视图一致性（CVC）需要同一颗牙的多视图配对，你们有吗？有多少对？"
- "分级标签是谁打的？有没有做一致性评估（kappa）？"
- "外部验证集从哪来？和训练集怎么隔离？"

**这些问题答不上来，就算模型跑出来 Dice=0.92 也会被打回省奖。**

反过来，**只要数据摸底做得扎实、证据链完整，就算 Dice 只有 0.82，也能冲国一**。因为评委看的是"真实性 + 工程完整度"，不是单一指标。

## 3. 当前数据现状（v0 空盘状态）

**截至本文档签发日，本项目尚未采集任何真实业务数据。**

这不是坏事，反而是启动数据治理的最佳时机：**在没有任何数据被污染的前提下，先把采集、脱敏、标注、分层、版本的全流程跑一遍**，后续采集才不会出现"边采边乱"。

## 4. 数据目标规模（保底 / 冲刺双口径）

采用 **保底目标 + 冲刺目标** 两级口径，避免一旦冲刺目标未达成，答辩现场没话讲。

### 4.1 保底目标（必须完成，否则项目不成立）

| 数据类型 | 保底数量 | 用途 |
|---|---|---|
| 全景片（带 mask 标注） | **500 张** | TPC-Net 主训练 |
| 全景片（带牙位框标注） | 500 张 | YOLOv8 牙位检测训练 |
| 口内照（带 mask 标注） | **300 张** | TPC-Net 跨视图辅助 |
| 同牙跨视图配对样本 | **150 对** | CVC 约束训练 |
| 带 C0/C1/C2/C3 分级标签的病灶 ROI | **800 个** | EDL 分级训练 |
| 带完整结构化特征的病例 | **400 例** | 风险融合训练 |
| 独立测试集（患者级隔离） | 100 例 | 外部验证 |

**保底目标的科学依据**：
- 500 张全景 + 300 张口内照基本覆盖 SegFormer-B2 finetune 的最小训练量
- CVC 只要有 150 对配对就能形成有效的一致性正则
- EDL 分级 800 个 ROI，每级约 200 个，能跑出有意义的 Macro-F1
- 风险融合用 LightGBM，400 例足够，且能出 SHAP 解释图

### 4.2 冲刺目标（答辩主叙事目标）

| 数据类型 | 冲刺数量 |
|---|---|
| 全景片（带 mask 标注） | 1200 张 |
| 口内照（带 mask 标注） | 800 张 |
| 同牙跨视图配对样本 | 400 对 |
| 带分级标签的病灶 ROI | 2000 个 |
| 带完整结构化特征的病例 | 1000 例 |
| 独立测试集 | 200 例 |

### 4.3 绝对红线（低于这个数，不能做的创新要砍掉）

| 如果实际只有这么多 | 必须砍掉的能力 |
|---|---|
| 全景片 mask < 300 张 | TPC-Net 降级为"UNet baseline"，不讲 TPE |
| 跨视图配对 < 80 对 | 砍掉 CVC，改为"多视图推理时后处理投票" |
| 分级标签 < 400 个 | EDL 降级为"softmax + MC Dropout"作为不确定性替代方案 |
| 结构化特征 < 200 例 | 风险融合改为规则引擎，不讲模型 |

## 5. 数据来源策略

采用 **三条腿走路**，每一条都落地执行，不依赖单一来源。

### 5.1 来源 A：公开数据集（基础盘）

**优先使用，立即就能开工。**

| 数据集 | 类型 | 规模 | 许可 | 用途 |
|---|---|---|---|---|
| Tufts Dental Database | 全景片 + 标注 | ~1000 张 | 学术 | TPC-Net 主训练 + 牙位检测 |
| DENTEX 2023 Challenge | 全景片 + 诊断标签 | ~1200 张 | Challenge 公开 | TPC-Net + 分级辅助 |
| Panoramic X-ray DB（多个 Kaggle 源） | 全景片 | 500~2000 张 | 多数 CC | 牙位预训练 |
| Children's Dental Panoramic Radiographs | 儿童全景片 | 约 500 张 | 学术 | 扩充年龄分布 |

**使用原则**：
- 公开数据集只用于**预训练和 baseline**，不作为答辩的"主成绩数据"
- 所有公开数据集必须在 Dataset Card 中注明出处和许可
- 公开数据集不能混入"独立测试集"

### 5.2 来源 B：合作医院采集（主盘）

**这是答辩的主叙事，必须拉到至少 1 家合作方。**

合作方优先级：
1. 学校合作口腔医院（最容易搞定）
2. 学校附近社区卫生服务中心
3. 指导老师的临床合作伙伴
4. 校医院口腔科（可能设备不全，但胜在审批快）

**合作要落地 3 件事**：
1. 签一份最小化的数据合作协议（1~2 页即可），明确数据脱敏后可用于竞赛和后续学术发表
2. 申请学校学术伦理委员会（IRB）审查，走"小规模、已脱敏、学生科研"通道，通常 2~3 周
3. 约定一个临床顾问（副主任医师以上），在答辩中可署名

**采集规格冻结**：

| 维度 | 要求 |
|---|---|
| 全景片分辨率 | ≥ 2560×1440 |
| 口内照 | 牙科反光镜拍摄 + 手机微距均可，分辨率 ≥ 1080p |
| 根尖片 | 数字化根尖片，≥ 300 dpi |
| 脱敏 | 现场脱敏，剥离 DICOM 头部所有患者标识 |
| 结构化数据 | 同步填写简化问卷（年龄、性别、刷牙频率、饮食、既往龋齿数、最近一次检查时间） |

### 5.3 来源 C：团队自采（补充盘）

用来填补**口内照**和**跨视图配对**的空缺。因为公开数据集里几乎没有"同一颗牙在多视图下的配对"。

**自采方案**：
- 团队成员自身 + 志愿者（朋友、同学，签知情同意书）
- 使用手机微距 + 冷光源 + 牙科检查镜
- 每人至少采集 "1 张全景 + N 张口内照（按象限）"
- 目标：30~50 人，每人约 5 张口内照
- 预计产出：150~250 张口内照，其中约 80 对可形成跨视图配对

## 6. 跨视图配对数据的专项处理

**这是整个数据规划的最硬难点，也是 CVC 创新能不能立住的唯一决定因素。**

### 6.1 什么叫跨视图配对

同一颗牙（例如右上第一磨牙 FDI=16），在以下多个视图中都有对应标注：
- 全景片中的那颗 16 号牙
- 口内照中的那颗 16 号牙
- （可选）根尖片中的那颗 16 号牙

配对的表现形式：一条 JSON 记录

```json
{
  "pairId": "PAIR000001",
  "toothCode": "16",
  "views": [
    {
      "imageId": "img_pan_001",
      "modality": "PANORAMIC",
      "bbox": [1120, 580, 1280, 740],
      "maskPath": "..."
    },
    {
      "imageId": "img_intra_001",
      "modality": "INTRAORAL",
      "bbox": [320, 180, 560, 480],
      "maskPath": "..."
    }
  ],
  "severityCode": "C2",
  "source": "clinical_visit_0017"
}
```

### 6.2 配对数据的产生路径

**只有一条现实可行路径**：

> 在同一次就诊中，同一位患者同时拍摄了全景片和口内照，且两侧都能定位到同一颗牙。

这意味着配对数据不能从单一公开数据集中凑，必须从合作医院的真实就诊记录中抽取，或从自采中产生。

**配对率估算**：
- 合作医院一次就诊，能拍到"同一颗牙"跨视图的概率 ≈ 20%~30%
- 自采环境下（主动安排），配对率可以做到 ≈ 70%

### 6.3 配对标注流程

专门为配对数据设计一条工作流：

1. 标注员先独立完成两个视图各自的牙位检测 + mask
2. 系统按 `patientId + visitId + toothCode` 自动寻找潜在配对
3. 临床顾问复核配对是否为同一颗牙
4. 通过的配对写入 `pair_manifest.jsonl`

### 6.4 如果配对数据凑不够怎么办

**门槛判断**：

| 配对对数 | 能做的 |
|---|---|
| ≥ 150 | Plan A：把 CVC 作为创新之一在答辩中主讲 |
| 80~149 | Plan B：CVC 作为"探索性工作"呈现，主创新改为 TPE + EDL |
| < 80 | Plan C：砍掉 CVC，改为"推理时多视图投票"作为后处理亮点 |

**Plan B 和 Plan C 的答辩口径必须提前准备好**，不能等到发现配对不够才现编。

## 7. 标注任务拆解与人手分配

### 7.1 标注任务分层

标注任务按难度和专业门槛分 3 级：

| 级别 | 任务 | 谁能做 | 工具 |
|---|---|---|---|
| L1 | 牙位检测框（bbox） | 经过 2 小时培训的学生标注员 | Label Studio / CVAT |
| L2 | 病灶 mask 分割 | 经过专项培训的学生 + 临床顾问复核 | Label Studio + SAM 辅助 |
| L3 | C0/C1/C2/C3 分级 | 必须是临床顾问或有口腔背景的高年级医学生 | Label Studio |

**SAM 辅助标注**是关键提效点：
- 使用 SAM（Segment Anything）+ 人工修正的方式，单张全景片 mask 标注时间可从 15 分钟降到 3~5 分钟
- 建议在 Python 仓库的 `tools/sam_assist/` 下搭一个小工具

### 7.2 推荐人手配置

| 角色 | 人数 | 职责 | 工作量预估 |
|---|---|---|---|
| 数据负责人 | 1 | 统筹采集、脱敏、入库、分层 | 全程 |
| 标注员（L1+L2） | 2~3 | 牙位框 + mask | 每人每天 40~60 张 |
| 临床顾问 | 1 | L3 分级 + L2 复核 + 仲裁 | 每周 4 小时 |
| 算法工程师 | 1 | 搭 SAM 辅助工具 + 质控脚本 | 第 1~2 周为主 |

### 7.3 产能估算

**以保底目标为例**：

- 牙位框 500 张 × 2 分钟 = 1000 分钟 ≈ 17 小时 ≈ 1 人 2~3 天
- Mask 800 张 × 4 分钟（有 SAM 辅助）= 3200 分钟 ≈ 53 小时 ≈ 2 人 5 天
- 分级 800 ROI × 1 分钟 = 800 分钟 ≈ 14 小时 ≈ 临床顾问 4 周

**结论**：保底目标的标注量在 **3 人 + 1 临床顾问 + 2 周** 内可以完成。瓶颈不在标注，而在数据采集速度。

## 8. 数据采集与标注进度表

按 16 周答辩倒计时规划，数据线必须在第 8 周之前完成，给模型训练和答辩打磨留够时间。

| 周次 | 数据采集 | 标注 | 训练数据可用里程碑 |
|---|---|---|---|
| W1 | 下载公开数据集（Tufts / DENTEX）；申请 IRB；谈合作医院 | 搭 Label Studio + SAM 辅助工具 | - |
| W2 | 公开数据集整理入库；开始合作医院沟通 | 跑通 20 张样例的 L1+L2 标注流程 | - |
| W3 | 合作医院首批数据 50~100 张；团队自采开始 | 公开数据集导入 + 格式转换 | D0：公开数据集 snapshot |
| W4 | 合作医院第二批 100~200 张 | 公开数据集 baseline 可训（500+ 全景片） | **D1：TPC-Net baseline 可训** |
| W5 | 合作医院累积 300 张；自采完成 | L3 分级启动；继续 mask 标注 | - |
| W6 | 合作医院累积 500 张 | 配对数据专项处理 | **D2：保底数据到位** |
| W7 | 补充采集（查漏补缺） | 所有标注完成 + 质控抽查 | **D3：可用于正式训练** |
| W8 | - | 独立测试集最终冻结 | **D4：版本 v1.0 snapshot 固化** |
| W9+ | - | 只做修正回流和小样补充 | 进入训练与评估阶段 |

**关键节点**：
- **W4 D1 里程碑**：即使合作医院数据还没到，只用公开数据集也必须能跑通 TPC-Net baseline，保证后续不会全线阻塞
- **W6 D2 里程碑**：保底数据规模必须到位，这是 Plan A 存活的最后期限
- **W8 D4 里程碑**：数据集必须打版本号并冻结，不再追加新数据到训练集

## 9. Plan A / Plan B / Plan C 策略

### 9.1 Plan A：理想情况（冲刺目标达成）

**前提**：
- 合作医院数据 ≥ 1000 张
- 跨视图配对 ≥ 400 对
- 分级标签 ≥ 2000 ROI

**答辩主叙事**：
- TPC-Net（TPE + CVC）是核心创新之一
- EDL 不确定性感知接入复核闭环
- 风险融合给出可解释的 SHAP 贡献

### 9.2 Plan B：保底情况（保底目标达成）

**前提**：
- 合作医院数据 ~ 500 张 + 公开数据集 1000 张
- 跨视图配对 ~ 150 对
- 分级标签 ~ 800 ROI

**答辩调整**：
- TPC-Net 主讲 TPE 的效果，CVC 作为"在有限配对数据下的一致性正则尝试"，讲成一个"诚实的工程实践"
- EDL 保留为主创新
- 风险融合不变
- 明确强调"我们的数据规模不追求大而追求治理完整"

### 9.3 Plan C：最差情况（保底目标未达成）

**前提**：
- 主要依赖公开数据集
- 合作医院数据 < 300 张
- 跨视图配对 < 80 对

**答辩调整**：
- TPC-Net 退为"基于牙位先验的分割增强（TPE only）"
- 砍掉 CVC，改为推理时多视图投票
- EDL 保留
- 主叙事重心全面偏向"平台闭环 + 医生修正回流 + 可信治理"
- 强调"软件赛道看的是工程完整度，数据规模不是决定项"

### 9.4 切换条件

- **W4 周末**：如果合作医院还没签协议，立即激活 Plan B 预案
- **W6 周末**：如果保底数据未到位，立即激活 Plan C 预案
- **切换必须留书面记录**：写在项目 issue 中，附带决策时点、数据截图、决策人

## 10. 数据与模型能力的映射关系

**本表是整个数据规划的核心决策依据。**

| 算法模块 | 数据依赖 | 保底需求 | 如果数据不足的降级方案 |
|---|---|---|---|
| YOLOv8 牙位检测 | 全景片 + 牙位 bbox | 500 张 | 公开数据集 + 迁移学习 |
| TPC-Net 主干分割 | 全景片 + mask | 500 张 | SegFormer-B2 预训练权重 + 小样 finetune |
| TPE 模块 | 上同 + FDI 编号 | 500 张 | 不影响 |
| CVC 模块 | **跨视图配对 150 对** | **必须** | **砍掉，改为推理后处理** |
| EDL 分级 | 病灶 ROI + C0~C3 标签 | 800 ROI | 降级为 softmax + MC Dropout |
| 风险融合 | 完整结构化特征 | 400 例 | 降级为规则引擎 |
| SHAP 解释 | 上同 | 400 例 | 规则引擎直接给 top-3 因子 |

## 11. 数据治理与脱敏流程

### 11.1 分层存储路径

严格按 03 号文档 §6 的数据分层设计实施：

```text
datasets/
├── raw_manifest/           # 原始数据登记（只登记，不直接训练）
│   ├── from_tufts/
│   ├── from_dentex/
│   ├── from_partner_hospital/
│   └── from_self_collect/
├── desensitized/           # 脱敏副本（训练只从这里取）
│   ├── images/
│   ├── annotations/
│   └── manifests/
├── snapshots/              # 版本固化的数据快照
│   ├── v0_public_only/
│   ├── v1_baseline/
│   ├── v1_1_enhanced/
│   └── v1_2_final/
├── dataset_cards/          # 每个 snapshot 对应一份 Dataset Card
│   ├── v0_public_only.md
│   ├── v1_baseline.md
│   └── ...
└── checksums/              # 每个 snapshot 的 MD5 清单
```

### 11.2 脱敏动作清单

每一张进入 `desensitized/` 的图像必须完成：

- [ ] 清除 DICOM 头部所有 PII 字段（PatientName、PatientID、AccessionNumber 等）
- [ ] 文件名哈希化（`sha256(original_path + salt)` 取前 16 位）
- [ ] EXIF 信息清除（手机自采照片必做）
- [ ] 结构化数据中的姓名、电话、身份证号全部剥离
- [ ] 日期字段模糊化到月份精度
- [ ] 脱敏脚本在 `tools/export_desensitized_data.py` 中统一实现

### 11.3 脱敏与业务库的边界

**硬规则**：

- Python 训练代码 **禁止** 连接业务库（Java 后端使用的 MySQL `caries_guard`）
- 训练只从 `desensitized/` 和 `snapshots/` 读取
- 医生修正回流通过 **`ana_correction_feedback` → 导出工具 → `desensitized/corrections/`** 三段式，中间必经脱敏和审批

## 14. 答辩口径（如何说"有多少数据"才不会被问穿）

### 14.1 推荐答辩话术模板

> "我们构建了一个面向真实筛查场景的数据集。总规模约 XXX 张，其中来自公开数据集 XX%，来自合作 XX 医院 XX%，来自团队自采 XX%。所有数据都按照统一的脱敏流程、患者级划分和标注质控流程治理，固化为版本 v1.0.0 的数据快照。我们不追求数据规模最大，而追求治理完整，所以每一张图都能追溯来源、标注人、质控状态和版本。"

### 14.2 要规避的表述

- ❌ "我们收集了十万张真实临床数据" （除非真的有十万张且能出证明）
- ❌ "我们的数据来自多家三甲医院" （除非每家都有合作协议）
- ❌ "我们做了大规模多中心研究" （学生项目几乎不可能做到）
- ❌ 拒绝回答"数据规模是多少"（评委会追问到你答为止）

### 14.3 被追问时的标准应对

**Q: 你们的数据规模是多少？**
A: 总规模 XXX 张（给出准确数字），其中合作医院 XX 张，公开数据集 XX 张，自采 XX 张。我们有一份 Dataset Card 可以展示详细分布。

**Q: 为什么只有这么多？**
A: 我们这个阶段的重点是证明平台能闭环运行、治理流程完整，不是证明我们有最大的数据规模。真实临床数据的采集和标注有非常高的合规和伦理门槛，这本身就是我们项目可信性设计的一部分。

**Q: 只有几百张全景片，能训出好模型吗？**
A: 我们用 SegFormer-B2 作为主干，它在 ADE20K 和 Cityscapes 上都有预训练权重，small-sample finetune 效果足够支持我们证明 TPE 等增强的有效性。我们的目标是对比"有没有 TPE 的差异"，不是刷榜。
