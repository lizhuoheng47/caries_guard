# B CariesGuard 数据现状清单与采集计划

## 1. 当前数据策略

CariesGuard 当前不以微调大模型作为近期主路线。数据采集目标调整为三类：

1. 影像算法适配器评估数据：用于 quality、detection、segmentation、grading 的离线评估；
2. RAG 知识图库数据：用于通用大模型检索增强；
3. 业务闭环数据：用于报告、复核、随访和风险规则评估。

## 2. 数据现状

已具备：

- Java 业务库病例、影像、附件、任务、报告、复核基础结构；
- MinIO 原始影像和 visual assets 存储链路；
- Python AI 推理和 RAG 运行库；
- Phase 5C grading + uncertainty E2E 证据；
- RAG 知识库、文档、chunk、调用日志基础设计。

仍需补强：

- 标准化口腔知识文档；
- 结构化指南和诊疗路径；
- 高质量影像标注集；
- 人工复核结果回流；
- RAG 回答质量评估集；
- 风险因素和复查周期规则库。

## 3. 数据采集对象

### 3.1 影像数据

用于算法适配器评估，不用于通用大模型微调。

采集字段：

- 影像类型；
- 拍摄设备和质量状态；
- 牙位；
- 病灶区域；
- 分级标签；
- 标注医生、复核医生；
- 标注版本；
- 脱敏状态。

### 3.2 知识图库数据

用于 RAG：

- 权威指南；
- 机构内诊疗规范；
- 口腔健康科普材料；
- 龋病分级说明；
- 影像质量规范；
- 报告模板；
- 风险因素说明；
- 复查周期规则。

每条知识必须记录来源、版本、适用范围、更新时间和审核状态。

### 3.3 业务反馈数据

用于持续评估：

- 医生是否采纳 AI 建议；
- high uncertainty 是否触发复核；
- 复核结论与 AI 分级差异；
- 报告修改内容；
- 随访结果；
- RAG 回答人工评分。

## 4. 标注计划

短期：

- 固定 50-100 条标准影像样例；
- 覆盖 C0/C1/C2/C3；
- 覆盖低/高 uncertainty；
- 覆盖质量不合格和失败回退；
- 建立 RAG 标准问答集。

中期：

- 引入双医生标注和仲裁；
- 按牙位、病灶、影像类型分层采样；
- 建立 review 回流数据集；
- 建立 RAG hallucination 和引用准确率评估集。

长期：

- 若未来数据、合规和评估条件成熟，可评估专用视觉模型训练；
- 通用大模型仍优先采用检索增强和安全约束，不以微调作为默认路径。

## 5. 脱敏与合规

- 原始影像不得含直接身份信息；
- 患者姓名、电话、身份证等不得进入 `caries_ai`；
- RAG 知识库不得混入未脱敏病例全文；
- 日志只保留摘要和引用，不保存敏感明文；
- 人工评估数据应记录评估人和版本。

## 6. 质量指标

影像算法：

- 分级一致率；
- uncertainty 校准情况；
- high uncertainty 召回率；
- visual asset 生成成功率；
- real 模式失败显式性。

RAG：

- 检索命中率；
- 引用准确率；
- 回答可读性；
- 幻觉率；
- 拒答合理性；
- 医生采纳率。

## 7. 近期执行方案

1. 整理知识库 v1：指南、分级说明、复查建议、报告模板；
2. 固化 5C E2E 样例 A/B/C/D；
3. 建立 RAG 标准问题集；
4. 将医生复核结果回流到数据采集表；
5. 每次知识库更新后运行 RAG 回归评估；
6. 每次算法适配器更新后运行 Docker E2E。
