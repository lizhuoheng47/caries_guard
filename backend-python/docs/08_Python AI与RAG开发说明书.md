# 1. 文档目标

本说明书用于解决以下问题：

# 2. 开发基线

## 2.1 项目定位

本项目不是单纯的模型演示工程，而是面向真实口腔筛查流程的**可信辅助决策平台**。Python 端必须服务以下闭环：

> Java 创建分析任务 → Python 获取影像与上下文 → 质量检查 / 检测 / 分割 / 分级 / 风险推断 → 回调业务快照 → Java 生成报告与复核流程 → 医生修正回流 → 脱敏进入候选训练样本池

## 2.2 为什么现在可以正式启动 Python 端

当前 Python 端已经具备正式启动条件，原因如下：

1. Java 与 Python 的职责边界已经明确；
2. 系统总体架构已经冻结为“模块化单体 + 独立 AI 服务 + RabbitMQ 解耦”；
3. Java 侧 analysis → report → followup 消费链已具备成型条件；
4. MinIO、RabbitMQ、MySQL、analysis callback、attachment/object key 规则已经基本冻结；
5. 业务数据与训练数据的边界已经写清，Python 可以独立建设 AI 运行层与训练治理层，而不污染业务库。

因此，**现在不是“能不能启动 Python”，而是“应该如何按正确顺序启动 Python”。**

## 2.3 Python 端职责

Python 端负责以下三层能力。

### 第一层：视觉诊断运行层

- 图像质量检查；
- 牙位检测与牙位编码映射；
- 病灶分割；
- 龋病分级；
- 风险评估子模型；
- 可解释性图层生成（mask / overlay / heatmap）；
- Java 回调与任务状态反馈。

### 第二层：文本智能层（RAG + 通用大模型）

- 知识库构建与检索；
- 医生知识检索；
- 患者报告通俗化说明；
- 护理建议与复查建议的受控生成；
- 问答日志与引用证据留痕。

### 第三层：训练治理辅助层

- 数据快照登记；
- 模型版本登记；
- 离线评估记录；
- 人工审批上线记录；
- 候选训练样本池管理。

## 2.4 Python 端明确不负责的内容

以下内容不应在 Python 端重复建设：

- 用户认证、角色权限、组织树、菜单；
- 患者主档、就诊、病例、随访等业务主线；
- Java 业务库中的最终业务状态流转；
- 报告版本归档与导出审计；
- 直接操作业务主表完成“越权更新”。

**原则：Python 负责结果生产与文本智能，Java 负责业务编排与结果承载。**

# 3. 总体架构与工程原则

## 3.1 固定架构

当前 Python 端必须嵌入以下固定架构，不再另起一套说法：

> **模块化单体 Java 平台 + 独立 Python AI 服务 + RabbitMQ 异步解耦 + MinIO 对象存储 + MySQL 元数据存储**

## 3.2 Python 端固定双层 AI 架构

Python 侧必须采用“**视觉诊断主链 + 文本智能层**”双层架构：

### A. 视觉诊断主链

用于处理：

- 影像质量；
- 牙位检测；
- 病灶分割；
- 龋病分级；
- 风险评估子模型；
- 可解释性图层生成。

### B. 文本智能层

用于处理：

- 患者版报告解释；
- 医生端知识问答；
- 口腔健康宣教建议；
- 风险因素文本说明；
- FAQ 与知识检索。

### 结论

- **视觉主链不被通用大模型替代；**
- **文本智能层不做 LoRA 微调，而做 RAG + 通用大模型。**

## 3.3 为什么不做 LoRA

本阶段取消 LoRA 的原因不是“不会做”，而是“当前阶段不值得做”：

1. 比赛周期内，LoRA 难以形成比 RAG 更有说服力的工程收益；
2. 文本智能层的主要问题不是“模型没记住知识”，而是“输出是否受控、是否引用正确、是否可追溯”；
3. 对答辩来说，RAG 更容易解释知识来源、更容易展示证据链；
4. 对开发来说，RAG 的可演示闭环快于微调闭环；
5. 医疗辅助场景下，**引用明确的知识检索**比“微调后模型自己说”更可信。

因此，Python 侧文本智能的正式路线为：

> 知识入库 → 文档切分 → Embedding → 检索 → Prompt 编排 → 通用大模型生成 → 证据与日志落库

## 3.4 Python 端工程设计原则

1. **先联调，后强化**：先让 Java 能消费 Python 结果，再迭代模型效果；
2. **先最小闭环，后训练治理**：先打通运行链，再做完整训练平台；
3. **先受控文本，后炫技文本**：先保证知识检索正确与可追溯，再追求文风丰富；
4. **只存必要数据**：Python 不复制 Java 主业务表，只保存 AI 运行元数据与治理数据；
5. **所有输出可追溯**：每次推理、检索、生成都必须能反查来源、模型版本与时间。

# 4. 技术栈与版本建议

## 4.1 Python 平台基线

- Python：**3.11**
- FastAPI：HTTP API 与回调服务
- Uvicorn / Gunicorn：服务运行
- Pydantic v2：数据校验与契约对象
- SQLAlchemy 2.x：ORM 与数据库访问
- Alembic：数据库迁移
- PyMySQL / mysqlclient：MySQL 驱动
- aio-pika 或 pika：RabbitMQ 生产/消费
- minio：MinIO SDK
- httpx：与外部服务 / 通用大模型网关通信
- Jinja2：模板编排或 Prompt 模板
- loguru 或标准 logging：日志
- pytest：测试
- pytest-asyncio：异步测试
- tenacity：重试与容错

## 4.2 向量检索建议

当前建议采用以下二级策略：

### 开发 / 演示基线

- 文档元数据落 MySQL；
- 向量索引优先使用 **FAISS 本地索引文件**；
- 这样不引入额外独立向量数据库，部署最轻。

### 可扩展方案

- 若后续需要更稳定的在线向量检索，可升级为 **Qdrant**；
- 但不作为当前比赛一期硬依赖。

### 当前决策

**一期冻结为：MySQL 元数据 + FAISS 本地向量索引文件。**

这样既符合“文档收敛为 Java 一册 / Python 一册”的目标，也不会增加额外基础设施复杂度。

## 4.3 不纳入一期基线的内容

以下内容不作为当前 Python 端必须项：

- LoRA / QLoRA 微调；
- 分布式训练平台；
- Ray / Kubeflow / Airflow；
- 多模型自动路由平台；
- 多租户知识平台；
- 大规模在线向量数据库集群。

# 5. 工程组织方式

## 5.1 推荐目录结构

建议在 `backend-python/` 内采用以下结构：

```text
backend-python/
├── app/
│   ├── api/                     # FastAPI 路由
│   ├── core/                    # 配置、日志、异常、基础工具
│   ├── schemas/                 # Pydantic 契约对象
│   ├── modules/
│   │   ├── quality_check/       # 质量检查
│   │   ├── tooth_detect/        # 牙位检测
│   │   ├── segmentation/        # 分割
│   │   ├── grading/             # 分级
│   │   ├── risk/                # 风险评估
│   │   ├── explainability/      # heatmap / overlay / mask
│   │   ├── rag/                 # 检索与生成编排
│   │   ├── knowledge/           # 知识入库、切分、索引重建
│   │   ├── governance/          # 模型版本、数据快照、审批治理
│   │   └── callbacks/           # Java 回调
│   ├── integration/
│   │   ├── mq/                  # RabbitMQ 适配
│   │   ├── storage/             # MinIO 适配
│   │   ├── llm/                 # 通用大模型网关适配
│   │   └── vector/              # FAISS / Qdrant 适配
│   ├── repositories/            # 数据访问层
│   ├── services/                # 应用服务编排
│   └── workers/                 # MQ 消费者、后台任务
├── alembic/
├── tests/
├── scripts/
├── main.py
└── requirements.txt
```

## 5.2 分层职责

### api

- 只负责 HTTP 入参与出参；
- 不直接拼接底层存储细节；
- 不在路由层写复杂推理业务。

### modules

- 负责单一能力域；
- 每个模块只解决一类问题，不跨域写大而全逻辑。

### integration

- 适配 RabbitMQ、MinIO、通用大模型、向量索引；
- 外部依赖都从这里进入，不要在业务模块散写 SDK 调用。

### repositories

- 对 MySQL 元数据与治理数据负责；
- 不直接操作 Java 业务库。

### services

- 编排完整用例，例如“消费 analysis.requested → 推理 → 回调 Java”。

### workers

- 承接 MQ 消费、索引重建、批处理、回调重试等后台任务。

# 6. Python 对外契约与联调边界

## 6.1 一期必须实现的接口

### 1. 图像质量检查

```text
POST /ai/v1/quality-check
```

用途：快速判断影像是否可分析。

### 2. 综合分析

```text
POST /ai/v1/analyze
```

用途：完成牙位检测、分割、分级、可解释性图层生成。

### 3. 风险评估

```text
POST /ai/v1/assess-risk
```

用途：基于影像快照 + 结构化因子输出风险等级与解释。

### 4. 模型版本查询

```text
GET /ai/v1/model-version
```

用途：返回当前启用模型和知识库版本，用于联调与演示。

## 6.2 二期扩展接口（RAG）

以下接口不作为一期联调硬依赖，但建议在 Python 端预留：

### 1. 患者解释生成

```text
POST /ai/v1/rag/patient-explanation
```

### 2. 医生知识问答

```text
POST /ai/v1/rag/doctor-qa
```

### 3. 知识库重建

```text
POST /ai/v1/knowledge/rebuild
```

### 4. 知识库文档导入

```text
POST /ai/v1/knowledge/documents
```

## 6.3 MQ 事件与回调约定

Python 端必须适配以下事件：

- `analysis.requested`
- `analysis.completed`
- `analysis.failed`
- `report.generate.requested`（预留）

### analysis.requested 消费要求

消费后必须至少完成：

1. 读取 Java 下发的 `taskNo / caseId / caseNo / image list / accessUrl / expireAt`；
2. 下载原图或通过预签名 URL 获取；
3. 执行最小分析链；
4. 将 visual asset 上传到 MinIO 或按约定返回 metadata；
5. 回调 Java analysis-result 接口；
6. 记录回调结果与失败原因。

## 6.4 Java / Python 严格边界

Python 不允许直接做以下事情：

- 直接改 `med_case.case_status_code`；
- 直接生成业务报告版本；
- 直接改患者、病例、随访主表；
- 绕过 Java 业务层直接把“最终结论”写进业务库。

**所有业务最终状态都必须由 Java 接管。**

# 7. RAG 总体设计

## 7.1 RAG 的定位

RAG 在本项目中不是“聊天小玩具”，而是以下受控文本能力的统一技术底座：

1. 患者版报告的通俗化解释；
2. 医生端知识检索与问答；
3. 护理建议与复查建议的文本生成；
4. 风险因素解释；
5. FAQ 与宣教内容输出。

## 7.2 RAG 的知识来源范围

当前允许进入知识库的内容建议限定为：

- 项目内冻结后的业务与算法文档；
- 口腔健康宣教规范文本；
- 患者常见问答库；
- 报告模板辅助文本与解释性模板；
- 医生复核流程说明；
- 风险等级解释规则；
- 已审核的内部知识条目。

当前不建议直接纳入：

- 原始业务病例全文；
- 患者可识别隐私信息；
- 未经审核的互联网爬取文本；
- 未经脱敏的修正记录原文。

## 7.3 RAG 的标准链路

```text
知识文档导入
    ↓
文档清洗与切分
    ↓
Embedding 生成
    ↓
向量索引构建
    ↓
查询改写 / Prompt 编排
    ↓
检索 TopK 片段
    ↓
通用大模型生成
    ↓
引用证据与生成日志落库
```

## 7.4 RAG 输出约束

RAG 生成结果必须遵守以下规则：

1. 不伪造引用来源；
2. 不生成超出知识范围的明确医疗承诺；
3. 高风险建议必须保守表达；
4. 面向患者的文本必须“可理解、不过度医疗化、不过度保证”；
5. 所有请求必须保留检索命中文档与最终生成结果的日志。

## 7.5 Prompt 分层原则

建议采用 3 层 Prompt：

### 系统 Prompt

定义：

- 角色边界；
- 安全边界；
- 语言风格；
- 禁止输出内容。

### 任务 Prompt

定义：

- 当前场景（患者解释 / 医生问答 / 护理建议）；
- 输出结构；
- 语气与粒度。

### 证据 Prompt

定义：

- 本次检索得到的知识片段；
- 风险等级或病例上下文；
- 需引用的知识依据。

# 8. 开发顺序与阶段收敛标准

本项目 Python 端必须按以下顺序推进。

---

### 目标

统一术语、技术路线与接口边界。

1. 冻结 Python 目录结构；
2. 冻结一期接口；

### 收敛标准

- 团队内不再讨论是否 LoRA；
- 不再出现第二套 Python 架构说法；
- 接口名、事件名、表名前缀全部定稿。

---

## 阶段 1：AI 服务底座可运行

### 目标

让 Python 服务可以启动、暴露接口、连接 MinIO/MySQL/RabbitMQ。

### 具体要做的事

1. 建 FastAPI 项目骨架；
2. 接入配置中心（环境变量）；
3. 接入日志与 traceId；
4. 接入 MySQL 元数据访问；
5. 接入 RabbitMQ 消费与发布；
6. 接入 MinIO 下载与上传；
7. 提供健康检查与模型版本接口。

### 收敛标准

- `docker compose up` 能启动 Python 服务；
- `/ai/v1/model-version` 可正常返回；
- Python 能连通 MySQL、MinIO、RabbitMQ；
- 有最小健康检查。

---

## 阶段 2：占位推理闭环打通

### 目标

在没有真实复杂模型的前提下，让 Java → Python → Java 的 analysis 主链走通。

### 具体要做的事

1. 消费 `analysis.requested`；
2. 根据 accessUrl 拉取原图；
3. 用规则或 mock 输出：
   - 牙位列表；
   - 分级结果；
   - uncertainty 分数；
   - 风险等级；
   - visual asset 占位图；
4. 回调 Java analysis-result；
5. 写入 Python 运行元数据与回调日志。

### 收敛标准

- Java 可以创建任务，Python 可以消费；
- Python 回调成功后，Java 能生成 report；
- 即使模型是假数据，也能闭环演示 analysis → report → followup。

---

## 阶段 3：视觉主链替换为真实模型

### 目标

逐步用真实质量检查、检测、分割、分级逻辑替换占位实现。

### 具体要做的事

1. 接入质量检查模型或规则；
2. 接入牙位检测模型；
3. 接入分割模型；
4. 接入分级模型；
5. 产出可解释性图层；
6. 稳定生成 `mask / overlay / heatmap`。

### 收敛标准

- 真实模型输出符合 Java 现有 callback 契约；
- visual asset 可在业务端展示；
- 失败任务能回写并重试；
- 基础推理耗时与错误日志可观测。

---

## 阶段 4：RAG 最小闭环落地

### 目标

让“知识库 + 通用大模型”真正变成可调用的文本能力，而不是 PPT 口号。

### 具体要做的事

1. 建知识库表与文档入库流程；
2. 实现文档切分与 embedding；
3. 实现向量索引构建；
4. 实现患者解释生成接口；
5. 实现医生问答接口；
6. 记录检索日志与大模型调用日志。

### 收敛标准

- 至少有一套固定知识库可以被检索；
- 至少有一个患者解释用例可演示；
- 至少有一个医生问答用例可演示；
- 所有结果都能反查命中的知识片段。

---

## 阶段 5：训练治理与版本治理补齐

### 目标

让系统具备“医生修正 → 候选样本 → 数据快照 → 候选模型 → 离线评估 → 人工审批上线”的可信叙事。

### 具体要做的事

1. 建数据快照与样本明细表；
2. 建模型版本表、评估表、审批表；
3. 建训练运行记录表；
4. 形成 Gold Set 管理最小闭环；
5. 固定候选模型审批流程。

### 收敛标准

- 能展示一个从修正样本到候选模型的完整证据链；
- 模型版本不是手工口头管理，而是有表、有记录、有审批时间。

# 9. 关键实现细则

## 9.1 Python 对 MinIO 的使用规范

Python 必须遵守当前 Java 已冻结的对象存储体系：

- 原图从 `caries-image` 读取；
- visual asset 写入 `caries-visual`；
- 不自行发明新的桶名；
- object key 必须遵守既有结构化规则；
- 回调 Java 时优先传回 attachment 元数据或对象元数据，不直接越过 Java 写业务快照。

## 9.2 Python 对 callback 的要求

回调必须包含：

- `taskNo`
- 结果状态
- 分析摘要
- 风险结果
- 不确定性
- visual asset 元数据
- 模型版本
- 失败原因（如有）

## 9.3 错误处理原则

Python 端出现任何失败时，都必须：

1. 记录失败任务；
2. 保留 traceId；
3. 可区分“模型失败 / 文件下载失败 / 回调失败 / 知识检索失败”；
4. 支持回调重试或任务重试。

## 9.4 日志与审计要求

Python 至少记录以下日志：

- 推理任务创建与完成；
- MQ 消费与回调；
- 文件下载与上传；
- RAG 检索命中情况；
- 通用大模型调用结果与耗时；
- 模型版本查询与治理动作。
