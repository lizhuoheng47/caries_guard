可以继续开发 `analysis` 模块了，但你现在最需要注意的，不是“赶紧写接口”，而是先把这个模块的**边界、状态流转、幂等和数据落库职责**锁死。按你现有文档，`caries-analysis` 的职责非常明确：它只负责调用 Python AI 服务、创建分析任务、跟踪任务状态、保存业务侧 AI 结果摘要、接收分析完成事件、支撑医生修正回流，**不承担训练治理平台职责**。这点千万不能写歪。

你最先要守住的第一个问题，是**模块边界不能脏**。
 Java 侧只做业务编排，Python 侧才做推理、结果生成、可视化输出；Java 不要去重复实现模型逻辑，也不要把训练集管理、模型审批、标注治理这些内容偷偷塞进 `analysis` 模块。你们 V1 正式只保留 `ana_task_record`、`ana_result_summary`、`ana_visual_asset`、`ana_correction_feedback`，再加上风险评估结果承接表，这些都属于业务侧最小 AI 协同边界。

第二个问题，是**analysis 模块绝对不能脱离病例状态机单独乱跑**。
 你的分析链路必须服从主状态机：`QC_PENDING -> ANALYZING -> REVIEW_PENDING`，不能直接 `CREATED -> ANALYZING`，也不能 `ANALYZING -> REPORT_READY` 跳过医生复核。并且 `ANALYZING -> REVIEW_PENDING` 的前提是“必须存在成功 AI 摘要结果”。所有状态变化都必须经过应用服务统一流转，不能 Controller 直接改表，更不能 Mapper 在外层随便 update 状态。

第三个问题，是**回调幂等**，这会是你这个模块最容易被忽视、但最容易现场翻车的点。
 文档已经明确要求：所有任务必须可追踪、可重试、可审计，回调必须幂等；推荐用 `taskNo` 作为幂等键。也就是说，同一个成功回调重复打过来，系统应该“识别为已处理并直接返回成功”，而不是重复写 `summary`、重复插 visual asset、重复改病例状态。失败回调也必须能记录异常，而不是简单吞掉。这个点还是 P4 analysis 的正式验收项。

第四个问题，是**表职责要清，不要一张表塞所有结果**。
 推荐做法已经很清楚：
 `ana_task_record` 负责任务创建、状态跟踪、耗时、异常、请求体；
 `ana_result_summary` 负责业务侧可消费摘要；
 `ana_visual_asset` 负责热力图、mask、overlay 等可视化资产引用；
 `ana_correction_feedback` 负责医生修正回流。
 你最容易犯的错，是把完整 AI 结果、展示字段、回调状态、修正记录全部堆进 `ana_result_summary` 一张表里。那样后面 report、dashboard、followup 全都会被拖垮。

第五个问题，是**请求契约和回调契约必须先冻结，再写代码**。
 Java -> AI 的请求体里，只传 AI 需要的字段，不传不必要的明文敏感信息；`taskNo` 必须全局唯一，`modelVersion` 必须可追溯。AI -> Java 的回调接口也已经建议固定为 `POST /api/v1/internal/ai/callbacks/analysis-result`，并且至少要做网关白名单 + HMAC 签名。这个地方别一边写代码一边改字段名，不然 analysis 会成为你们后面返工最多的模块。

第六个问题，是**analysis 不是孤岛，它依赖前置模块质量**。
 如果 `image` 模块的质检、主图、附件关系没稳住，analysis 会被脏数据拖死。因为文档已经把触发分析的前置条件写死了：`QC_PENDING -> ANALYZING` 必须有“可分析影像”；而 image 模块还承担 `med_image_quality_check`、对象存储访问、主图规则这些基础能力。前置没打稳，analysis 开发越快，后面返工越狠。

第七个问题，是**当前仓库其实还没真正开始 analysis 业务实现**。
 现状不是没有工程，而是“表结构先行，协议冻结，业务未落地”。文档明确说了：analysis 模块 Java 代码尚未开始实现；同时系统里的权限、data scope、敏感数据保护也还没有完全生效。所以你现在开发时，不能默认这些基础能力已经完善，要有“当前仓库真实进度落后于文档目标态”的意识。

第八个问题，是**分层不能塌**。
 你这个模块特别容易写成“Controller 收回调 -> 直接 Mapper 落库 -> 顺手改 case 状态”。这在国一标准下是明显减分项。按开发说明书，Controller 只收参出参，App 层负责用例编排和事务边界，Domain 负责核心规则，Infrastructure 才负责 Mapper、MQ、HTTP 客户端和对象存储。analysis 模块越是异步、跨服务，越要守这个分层。

第九个问题，是**测试必须跟着设计走，不是最后补**。
 analysis 模块正式验收标准已经写得很明确：可创建 AI 任务、任务状态可追踪、AI 成功回调可写入摘要与资产、失败回调可记录异常、回调幂等、病例状态同步正确。推荐测试还要求覆盖：创建任务成功、成功回调写回成功、失败回调写回成功、重复回调幂等、修正反馈保存成功。也就是说，你一开始就该按这些场景设计接口和服务，不然后面补测试会发现结构根本不对。

我给你最真实的结论是：

**analysis 模块当前最大的风险，不是技术不会写，而是会写成“一个能跑的回调模块”，却不是“一个能撑住答辩的业务编排模块”。**

你接下来开发时，优先级应该这样排：

1. 先冻结 analysis 的 command / event / callback DTO
2. 再实现 `ana_task_record` 的创建与状态流转
3. 再实现回调幂等写入 `summary + visual asset + risk result`
4. 再接病例状态机 `ANALYZING -> REVIEW_PENDING`
5. 最后做医生修正回流 `ana_correction_feedback`

一句话说，**先把主链路做对，再做结果做全，再做展示做漂亮**。这才是你现在最该注意的。



## 一、模块定位

`caries-analysis` 只做这几件事：

- 创建分析任务
- 查询任务状态
- 接收 AI 成功/失败回调
- 落库 `ana_task_record / ana_result_summary / ana_visual_asset / ana_correction_feedback`
- 回写 `med_risk_assessment_record`
- 推动病例状态从 `QC_PENDING -> ANALYZING -> REVIEW_PENDING`

不要做这些：

- 不做模型训练管理
- 不做标注平台
- 不做模型审批系统
- 不在 Java 侧实现推理逻辑
- 不把 report / followup 逻辑硬塞进来

这条边界必须守住。

建议方法：

### 2. app 层

#### `AnalysisTaskAppService`

负责“创建任务”这个完整用例编排：

- 校验病例状态是否允许分析
- 校验影像是否满足分析条件
- 生成 `taskNo`
- 落 `ana_task_record`
- 更新病例状态为 `ANALYZING`
- 发布 `analysis.requested` 事件

#### `AnalysisCallbackAppService`

负责“处理回调”这个完整用例编排：

- 验签
- 按 `taskNo` 查任务
- 幂等判断
- 成功回调则更新 task 状态、落 summary / visual asset / risk record
- 失败回调则更新 task 状态、记录 error_message
- 成功后更新病例状态为 `REVIEW_PENDING`

#### `AnalysisQueryAppService`

负责查询聚合：

- 任务详情
- 摘要详情
- 分页查询

#### `CorrectionFeedbackAppService`

负责医生修正用例：

- 写入原始结果与修正结果
- 标记是否导出训练桥接
- 触发报告重生成或复核状态逻辑

------

### 4. domain service

#### `AnalysisTaskDomainService`

处理任务创建规则：

- 是否允许从当前病例状态触发分析
- 是否已有进行中的任务
- 是否需要重试
- `taskNo` 生成策略是否重复

#### `AnalysisCallbackDomainService`

处理回调规则：

- 成功/失败状态机
- 结果是否完整
- 视觉资产与风险评估是否允许为空
- 病例状态更新规则

#### `AnalysisIdempotencyDomainService`

这个类非常关键。

职责：

- 用 `taskNo` 判断是否已处理成功
- 重复成功回调直接返回
- 已失败是否允许受控重试
- 防止重复写 summary / visual assets / risk result

#### `CorrectionFeedbackDomainService`

处理修正规则：

- 是否允许在当前病例状态提交修正
- 修正内容是否完整
- 是否需要强制指定 `sourceImageId`
- 是否允许导出训练桥接

## 六、你开发时最该防的几个坑

### 1. 不要 Controller 直接改表

必须走 AppService，不然状态、事务、幂等全会散掉。

### 2. 不要把回调接口写成“收到就 insert”

必须先做：

- 验签
- 查 taskNo
- 幂等判断
- 状态校验
- 再落库

### 3. 不要把 raw_result_json 当万能字段

它可以存原始结果，但业务常用字段必须抽出来聚合，不然 report 和 dashboard 查询会越来越难看。

### 4. 不要让 analysis 模块直接依赖前端对象

必须使用 `Command / DTO / VO` 分离。

### 5. 不要绕过病例状态机

analysis 成功回写后直接把病例 update 成 `REPORT_READY` 这种操作，绝对不行。

## 八、我给你的最终判断

如果按国一标准，`analysis` 模块不是看你“能不能调通 AI”，而是看你能不能做成：

- 有任务主线
- 有状态机
- 有幂等
- 有结果承载
- 有修正回流
- 有可追溯审计

也就是说，它本质上是一个 **业务编排中枢模块**，不是一个“回调接收器”。
