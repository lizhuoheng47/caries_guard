Phase 4: Python 持久层工程化实施计划
Context
CariesGuard backend-python 当前依靠 MetadataRepository.ensure_schema() 在应用启动时一次性 CREATE TABLE IF NOT EXISTS 创建全部 20 张业务表(AI 运行域 4、RAG/知识库域 8、模型治理域 8)。

问题: schema 生命周期与应用启动耦合 — 没有版本控制、没有回滚轨迹、Docker/CI/Prod 多环境无法对齐、并行分支字段冲突无审计、ensure_schema() 内的 DDL 与团队后续迭代会越来越脆弱。

目标(用户已确认):

完整执行 Phase 4 文档 §8 的 8 个 Step
Docker 采用 entrypoint 脚本自动 alembic upgrade head 后启动 uvicorn(方案 B)
同步 SQLAlchemy 2.x + PyMySQL(保持现有 def 路由零改造)
ensure_schema() 通过 CG_DB_SCHEMA_BOOTSTRAP_ENABLED 环境变量灰度下线(阶段 A)
最终形态: caries_ai 由 Alembic 管理版本,SQLAlchemy 2.x 提供统一 engine/session,20 张表分三个域 ORM 模型,Repository 按域拆分,新增字段/索引/表必须走迁移流程。

关键事实(已通过 Explore 调研确认)
当前唯一 Repository: metadata_repository.py(~870 行,内含全部 DDL)
表统计: 20 张(Phase 4 文档说 19,实际多一张 trn_training_run),全部 InnoDB + utf8mb4_unicode_ci
启动入口: main.py:26 → container.py:27 → MetadataRepository(settings) 触发 ensure_schema()
配置: config.py:67-72 已有 mysql_* 全套字段,数据库名 caries_ai
依赖: requirements.txt 仅 PyMySQL==1.1.1,无 sqlalchemy/alembic
Dockerfile: Dockerfile CMD ["python", "-m", "app.main"],无 entrypoint 脚本
测试: tests/unit/ 6 个文件,均无 DB fixture,无 conftest.py
Service 用 repository 的典型例子: rag_service.py、knowledge_service.py,事务边界目前由 repository 内部 _transaction() context manager 管理
实施路线(8 Step,对应文档 §8)
Step 1 — 基线梳理:对齐代码与库的真实结构
目标: 让 ORM/Alembic baseline 建立在确定的事实上。

动作:

启动 Docker 栈(docker compose up -d mysql),docker exec 进入 MySQL 对 caries_ai 每张表执行 SHOW CREATE TABLE、SHOW INDEX,落档为 backend-python/docs/13_caries_ai_baseline_schema.md
与 _table_statements() 和 _index_statements() 逐表逐索引比对,记录任何分歧(默认值、字符集、JSON vs TEXT、索引顺序)
输出:docs/13_caries_ai_baseline_schema.md(20 张表的完整字段+索引+约束清单),作为 Step 6 baseline 迁移的唯一依据
验证: 文档中每张表都标注"代码与库一致 / 仅代码 / 仅库",且分歧项有结论(以哪边为准)。

Step 2 — 引入 SQLAlchemy 基础设施
新增文件: backend-python/app/core/db.py

动作:

requirements.txt 追加 SQLAlchemy==2.0.36(同步,与 PyMySQL 兼容)
config.py 新增字段:
db_pool_size: int = int_env("CG_DB_POOL_SIZE", 5)
db_max_overflow: int = int_env("CG_DB_MAX_OVERFLOW", 10)
db_pool_recycle: int = int_env("CG_DB_POOL_RECYCLE_SECONDS", 1800)
db_echo: bool = bool_env("CG_DB_ECHO", False)
新增方法 def build_mysql_url(self) -> str: 返回 mysql+pymysql://{user}:{pass}@{host}:{port}/{db}?charset=utf8mb4
app/core/db.py 提供:
engine = create_engine(settings.build_mysql_url(), pool_pre_ping=True, pool_recycle=..., pool_size=..., max_overflow=..., echo=...)
SessionLocal = sessionmaker(bind=engine, autoflush=False, autocommit=False, expire_on_commit=False)
@contextmanager def session_scope() -> Iterator[Session]: — yield session,正常 commit / 异常 rollback / 总是 close
模块级 get_engine() / get_sessionmaker() 单例(用 lru_cache)
不动 MetadataRepository,新基础设施先与现有 PyMySQL 共存
验证: 启动 python -m app.main,SQLAlchemy 引擎初始化无错;新增单测 tests/unit/test_db.py 验证 session_scope() 能 SELECT 1。

Step 3 — ORM Base 与 Mixin
新增文件: backend-python/app/models/__init__.py、backend-python/app/models/base.py

动作:

base.py 定义:
class Base(DeclarativeBase): pass
class TimestampMixin: created_at、updated_at(Mapped[datetime],server_default=func.now())
class SoftDeleteMixin: deleted_flag: Mapped[str] = mapped_column(CHAR(1), default="0")
class StatusMixin: status: Mapped[str] = mapped_column(VARCHAR(32), default="ACTIVE")
class RemarkMixin: remark: Mapped[str | None] = mapped_column(VARCHAR(500))
class AuditMixin: created_by、updated_by、org_id(Mapped[int | None])
不强行给所有表上全部 mixin。日志类(rag_retrieval_log、llm_call_log、trn_dataset_sample、kb_document_chunk、ai_callback_log)只用 Timestamp/AuditLite,文档 §5.2、§3.2 已明确
验证: from app.models.base import Base, TimestampMixin 不报错。

风险与对应缓解
风险	缓解
Step 1 基线漏字段 → Step 6 baseline 错	Step 4 的反射对账测试是兜底 — Schema diff 不为空就拒绝进入 Step 6
Step 5 拆 Repository 时事务行为变化	每个 service 一次 PR + 跑 RAG E2E,变更最小化
Alembic baseline 注入到已有库时误执行 DDL	entrypoint 判断 alembic_version 存在性后选择 stamp 或 upgrade,首次注入手动 stamp
ORM expire_on_commit 默认 True 导致 service 拿到的 dict 过期	sessionmaker(expire_on_commit=False),且方法内立即转 dict 返回
Docker entrypoint 脚本权限/换行符(Windows 开发)	git config core.autocrlf=input,Dockerfile 中显式 chmod +x
