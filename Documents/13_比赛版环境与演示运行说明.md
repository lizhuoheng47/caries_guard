# 比赛版环境与演示运行说明

## 1. 文档目的

本文档只说明当前仓库已经存在、且可复现的比赛版运行路径。  
它不假设存在额外前端仓库，也不假设存在尚未提交的一键 seed 平台。

## 2. 当前推荐环境

- Docker Desktop 或等价 Docker Engine
- PowerShell 7+ 或 Windows PowerShell
- 可用端口：
  - `8080` Java
  - `8001` Python
  - `13306` MySQL
  - `16379` Redis
  - `5672` / `15672` RabbitMQ
  - `9000` / `9001` MinIO

## 3. 环境变量准备

根目录复制：

```powershell
Copy-Item .env.docker.example .env
```

比赛版建议至少确认以下参数：

```env
CARIES_COMPETITION_MODE_ENABLED=true
CARIES_ANALYSIS_MODEL_VERSION=caries-v1

CG_AI_RUNTIME_MODE=mock
CG_UNCERTAINTY_REVIEW_THRESHOLD=0.35
CG_RAG_KNOWLEDGE_VERSION=v1.0
CG_LLM_PROVIDER_CODE=MOCK
CG_LLM_MODEL_NAME=template-llm-v1
```

说明：

- `mock` 是当前最稳定、最容易复现的比赛演示模式；
- 如果已经验证过真实适配器，可切到 `hybrid`；
- 不建议在未验证充分前直接把比赛演示切到 `real`。

## 4. 启动步骤

```powershell
docker compose up -d --build
docker compose ps
```

健康检查：

```powershell
Invoke-RestMethod http://127.0.0.1:8080/actuator/health
Invoke-RestMethod http://127.0.0.1:8001/ai/v1/health
```

## 6. 推荐演示顺序

### 第一步：分析链路

运行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\phase5-analysis-docker-e2e.ps1 -SkipComposeUp -Phase5COnly -WaitSeconds 180
```

该脚本会：

- 生成脱敏病例、就诊、影像和分析任务
- 触发 Java -> RabbitMQ -> Python -> callback 闭环
- 产出低 uncertainty 与高 uncertainty 两类结果证据

### 第二步：RAG 链路

运行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\phase3-rag-docker-e2e.ps1 -SkipComposeUp
```

该脚本会：

- 生成最小 analyzed case
- 调用 Java 医生问答接口
- 校验 citations、retrieval log 和 llm log

### 第三步：人工复核与反馈

当前仓库已经具备 review / feedback 相关业务模型与接口，但默认 Docker 启动后不会自动生成固定的“医生修正样例”。  
因此比赛演示中应将这一步定义为：

- 基于高 uncertainty 样例进入复核语义
- 结合已有接口或集成页面做人工确认
- 强调“AI 辅助，不替代医生”

## 7. 关停与清理

```powershell
docker compose down
```

如需完全清理临时数据和对象存储卷：

```powershell
docker compose down -v
```

## 8. 当前边界说明

- 当前仓库最稳定的比赛演示方式是“Docker 全栈 + PowerShell E2E 脚本”。
- 当前没有额外提交 `competition-up.ps1/sh` 这类比赛专用包装脚本。
- 当前 demo 数据以脚本临时写入为主，因此每次运行生成的病例编号会不同。
