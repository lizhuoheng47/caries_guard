# 15. 比赛版验收与回归说明

为了确保在任何外部展台机、路演大屏上均能完美不出错地展示本工程，我们设置了自动与手工相结合的双重验收标准。推荐在连接大屏前 15 分钟跑通本回归。

## 1. 自动化探针验收 (Automated Acceptance)

在正式把鼠标交给来宾或评审前，直接执行代码库的根目录验收防错脚本：
```powershell
./scripts/competition-acceptance.ps1
# 或者对于 Linux/Mac 用户
./scripts/competition-acceptance.sh
```

成功判定标准必须在黑框终端中看到没有任何由于网络或缺失依赖抛红的报错，并且严格返回所有 `[PASS]` 检查项，特别注意新引入的**语义化校验**：
```text
[PASS] Evidence Directories Created
[PASS] Java Health
[PASS] Python Health
[PASS] Metrics Semantic Range Check [0, 1]
[PASS] Numerator/Denominator Logical Check
[PASS] Runtime Configuration Check
```
如果一切标记为绿色的 `PASS`，说明系统不仅连通，而且其 AI 治理数据（回调成功率、资产生成率、复核率等）均通过了严谨的逻辑检验（如分子必小于等于分母、比率介于0和1之间）。

## 2. 探针异常速查 (Fallback Handling)

如果您的验证脚本爆出 `FAIL` 警告：

- **若是 Java / Python Health [FAIL]**
  底层容器引擎大概率未启动或端口 `8080/8001` 被其他本地软件占用。
  **急救手段**：打开 Docker Dashboard 查看日志，或者执行 `./scripts/competition-reset.ps1` 清理后重启。

- **若是 Metrics Semantic / Numerator/Denominator Check [FAIL]**
  说明后台返回的 AI 治理统计数据存在逻辑漏洞（如缺少数据结构，或比率超过 100%）。
  **急救手段**：确保持续注入正确的模拟流水，或检查 Python 端（`caries_ai`）的 Alembic 表（如 `ai_callback_log`）是否正常映射。

- **若是 Runtime Configuration Check [FAIL]**
  说明系统没有正确连接到模型注册中心 `mdl_model_version` 或该表尚未初始化。
  **急救手段**：执行数据 seed 脚本补充内置的知识库与大模型注册记录。

## 3. 手工 UI 回归清单 (Manual Checklist)

虽然接口层面我们可以用脚本锁死，但前端路由可能会因缓存行为未如期映射：

1. **比赛级路由收拢确认**
   登入 `http://localhost:8080` (admin/123456)。
   侧边栏左侧**只能**看到 6 个比赛映射入口：
   - Cases & Imaging
   - AI Analysis Tasks
   - AI Result Detail
   - Knowledge & Citation
   - Review & Feedback
   - AI Runtime & Evaluation

2. **案例数据留存确认**
   点进 `AI Analysis Tasks`，列表中应包含并仅包含两条由 seed 赋予的关键记录（不应有真实环境中多达几十页的杂单）：
   - `C-LOW-UNCERTAINTY`
   - `C-HIGH-UNCERTAINTY`

3. **AI 治理看板的防御性展示确认 (Defensibility Check)**
   点进 `AI Runtime & Evaluation` 看板页时：
   - 确认各项比率旁必须有明确的分子/分母说明（如 `Review Completed: 12 / Suggested: 15`），以应对评审对数据造假的质疑。
   - 确认 Runtime Mode、Knowledge Version 与 LLM 模型名称准确显示，展现配置解耦的框架优势。
