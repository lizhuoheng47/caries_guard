# 15. 比赛版验收与回归说明

为了确保在任何外部展台机、路演大屏上均能完美不出错地展示本工程，我们设置了自动与手工相结合的双重验收标准。推荐在连接大屏前 15 分钟跑通本回归。

## 1. 自动化探针验收 (Automated Acceptance)

在正式把鼠标交给来宾或评审前，直接执行代码库的根目录验收防错脚本：
```powershell
./scripts/competition-acceptance.ps1
```

成功判定标准必须在黑框终端中看到没有任何由于网络或缺失依赖抛红的报错，并且严格返回：
```text
[PASS] Java Health
[PASS] Python Health
[PASS] Competition Mode Active
[PASS] Demo Case Seed Found
[PASS] Knowledge Version Initialized
[PASS] Analysis Pipeline Readiness
```
如果一切标记为绿色的 `PASS`，说明系统内所有的齿轮扣合完毕，您可以进行完整演示流。

## 2. 探针异常速查 (Fallback Handling)

如果您的验证脚本爆出 `FAIL` 警告：

- **若是 Java / Python Health [FAIL]**
  底层容器引擎大概率未启动或端口 `8080/8001` 被其他本地软件占用。
  **急救手段**：打开 Docker Dashboard 查看日志，或者执行 `./scripts/competition-reset.ps1` 清理后重启。

- **若是 Competition Mode [FAIL]**
  说明后台加载到了错误的常规配置文件，或您手工操作覆盖了。
  **急救手段**：检查外层是否挂载了 `env/competition.env` 并且未被 `.env` 内部的变量覆写。执行 `docker compose ...` 时必须指定 env-file。

- **若是 Demo Case / Knowledge Version [FAIL]**
  说明演示环境处于一个空转状态（挂载了空的 MySQL 库区或向量集没有写盘）。
  **急救手段**：这很可能是因为上次没有正常 shutdown 导致的挂载冲突。必须手工执行 `./scripts/competition-reset.ps1` 进行一次暴力裁切（包括 `-v` 清理卷），再重新 up。

## 3. 手工 UI 回归清单 (Manual Checklist)

虽然接口层面我们可以用脚本锁死，但前端路由可能会因缓存行为未如期映射：

1. **比赛级路由收拢确认**
   登入 `http://localhost:8080` (admin/123456)。
   侧边栏左侧**只能**看到 6 个比赛映射入口：
   - Cases & Imaging
   - AI Analysis Tasks
   - AI Result Detail
   - Knowledge & Citation
   - Review & Feedback
   - AI Runtime & Evaluation
   不应看见 System 或 Report Template 管理。

2. **案例数据留存确认**
   点进 `AI Analysis Tasks`，列表中应包含并仅包含两条由 seed 赋予的关键记录（不应有真实环境中多达几十页的杂单）：
   - `C-LOW-UNCERTAINTY`
   - `C-HIGH-UNCERTAINTY`

3. **模型透明度展示标志确认**
   点进 `AI Runtime & Evaluation` 看板页时。
   确认 Runtime Mode 显示为 `mock` 或 `hybrid`。这向评审证明这套 AI 引擎具备物理适配插件接入框架。
