# 比赛版验收与回归说明

## 1. 文档目的

本文档定义当前仓库已经具备的比赛版验收路径，以及哪些能力仍需人工补充检查。

## 2. 验收分层

### 2.1 基础环境验收

必须通过：

1. `docker compose up -d --build`
2. Java 健康检查通过
3. Python 健康检查通过
4. 管理员账号可以登录

健康检查命令：

```powershell
Invoke-RestMethod http://127.0.0.1:8080/actuator/health
Invoke-RestMethod http://127.0.0.1:8001/ai/v1/health
```

### 2.2 Java 工程回归

推荐至少执行：

```powershell
mvn -pl caries-boot -am -DskipTests compile
mvn -pl caries-system -am test -Dtest=AuthAppServiceTests,SystemAdminQueryAppServiceTests,SystemPermissionAuthorityServiceTests -Dsurefire.failIfNoSpecifiedTests=false
```

目标：

- 编译通过
- 比赛模式相关过滤逻辑通过

### 2.3 Python 工程回归

推荐至少执行：

```powershell
.\.venv\Scripts\python -m pytest tests\unit\test_rag_service.py tests\unit\test_inference_pipeline_phase5b.py tests\unit\test_schema_alias.py
```

目标：

- RAG 服务测试通过
- 推理链关键单元测试通过
- schema alias 契约通过

## 3. 自动化验收脚本

### 3.1 分析闭环

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\phase5-analysis-docker-e2e.ps1 -SkipComposeUp -Phase5COnly -WaitSeconds 180
```

通过标准：

- 能自动生成 fixture
- analysis task 能完成
- 能看到 low uncertainty 与 high uncertainty 两类结果
- callback 和 visual assets 校验通过

### 3.2 RAG 闭环

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\phase3-rag-docker-e2e.ps1 -SkipComposeUp
```

通过标准：

- 能自动生成最小 analyzed case
- Java 医生问答接口返回非空答案
- 至少返回一条 citation
- `rag_request_log`、`rag_retrieval_log`、`llm_call_log` 留痕存在

## 4. 需要人工补充验收的部分

当前以下部分仍建议人工验收：

1. 比赛模式下的菜单与页面暴露是否符合答辩口径
2. 高 uncertainty 样例的 review 提交流程
3. 医生修正反馈是否形成完整展示链
4. RAG 文案是否保持保守、未越权诊断

原因很简单：当前仓库的自动化主力仍集中在 analysis 和 RAG，review / feedback 的比赛演示还没有固定化为一键脚本。

## 5. PASS / FAIL 标准

### PASS

- 基础环境通过
- Java 编译通过
- Java 比赛模式测试通过
- Python 关键单元测试通过
- phase5 分析 E2E 通过
- phase3 RAG E2E 通过

### FAIL

任一以下情况都应视为失败：

- Java 或 Python 健康检查失败
- analysis task 无法闭环
- RAG 不返回 citation
- callback 不落库
- 比赛模式开启后核心链路被误伤

## 6. 失败后的恢复建议

1. 先执行 `docker compose ps` 检查容器状态
2. 检查 `.env` 中模型、知识和 competition mode 参数
3. 如数据污染明显，执行：

```powershell
docker compose down -v
docker compose up -d --build
```

4. 重新跑 `phase5` 与 `phase3` 脚本

## 7. 对外表述边界

验收文档中应避免以下夸大表述：

- “已完成临床级全面验证”
- “医生反馈闭环已经全自动化验收”
- “默认就是高精度真实模型演示”

当前更准确的说法应是：

- 已形成可复现的 analysis 与 RAG 验收路径
- 已形成比赛模式下的权限和菜单收束能力
- review / feedback 已纳入系统主线，但自动化验收仍在补强
