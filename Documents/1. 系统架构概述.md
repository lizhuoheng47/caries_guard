# 系统架构概述

本文档以 `E:\caries_guard\backend-java` 当前代码、`application-local.yml`、Flyway 和本地数据库 `cg` 为准。

## 1. 系统定位

当前项目是一个面向口腔筛查/复诊闭环的 Java 后端业务平台，核心目标不是单独展示模型能力，而是把以下链路串起来：

1. 登录鉴权
2. 患者建档
3. 就诊建档
4. 病例创建
5. 图像上传与质检
6. AI 分析任务创建
7. AI 回调写回
8. 风险结果承载
9. 报告生成
10. 随访触发与管理
11. 统计看板

当前工程结构采用：

> 模块化单体 Java 后端 + 外部 AI 服务协同接口 + RabbitMQ/日志两种分析消息发布模式

这里必须强调两点：
- 当前仓库里没有完整 Python AI 服务源码。
- 当前仓库里没有独立的模型治理平台或训练数据平台实现。

### 2.3 当前实际运行口径

本地配置文件：`caries-boot/src/main/resources/application-local.yml`

关键事实：
- 数据库：`cg`
- 分析消息模式：`rabbit`
- RabbitMQ 健康检查：启用
- 对象存储根目录：`${user.dir}/var/image-storage`
- 对象存储 provider code 默认值为 `MINIO`
- `MINIO` 由 `MinioObjectStorageService` 落地，`LOCAL_FS` 由 `LocalObjectStorageService` 落地

这意味着：
- 文档应写成“当前 Java 后端已经落地 MinIO provider”。
- E2E 环境使用 `LOCAL_FS`，本地共享卷联调也可以切到 `LOCAL_FS`。
- `provider-code` 与实现类已经对齐：`MINIO` -> `MinioObjectStorageService`，`LOCAL_FS` -> `LocalObjectStorageService`。

## 3. 与来源设计稿的统一修正

### 3.1 对象存储

来源稿常写：
- MinIO
- 对象存储服务独立可用

当前真实实现：
- `caries-image` 中由 `ObjectStorageService` 抽象统一对象读写，默认 `MINIO` 走 `MinioObjectStorageService`，`LOCAL_FS` 走 `LocalObjectStorageService`
- 路径结构为 `attachments/yyyy/MM/dd/md5.ext`
- 通过 `ObjectStorageService` 作为抽象接口

最终文档口径：
- 保留“对象存储抽象层”的设计
- 明确“当前 local profile 默认 provider 为 `MINIO`，E2E profile 使用 `LOCAL_FS`”
- MinIO 已落地；OSS 仍只是 provider 预留口径

### 3.2 Risk 模块

来源 UML 中经常把 `Risk` 画成独立业务模块。

当前真实实现：
- 风险评估结果由分析回调时写入 `med_risk_assessment_record`
- 后续由报告、随访、看板共同消费
- 没有独立的 `caries-risk` 模块
- 没有独立 Risk 管理 API

最终文档口径：
- 保留“风险评估能力”
- 不保留“当前已落地独立 Risk 模块”的描述

### 3.3 模型治理 / ModelAdmin

来源设计中的治理原则是合理的：
- 候选模型
- 离线评估
- 专家审核
- 人工审批上线
- 模型资产治理

当前真实实现：
- `ana_task_record` 中已记录 `model_version`
- AI 回调 DTO 中带 `modelVersion`
- dashboard 有 `model-runtime` 统计摘要
- 但没有独立的 ModelAdmin 模块、表结构、管理端入口、独立 API

最终文档口径：
- 保留来源中的治理原则
- 明确“当前 Java 业务平台尚未实现独立的 ModelAdmin 模块”
- 可以说“已有轻量模型版本登记与运行统计能力”，但不能说“已完成独立 ModelAdmin 平台”

### 3.4 部署图

来源部署图常写：
- Docker Compose
- Frontend
- AI Service
- MQ
- MinIO
- 多服务联动

当前真实可确认的事实：
- Java 后端本地 profile 可运行
- MySQL / Redis / RabbitMQ 为依赖
- 本地对象存储是文件系统
- 仓库中没有完整前端与 Python AI 服务实现

最终文档口径：
- 可以保留“目标部署形态”作为扩展方案
- 但“当前已验证部署形态”只能写 Java 后端本地 profile + 本地依赖

## 5. 当前主数据流

### 5.1 病例到分析

1. 创建患者
2. 创建就诊
3. 创建病例
4. 上传附件
5. 创建病例图像
6. 保存图像质检
7. 创建分析任务
8. Java 发布 `analysis.requested`
9. 外部 AI 服务消费并回调

### 5.2 分析到报告/随访

1. AI 回调 `SUCCESS`
2. Java 写 `ana_result_summary`
3. Java 写 `ana_visual_asset`
4. Java 写 `med_risk_assessment_record`
5. 病例推进到 `REVIEW_PENDING`
6. 报告生成
7. 若高风险或建议复查，则自动触发随访

> 一个已完成核心业务闭环的 Java 口腔筛查后端平台，已经具备病例、图像、分析回调、风险结果、报告、随访和看板能力；同时保留了模型治理、训练数据治理和目标部署形态的设计原则，但这些治理能力当前尚未在本仓库中落地为独立模块或完整平台。
