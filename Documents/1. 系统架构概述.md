# 1. 系统架构概述

## 1. 产品定位

CariesGuard 是面向口腔龋病筛查、报告生成、复核和随访管理的辅助决策平台。系统以 Java 业务平台为主链，以 Python AI/RAG 服务为智能能力层，以 MySQL、MinIO、RabbitMQ、Redis 作为基础设施。

当前 AI 路线：

- 不采用业务专用大模型微调作为主路线；
- 采用通用大模型 + 知识图库 + 结构化病例上下文的 RAG 方案；
- 影像 AI 使用可替换算法适配器输出结构化结果；
- 业务判断以规则、阈值、复核和审计闭环保证可靠性。

## 2. 总体架构

```text
Web / App / Internal Client
  -> Java Backend
      -> caries_biz
      -> Redis
      -> MinIO
      -> RabbitMQ
      -> Python AI/RAG
          -> caries_ai
          -> MinIO
          -> Vector Index
          -> General LLM Provider
```

## 3. 服务分工

Java 后端：

- 用户、组织、权限；
- 患者、就诊、病例、影像、附件；
- AI analysis task；
- callback 落库；
- 报告、复核、随访；
- RAG 调用入口和业务展示。

Python 后端：

- AI 推理流水线；
- visual asset 生成；
- RAG 知识检索；
- 通用大模型调用；
- AI/RAG 日志；
- 模型和知识版本治理。

## 4. 核心链路

### 4.1 影像分析链路

```text
病例影像上传
  -> Java 创建 analysis task
  -> RabbitMQ
  -> Python InferencePipeline
  -> callback
  -> Java 落库并推进报告/复核
```

### 4.2 RAG 问答链路

```text
医生/患者问题
  -> Java 鉴权与上下文组装
  -> Python RAG
  -> 知识图库检索
  -> 通用大模型生成
  -> 引用和安全约束
  -> Java 返回
```

## 5. AI 可靠性策略

- 结构化结果优先，文本解释次之；
- 高 uncertainty 触发复核；
- RAG 回答必须依赖知识引用；
- 通用大模型不得直接改写业务状态；
- 所有 AI 结果必须可追溯；
- `real` 模式失败必须显式失败。

## 6. 数据库

- `caries_biz`：Java 业务库；
- `caries_ai`：Python AI/RAG 运行库。

两库通过任务号、病例号、影像 ID 建立弱引用，不跨库迁移。

## 7. 对象存储

MinIO bucket：

- `caries-image`：原始影像；
- `caries-visual`：MASK/OVERLAY/HEATMAP；
- `caries-report`：报告文件；
- `caries-export`：导出文件。

Java 是附件主数据权威，Python 只生成并回传 visual asset 元数据。
