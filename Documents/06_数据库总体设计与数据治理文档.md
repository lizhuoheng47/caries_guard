# 数据库总体设计与数据治理文档

更新日期：2026-04-15

本文档以 Flyway 单基线和当前 Java 代码为准，说明业务库结构、数据治理边界、模型治理落点和训练数据治理口径。

## 1. 总体结论

1. 当前业务平台库已经覆盖运行闭环，总表数为 38 张。
2. 当前 Flyway 迁移为单基线 `V001__baseline_schema.sql`。
3. 基线已补齐模型版本登记表、AI trace/推理耗时字段、修正反馈训练准入字段、角色/菜单/数据权限种子、visual trace、attachment object key 治理和 analysis 数据字典冻结。
4. 业务平台库比较完整，但训练治理库、标注平台、完整 ModelAdmin 平台仍主要是设计与流程规范，没有独立业务模块。
5. 风险评估不独立成模块，落在 `med_risk_assessment_record` 和 analysis/report/followup/dashboard 链路中。
6. 对象存储默认使用 MinIO，数据库通过 `med_attachment.storage_provider_code` 保留 provider 信息。

## 4. 核心业务表说明

### 4.1 `med_attachment`

用途：附件元数据，不存二进制内容。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `bucket_name` | MinIO bucket 或 LOCAL_FS bucket 目录 |
| `object_key` | 对象 key |
| `storage_provider_code` | `MINIO` 或 `LOCAL_FS` |
| `md5` | 文件摘要，用于去重和训练样本追溯 |
| `access_level_code` | 访问级别 |
| `org_id` | 机构隔离 |

当前 MinIO 已实现，`storage_provider_code=MINIO` 不是设计占位。

### 4.2 `ana_task_record`

用途：AI 分析任务主表。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `task_no` | 跨系统任务编号 |
| `case_id` | 病例 ID |
| `patient_id` | 患者 ID |
| `task_status_code` | 任务状态 |
| `model_version` | 模型版本 |
| `retry_from_task_id` | 重试来源任务 |
| `trace_id` | Python AI 服务 trace ID |
| `inference_millis` | 推理耗时 |
| `started_at` | 任务开始时间 |
| `completed_at` | 任务完成时间 |

### 4.3 `ana_result_summary`

用途：AI 结果摘要和原始 JSON。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `task_id` | 分析任务 ID |
| `summary_json` | 摘要 JSON |
| `raw_result_json` | 原始模型结果 |
| `overall_highest_severity` | 最高严重程度 |
| `uncertainty_score` | 不确定性评分 |
| `review_suggested_flag` | 是否建议复核 |

### 4.4 `ana_correction_feedback`

用途：医生修正反馈，也是训练候选治理入口。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `case_id` | 病例 ID |
| `task_id` | 分析任务 ID |
| `correction_content_json` | 修正内容 |
| `is_exported_for_train` | 是否已导出训练 |
| `training_candidate_flag` | 是否训练候选 |
| `desensitized_export_flag` | 是否脱敏导出 |
| `dataset_snapshot_no` | 数据集快照号 |
| `review_status_code` | 审核状态 |
| `reviewed_by` | 审核人 |
| `reviewed_at` | 审核时间 |

### 4.5 `ana_model_version_registry`

用途：最小模型版本治理表，不等同于完整 ModelAdmin 平台。

字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `model_code` | 模型编码 |
| `model_version` | 模型版本 |
| `model_type_code` | 模型类型 |
| `approved_flag` | 是否批准上线 |
| `approved_by` | 审批人 |
| `approved_at` | 审批时间 |
| `status` | `CANDIDATE`、`EVALUATING`、`APPROVED`、`RETIRED` |
| `remark` | 备注 |
| `created_at`、`updated_at` | 审计时间 |

数据库基线已初始化 `caries-detector / caries-v1`。

## 5. 权限与菜单数据

数据库基线已初始化以下内容：

| 类型 | 内容 |
| --- | --- |
| 角色 | `ORG_ADMIN`、`DOCTOR`、`SCREENER` |
| 菜单 | 患者、就诊、病例、影像、分析任务、报告、随访、看板、AI 运行看板 |
| 权限点 | `patient:view`、`visit:view`、`case:view`、`image:view`、`analysis:view`、`report:view`、`followup:view`、`dashboard:view`、`dashboard:ops:view` |
| 数据权限 | ORG_ADMIN 为 ORG，DOCTOR/SCREENER 多数为 SELF，看板为 ORG 聚合 |
| 列脱敏 | `patientName`、`phone`、`idCard`、`originalName` 等字段在种子规则中配置 masking policy |

## 6. 高敏字段保护

当前业务库采用以下模式：

| 模式 | 例子 | 说明 |
| --- | --- | --- |
| 密文列 | `real_name_enc`、`phone_enc`、`id_card_enc` | 保存加密值 |
| Hash 列 | `phone_hash`、`id_card_hash` | 用于精准查询/去重 |
| 脱敏列 | `phone_masked`、`real_name_masked` | 用于前端展示 |
| 机构隔离 | `org_id` | 各业务表主要隔离字段 |
| 逻辑删除 | `deleted_flag` | 大部分管理和业务表使用 |

训练导出不得使用密文字段和可逆身份信息。

## 7. MinIO 与附件治理

当前对象存储实现：

| provider | 实现类 | 状态 |
| --- | --- | --- |
| `MINIO` | `MinioObjectStorageClient` | 已实现，默认启用 |
| `LOCAL_FS` | `ImageObjectStorageServiceAdapter` | 已实现，e2e/兼容使用 |
| `OSS` | 无实现 | 仅可作为未来扩展 |

数据库层通过 `med_attachment.storage_provider_code` 标记真实 provider。Python 和训练导出程序应优先使用 Java 生成的 `accessUrl` 或 MinIO SDK，不应把 `objectKey` 当本地路径。

## 8. 模型治理边界

已落地：

1. `ana_model_version_registry`
2. `ana_task_record.model_version`
3. `ana_task_record.trace_id`
4. `ana_task_record.inference_millis`
5. `/api/v1/dashboard/model-runtime` 模型运行看板
6. `ana_correction_feedback` 训练准入字段

未落地：

1. 独立 ModelAdmin 模块。
2. 模型包上传和自动发布平台。
3. 标注平台。
4. 训练数据集快照管理 UI。

## 9. 数据治理原则

保留以下治理原则：

1. 业务数据与训练数据边界必须清晰。
2. 高敏字段不得直接导出到训练流程。
3. 训练候选必须经过审核。
4. 每个训练快照必须保留来源病例、影像、任务、模型版本和修正记录。
5. 模型版本上线应保留审批记录。
6. Dashboard 应按模型版本统计任务数、成功率、失败率、平均推理耗时、不确定性和修正反馈。

## 10. 当前边界说明

1. 当前业务库完整度较高，可以支撑平台演示和 Java/Python 联调。
2. 当前训练治理能力是最小落点，不是完整 MLOps 平台。
3. 当前风险评估能力在分析链路中，不是独立风控系统。
4. 当前 MinIO 需要运行外部 MinIO 服务，Java 代码已经实现 provider，但不会自动启动 MinIO 进程。
