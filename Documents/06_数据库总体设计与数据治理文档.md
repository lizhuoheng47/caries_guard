# 数据库总体设计与数据治理文档

本文档同时包含两层信息：
- 当前业务平台数据库已经实现的内容
- 来源设计稿中应保留、但当前尚未落地为独立平台的治理原则

## 1. 文档边界

本文档覆盖：
- `backend-java` 当前业务平台数据库
- 当前数据治理能力
- 当前数据隔离、敏感数据保护、审计与迁移机制
- 对训练数据治理的建议接口与扩展方向

本文档不把以下能力写成“当前已完成”：
- 标注平台
- 训练数据快照平台
- 模型资产治理平台
- 独立 ModelAdmin 模块
- 独立 Risk 模块

## 2. 当前数据库现状

### 2.1 基线

当前本地数据库：`cg`

当前真实状态：
- Flyway 版本到 `V013`
- 总表数 `38`
- 业务平台运行主链所需表已经比较完整
- 默认数据库只有系统种子数据，没有业务样例数据

### 2.2 当前真实数据量结论

本地库当前可确认：
- `sys_user = 1`
- `sys_role = 1`
- `sys_user_role = 1`
- `sys_menu` 由 Flyway `V014__14_minio_model_governance_and_permissions.sql` 初始化业务菜单
- 患者/病例/图像/分析/报告/随访相关表基本为空

结论：
- 数据库结构已比较完整
- 数据内容不适合作为开箱即用演示样例库

## 4. 当前数据库设计原则

### 4.1 当前已落实的原则

1. `org_id` 作为机构隔离基础字段
2. `deleted_flag` 作为逻辑删除标记
3. 统一审计字段：`created_by / created_at / updated_by / updated_at`
4. 敏感字段加密、哈希、脱敏并行存储
5. 通过 Flyway 管理结构版本
6. 通过日志表保留登录和操作轨迹

### 4.2 当前部分落实、仍需推进的原则

1. 数据权限矩阵已存在框架，但未完全下推到业务查询层
2. 模型版本已留痕，但未形成完整治理资产
3. 医生修正反馈已入库，但未形成训练准入治理闭环

## 5. 来源设计稿中应该保留的治理原则

下面这些原则是对的，应继续保留：
- 业务数据与训练数据边界分离
- 高敏字段保护
- 列级脱敏
- 数据权限矩阵
- 训练数据集快照
- 标注与质控流程
- Dataset Card
- Gold Set
- 人工审批上线

但必须统一结论：
- 当前业务平台库已经比较完整
- 训练治理库、标注平台、模型资产治理仍主要停留在设计与流程规范层
- 不能把两者混成“都已经开发完成”

## 6. 当前真实存储与对象数据治理

### 6.1 当前事实

- 文件元数据保存在 `med_attachment`
- 病例图像业务记录保存在 `med_image_file`
- 当前 `MINIO` provider 由 `MinioObjectStorageService` 完成对象读写，`LOCAL_FS` provider 由 `LocalObjectStorageService` 完成本地落盘
- local 配置中的 provider code 已明确为 `MINIO`，并有真实 MinIO SDK 实现

### 6.2 文档口径要求

- 可以保留“对象存储抽象层”的设计表述
- 可以写成“当前 Java 后端已支持并默认使用 MinIO provider；实际文件是否进入 MinIO 取决于运行环境中的 MinIO 服务配置”
- 应明确：当前 local 环境真实落地是本地文件系统

## 7. 当前风险评估数据治理口径

来源稿中把 Risk 画成独立模块，这会误导。

当前真实情况：
- 风险结果由 `caries-analysis` 在回调成功时写入 `med_risk_assessment_record`
- 报告、随访、dashboard 消费该表
- 没有独立 Risk 模块和独立 Risk 管理表群

正确表述：
- 当前已实现风险评估结果承载能力
- 当前未实现独立风险管理平台

## 9. 医生修正反馈与训练候选治理

### 9.1 当前已实现

当前业务平台已存在：
- `ana_correction_feedback`

当前意义：
- 保存医生对分析结果的修正反馈
- 为后续训练候选样本筛选提供业务留痕

### 9.2 当前未实现

未实现的内容：
- 是否可作为训练样本的准入标记
- 脱敏导出审批
- 数据集快照编号
- 审核状态与审核人留痕

### 9.3 建议补强字段

建议在 `ana_correction_feedback` 增加：
- `training_candidate_flag`
- `desensitized_export_flag`
- `dataset_snapshot_no`
- `review_status_code`
- `reviewed_by`
- `reviewed_at`

这样才能把来源中的：
- 修正回流
- 脱敏审批
- 训练快照

真正落到业务承载层，而不是只停留在文档里。

## 12. 当前数据库部署结论

当前能确认的部署事实：
- Java 后端连接单库 `cg`
- E2E 使用独立库 `cg_e2e`
- 本地依赖 MySQL / Redis / RabbitMQ
- 对象数据默认落本地文件系统

不能写成当前已实现的事实：
- 生产级多节点数据库拓扑
- 训练治理独立数据库
- MinIO 存储集群
- 全家桶 Docker Compose 交付物

## 13. 最终统一结论

当前数据库与治理口径应这样表述：

> 业务平台数据库已经能够完整支撑病例、图像、分析、风险、报告、随访、审计与统计闭环；同时保留训练数据治理、模型治理、数据快照与人工审批上线等设计原则，但这些治理能力当前尚未在业务平台数据库中落地为完整独立体系。
