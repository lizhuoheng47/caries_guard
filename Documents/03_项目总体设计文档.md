# 项目总体设计文档

本文档以当前 `backend-java` 代码为准，兼容保留来源设计稿中有价值的治理原则，但严格区分：
- 当前已落地
- 当前未落地但建议保留

## 1. 项目目标

当前项目的真实目标是交付一个可运行、可联调、可展示、可追溯的口腔筛查业务后端，而不是一个只展示 AI 推理结果的 Demo。

当前主线闭环：
1. 登录鉴权
2. 患者建档
3. 就诊建档
4. 病例创建
5. 图像上传与质检
6. AI 任务创建与回调
7. 风险结果承载
8. 报告生成
9. 随访触发与管理
10. 看板统计

## 3. 当前真实实现与来源设计稿的差异

### 3.1 存储层

来源写法：
- MinIO 对象存储

当前事实：
- `LocalObjectStorageService` 是 `LOCAL_FS` provider 的本地/E2E 实现；`MinioObjectStorageService` 是当前 `MINIO` provider 实现
- 文档只能写“对象存储抽象已建立，本地默认是本地文件系统”

### 3.2 风险评估

来源写法：
- 独立 `Risk` 模块

当前事实：
- 风险评估结果由 `caries-analysis` 回调写入 `med_risk_assessment_record`
- 看板和报告消费该结果
- 当前没有 `caries-risk`

### 3.3 模型治理

来源写法：
- 独立 `ModelAdmin` 模块

当前事实：
- 只有 `modelVersion` 留痕能力
- 没有独立模块、独立表、独立 API、独立前端入口

### 3.4 部署形态

来源写法：
- Docker Compose + Frontend + AI Service + MinIO + MQ

当前事实：
- 只确认 Java 后端 local profile 和本地依赖可运行
- 外部 AI 服务与前端未在本仓库完整交付

## 4. 当前主业务设计

### 4.1 患者/就诊/病例

- `PatientCommandAppService` 负责建档与敏感数据保护
- `VisitCommandAppService` 负责就诊创建
- `CaseCommandAppService` 负责病例创建和状态流转
- `CaseStatusMachine` 负责病例状态校验

### 4.2 图像与质检

- `AttachmentAppService` 负责附件上传、签名访问
- `CaseImageAppService` 负责病例图像绑定与质检
- 分析任务只挑选 `qualityStatusCode = PASS` 的图像

### 4.3 AI 协同

- `AnalysisTaskAppService` 负责构造 `AiAnalysisRequestDTO`
- `RabbitAnalysisTaskEventPublisher` 在 `local` 下真实发布 Rabbit 事件
- `AnalysisCallbackAppService` 负责验签、幂等、写回、状态流转

### 4.4 报告与随访

- `ReportAppService` 负责生成报告和导出审计
- `FollowupTriggerService` 负责从报告结果自动触发随访
- `FollowupPlanAppService` / `FollowupTaskAppService` / `FollowupRecordAppService` 负责随访管理

### 4.5 看板

- `DashboardStatsRepository` 直接基于 SQL 做聚合
- 当前统计维度偏业务闭环，AI 治理维度仍偏少

## 5. 当前数据库设计结论

当前业务平台数据库特征：
- Flyway 已到 `V013`
- 本地 `cg` 库共有 `38` 张表
- 已完整覆盖系统、患者、病例、图像、分析、报告、随访、审计、看板所需运行主链
- 默认库只有系统种子数据，没有业务样例数据

必须统一的结论：
- 业务平台库已经比较完整
- 训练治理库、标注平台、模型资产治理仍主要停留在设计与流程规范层
- 不能把两者混成“都已经开发完成”

## 6. 非功能设计现状

### 6.1 当前已有

- JWT 鉴权
- 登录日志与操作日志
- 敏感数据加密、哈希、脱敏
- `org_id` 级别的数据隔离
- Spring Boot Actuator 健康检查
- Flyway 版本化迁移

### 6.2 当前不足

- 菜单种子缺失，非管理员角色演示不完整
- 数据权限规则框架存在，但业务模块下推不完整
- PDF 报告只支持极简 ASCII 输出
- 对象存储 provider 命名与实际实现存在偏差

## 7. 必须保留的治理原则

以下是来源稿里值得保留的部分，但要标注为“治理目标”，不是“当前已实现”：

1. 业务数据与训练数据边界分离
2. 候选模型离线评估与人工审批上线
3. 高敏字段保护与列级脱敏
4. 数据快照、Dataset Card、Gold Set
5. 医生修正结果回流训练候选样本
6. 模型版本可追溯、可统计、可审计

## 8. 按模块的建议补强清单

### 8.5 `caries-dashboard`

## 9. 当前交付级结论

当前项目已经具备：
- 真实业务闭环
- 分析协同接口
- 风险结果承载
- 报告与随访主链
- 看板统计

当前项目尚不应被描述为：
- 已完成完整模型治理平台
- 已完成完整训练数据治理平台
- 已完成独立 Risk 模块
- MinIO provider 已在 Java 后端落地；MinIO 服务实例和部署编排需由运行环境提供
- 已完成完整 Docker Compose 全家桶交付版
