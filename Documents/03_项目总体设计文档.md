# 03 项目总体设计文档

## 1. 总体目标

CariesGuard 提供龋病筛查、AI 辅助分析、报告生成、复核和随访管理能力。系统强调可追溯、可复核、可解释和可迭代，而不是直接用 AI 替代医生诊断。

## 2. 技术路线

AI 技术路线统一为：

- 影像结果由算法适配器输出结构化结果；
- 自然语言解释、问答和报告草稿由通用大模型生成；
- 通用大模型必须接入知识图库和病例上下文；
- 不以业务专用微调大模型作为当前主路线；
- 高不确定性结果进入复核。

## 3. 系统模块

| 模块 | 说明 |
| --- | --- |
| 用户权限 | 账号、角色、组织、审计 |
| 患者病例 | 患者、就诊、病例 |
| 影像附件 | MinIO 附件、影像文件 |
| AI 分析 | 任务、回调、summary、visual assets |
| 报告 | 报告草稿、确认、导出 |
| 复核 | review pending、医生复核 |
| 随访 | followup plan、提醒和结果 |
| RAG | 知识库、检索、通用大模型回答 |
| 治理 | 模型/算法/知识版本、评估、审批 |

## 4. AI 分析设计

`InferencePipeline` 只负责编排，业务规则下沉到子流水线：

- `QualityPipeline`
- `DetectionPipeline`
- `SegmentationPipeline`
- `GradingPipeline`
- 后续 `RiskPipeline`

所有子流水线支持 `mock / hybrid / real`。

## 5. RAG 设计

RAG 由以下部分组成：

- 知识库管理；
- 文档切片；
- 向量检索；
- 结构化病例上下文；
- 通用大模型调用；
- 引用和安全约束；
- 请求与检索日志。

RAG 输出不能直接更新病例、报告或复核状态，只能作为辅助说明。

## 6. 数据设计

- Java 业务库：`caries_biz`
- Python AI/RAG 库：`caries_ai`
- MinIO：影像、visual、报告、导出
- Redis：缓存和会话
- RabbitMQ：analysis task 异步消息

## 7. 安全与合规

- 患者敏感信息不进入 Python 日志；
- RAG prompt 不包含非必要明文隐私；
- callback 签名校验；
- MinIO bucket 私有；
- 结果保留审计；
- 医疗结论必须保留医生复核入口。
