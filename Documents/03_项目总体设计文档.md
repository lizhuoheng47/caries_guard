## 一、修订说明与设计原则

本版文档在原始方案基础上进行了重构，核心目标不是继续堆砌概念，而是提升以下三项能力：

1. **可落地性**：删去高风险但当前难以落地的过度承诺，保留真正能实现、能演示、能解释、能复盘的能力。
2. **可答辩性**：所有主张都要求对应到清晰的证据链，避免“写得很强，现场拿不出”。
3. **软件赛道匹配度**：强调平台化、业务闭环、可信可追溯和工程完整度，而不是把作品做成单纯的模型展示系统。

### 1.1 本次修订的核心动作

- 将原有“5 个并列创新点”压缩为“**2 个核心创新 + 1 个平台级亮点**”。
- 将“自动切换主模型”修订为“**离线评估 + 人工审批上线**”，更符合医疗软件可信要求。
- 增加“**业务数据与训练数据边界设计**”专章，解决原方案中业务库与训练集边界不够清晰的问题。
- 统一架构口径，明确当前落地架构为“**模块化单体 + 独立 AI 服务 + RabbitMQ 解耦**”，不再在主文档中强调过重的生产级集群方案。
- 补充“证据链与验收指标”，把评审重点从“愿景”转换为“证据”。

### 1.2 国一版设计原则

- **平台优先于模型堆砌**：先证明系统能支撑真实筛查流程，再证明模型增强了系统能力。
- **可信优先于炫技**：所有结论必须能解释、能复核、能追溯。
- **闭环优先于单点**：识别、分级、预警、报告、复核、回流必须形成业务闭环。
- **收敛优先于贪大**：主叙事必须清楚，避免多条主线并行导致评委抓不到重点。

## 二、项目概述

### 2.1 项目简介

龋病是全球最常见的口腔慢性疾病之一，具有发病率高、早期不易察觉、基层筛查依赖医生经验、随访管理薄弱等问题。传统龋齿相关 AI 工作往往只关注“是否识别到病灶”，但真实筛查场景更关注的是：

- 哪颗牙有问题；
- 病灶是否可信；
- 是否需要医生复核；
- 风险等级如何；
- 如何形成患者可理解的报告；
- 后续如何随访并让系统持续改进。

本项目设计并实现一套面向真实口腔筛查流程的**多模态龋齿智能识别与分级预警平台**。平台以全景片、根尖片、口内照为核心输入，结合患者基础行为与病史等结构化信息，实现“**筛查—分级—预警—报告—复核—回流**”一体化闭环。

### 2.2 项目目标

#### 技术目标

- 构建面向牙位先验的龋病分割模型，提升复杂口腔影像下的病灶定位能力。
- 构建具备不确定性感知能力的分级模型，使系统能识别“模型不应过度自信”的场景。
- 构建影像结果与结构化风险因子融合的风险预警模块，提升平台决策完整性。

#### 工程目标

- 交付一套可部署、可演示、可扩展的 B/S 平台。
- 完成医生端、筛查端、管理端核心业务流程闭环。
- 实现医生修正结果回流、模型版本管理、报告归档和审计追踪能力。

#### 应用目标

- 支持基层门诊、学校筛查、社区医疗、体检中心等场景。
- 提升筛查效率、结果一致性和高风险患者识别能力。
- 为后续扩展牙周病、根尖周病等口腔疾病提供架构基础。

### 2.3 预期交付物

| 交付类型 | 内容 |
|---|---|
| 软件系统 | React 前端 + Spring Boot 后端 + Python AI 服务 + Docker Compose 部署方案 |
| 模型与算法 | TPC-Net 分割模型、EDL 分级模型、多模态风险评估模块 |
| 数据资产 | 脱敏后的训练数据集、标注规范、数据版本清单、数据卡片 |
| 工程文档 | 总体设计、UML 图集、API 文档、部署手册、用户手册、答辩讲稿 |
| 答辩资产 | 端到端演示脚本、关键病例示例、错误分析案例库、消融实验表 |

## 三、项目定位与核心价值

### 3.1 一句话定位

**面向真实口腔筛查流程的可信辅助决策平台，而不是单纯的龋齿识别 Demo。**

### 3.2 与普通“识别系统”的本质区别

| 维度 | 普通识别系统 | 本项目 |
|---|---|---|
| 输出 | 是否有龋病 | 病灶定位、分级、不确定性、风险等级、报告与复核建议 |
| 用户 | 只面向技术演示 | 面向医生、筛查人员、管理者 |
| 流程 | 一次性推理 | 筛查、复核、报告、回流、随访闭环 |
| 可信性 | 给结果 | 给结果、给依据、给不确定性、给审计链 |
| 生命周期 | 模型展示 | 平台运行 + 版本迭代 + 人工修正回流 |

### 3.3 目标用户

| 用户角色 | 核心诉求 | 系统价值 |
|---|---|---|
| 口腔医生 | 提升阅片效率与一致性 | 病灶高亮、分级辅助、不确定性提示、复核入口 |
| 筛查人员 | 快速完成大批量初筛 | 批量上传、自动分析、可疑病例筛出 |
| 机构管理者 | 掌握整体筛查和高危分布 | 统计看板、趋势分析、随访追踪 |
| 科研管理员 | 管理数据回流和模型版本 | 数据版本管理、模型评估记录、训练样本回流 |
| 患者/家属 | 看懂结果并接受建议 | 通俗化报告、复查建议、日常护理建议 |

### 3.4 核心价值主张

1. **对医生**：不是替代，而是提升阅片效率、降低漏诊风险。
2. **对筛查**：把筛查结果从“肉眼经验”提升为“可追踪、可解释、可复核”的流程化输出。
3. **对机构**：把散落的病例、影像、风险因子和报告统一纳入平台闭环。
4. **对比赛**：兼具临床价值、算法深度、软件工程完整度和可信性设计。

## 四、国一版主创新设计

本项目最终只保留以下三张“主牌”，其余能力降级为系统功能或辅助亮点，避免创新点过多导致叙事分散。

### 创新一：TPC-Net —— 基于牙位感知与跨视图一致性的龋病分割模型

#### 问题动机

传统分割模型通常把口腔影像当作通用自然图像处理，忽略了两个关键临床先验：

- 龋病在不同牙位上的发生模式和表现形态存在差异；
- 同一颗牙在全景片、根尖片、口内照中的信息具有互补性。

#### 技术方案

- 使用 YOLOv8 完成牙位检测与 FDI 编号映射；
- 引入 **Tooth Position Embedding（TPE）** 将牙位先验注入分割主干中层特征；
- 引入 **Cross-View Consistency（CVC）** 约束，使同一颗牙在多视图中的预测更一致；
- 分割主干采用 SegFormer-B2，兼顾效果与工程复杂度。

#### 竞赛价值

- 有技术辨识度，便于做消融实验；
- 能解释为什么比“直接上 UNet”更适合本任务；
- 既有研究深度，又能落到软件系统的牙位级可视化结果展示上。

### 创新二：EDL 分级 —— 面向临床复核流程的不确定性感知分级机制

#### 问题动机

医疗辅助决策系统最危险的情况不是“模型不知道”，而是“模型错了却非常自信”。

#### 技术方案

- 将传统 softmax 分类头替换为基于 Dirichlet 参数化的 Evidential Deep Learning 分级头；
- 同时输出严重程度分级结果和不确定性分数；
- 将不确定性分数直接嵌入业务流程：
  - 高不确定性样本自动进入“待复核队列”；
  - 高不确定性样本优先纳入人工修正池；
  - 报告文本在高不确定性场景下使用更保守表述。

#### 竞赛价值

- 强调“可信 AI”而不是“模型炫技”；
- 能与医生复核、报告生成、修正回流形成工程闭环；
- 答辩时更容易回答“为什么医生会信任你们的系统”。

### 创新三：医生修正回流闭环 —— 让系统从一次性 Demo 走向可演进平台

#### 问题动机

多数参赛作品只做一次性推理，无法证明系统具备持续优化能力。

#### 平台方案

- 医生对分割边界、牙位、分级结果进行修正；
- 修正结果进入 `correction_feedback` 表和“候选训练样本池”；
- 训练侧仅使用经过脱敏和审批的数据副本，不直接使用业务库原始数据；
- 候选模型经过离线评估、专家审核后人工批准上线；
- 所有版本具备注册、评估、审批、回滚记录。

#### 竞赛价值

- 体现软件赛道最看重的“产品闭环”；
- 让平台具备可成长性和可信追踪能力；
- 能把“业务流程”和“模型演化”真正串起来。

### 4.1 非主创新但保留的加分能力

以下内容保留，但不再作为并列主创新讲述：

- SHAP 特征贡献解释；
- 患者版通俗报告；
- 风险变化趋势图；
- 随访计划生成；
- 健康行为模拟建议。

其中“健康行为模拟建议”定位为**患者教育辅助**，不表述为医疗治疗建议，不作为核心技术卖点。

## 五、总体技术架构

### 5.1 架构模式

本项目当前落地架构明确为：

> **模块化单体 + 独立 AI 服务 + RabbitMQ 异步解耦**

选择理由如下：

- 研发效率高，适合竞赛周期；
- 架构清晰，模块边界明确；
- 避免为了“看起来高级”而过度微服务化；
- Java 负责业务平台，Python 负责模型训练与推理，职责分离清晰；
- 后续如需演进为微服务，具备平滑拆分基础。

### 5.2 分层架构

```text
┌─────────────────────────────────────────────────────────────┐
│ 表现层：React SPA（医生端 / 筛查端 / 管理端 / 患者报告页） │
├─────────────────────────────────────────────────────────────┤
│ 接入层：Nginx（HTTPS、静态资源、反向代理、基础限流）       │
├─────────────────────────────────────────────────────────────┤
│ 业务层：Spring Boot 模块化单体                             │
│   ├── auth / user / org                                    │
│   ├── patient / case / image                               │
│   ├── analysis / risk / report                             │
│   └── followup / dashboard / model-admin                   │
├─────────────────────────────────────────────────────────────┤
│ 算法层：Python FastAPI                                     │
│   ├── preprocess                                            │
│   ├── tooth-detect                                          │
│   ├── tpc-net-segmentation                                  │
│   ├── edl-grading                                           │
│   ├── multimodal-risk                                       │
│   └── explainability                                        │
├─────────────────────────────────────────────────────────────┤
│ 中间件与存储层：MySQL / Redis / MinIO / RabbitMQ           │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 通信策略

| 通信链路 | 方式 | 说明 |
|---|---|---|
| 前端 → 后端 | RESTful API + WebSocket | REST 处理业务请求，WebSocket 推送分析进度 |
| Java → Python | HTTP 同步 + MQ 异步 | 轻量查询走 HTTP，分析任务和结果通知走 MQ |
| 模块内协作 | 领域事件 | 降低模块耦合 |
| 文件访问 | MinIO 预签名 URL | 受控访问原图、掩膜图、报告文件 |

### 5.4 为什么不做过度工程化

本项目主文档不将 K8s、ELK、复杂分布式治理体系作为当前落地主架构，原因是：

- 当前比赛更看重业务闭环、技术合理性和演示稳定性；
- 复杂架构若无法真实落地，会削弱可信度；
- 竞赛答辩中“克制”比“堆栈”更容易获得评委认可。

## 六、业务数据与训练数据边界设计

这是本项目的关键可信设计之一，必须单独说明。

### 6.1 设计原则

- **业务运行库不直接等于训练数据集**；
- 训练使用的数据必须经过脱敏、审批和版本固化；
- 医生修正结果不能未经治理直接回灌到模型训练；
- 所有训练数据必须具备可追溯版本号和来源说明。

### 6.2 数据分层

| 数据层 | 内容 | 是否含业务身份信息 | 用途 |
|---|---|---|---|
| 业务运行层 | 患者档案、病例、原始影像、报告、随访记录 | 是 | 日常业务运行 |
| 脱敏副本层 | 去标识化后的影像与结构化特征 | 否 | 标注与训练准备 |
| 标注与质控层 | 标注结果、仲裁结果、金标准样本 | 否 | 训练集构建与评估 |
| 训练数据层 | 固化版本的数据快照 | 否 | 模型训练与离线评估 |
| 模型资产层 | 模型权重、评估指标、版本记录 | 否 | 推理部署与版本管理 |

### 6.3 数据流转约束

```text
业务系统原始数据
    ↓（脱敏与审批）
脱敏数据副本
    ↓（标注 / 复核 / 仲裁）
金标准样本池 / 候选训练样本池
    ↓（版本固化）
训练数据集 vX.Y.Z
    ↓（训练与离线评估）
候选模型
    ↓（专家审核 + 人工审批）
上线模型
```

### 6.4 医生修正数据如何进入训练流程

1. 医生在业务系统中完成修正；
2. 修正记录先进入业务侧 `correction_feedback`；
3. 数据管理员定期导出经过脱敏处理的候选样本；
4. 临床顾问与算法工程师确认可纳入训练范围；
5. 生成训练数据快照并登记版本号；
6. 模型离线训练与评估；
7. 通过审批后再替换推理服务中的主模型版本。

### 6.5 医疗可信口径

本项目明确采用“**人工审批上线**”而不是“自动切换主模型”的策略，理由如下：

- 医疗辅助决策系统需要保留人工责任边界；
- 模型表现提升不等于所有场景都适合立即上线；
- 人工审批能提供更完整的版本治理与答辩可信度。

## 七、功能模块划分

平台按业务领域划分为 9 个核心模块。

### 模块 1：用户与权限管理

**目标**：提供多角色、多机构、可审计的访问控制能力。

**核心功能**：

- 登录、登出、JWT 鉴权；
- 角色权限控制；
- 科室/机构管理；
- 操作审计日志；
- 登录异常监控与账号锁定。

### 模块 2：患者与病例管理

**目标**：管理患者档案和病例全生命周期。

**核心功能**：

- 患者档案创建与查询；
- 病例编号生成；
- 风险因子录入；
- 历史病例关联；
- 复诊对比入口；
- 病例状态流转。

### 模块 3：影像上传与质量管理

**目标**：完成口腔影像接入、管理和基础质量控制。

**核心功能**：

- 单张与批量上传；
- 格式、大小、MD5 校验；
- 影像元数据管理；
- 上传后自动触发质量评估；
- 影像类型识别；
- 不合格图像提示重传。

### 模块 4：AI 分析中心

**目标**：承载分割、分级、风险推理与解释生成。

**核心功能**：

- TPC-Net 分割；
- EDL 分级；
- 牙位检测与映射；
- 结果可视化生成；
- 分析任务状态管理；
- 失败重试与异常记录。

### 模块 5：风险评估与复核建议

**目标**：把“识别结果”升级为“可信决策建议”。

**核心功能**：

- 多模态风险评分；
- 高风险标记；
- 不确定性阈值判断；
- 复核建议；
- 特征贡献解释；
- 复查周期推荐。

### 模块 6：双报告生成

**目标**：同时生成医生版专业报告和患者版通俗报告。

**核心功能**：

- 医生版报告：影像结果、病灶定位、分级、不确定性、复核建议；
- 患者版报告：通俗解释、风险卡片、日常建议、复查提醒；
- PDF 导出与归档；
- 报告版本管理。

## 八、核心算法与可信决策设计

### 8.1 模型体系总览

```text
输入：全景片 / 根尖片 / 口内照 + 结构化风险因子
  ├── 牙位检测（YOLOv8）
  ├── TPC-Net 分割（TPE + CVC）
  ├── EDL 分级
  ├── 多模态风险评估
  └── 解释生成（热力图 + 特征贡献）
输出：病灶区域、牙位、分级、不确定性、风险等级、报告要素
```

### 8.2 TPC-Net 设计要点

- 主干：SegFormer-B2；
- 牙位感知：TPE 注入中层特征；
- 跨视图约束：CVC 损失；
- 输出：分割掩膜 + 全局图像特征；
- 评估重点：Dice、IoU、Boundary IoU、不同牙位表现差异。

### 8.3 EDL 分级设计要点

- 输入：图像特征 + 牙位定位结果；
- 输出：简化四级分类（C0/C1/C2/C3）与不确定性分数；
- 业务联动：高不确定性直接进入复核流程；
- 评估重点：Macro-F1、Weighted Kappa、ECE、拒识曲线。

### 8.4 多模态风险评估策略

本项目将风险评估定位为“平台能力补全”，而不是主创新。方案采用适中可落地路线：

- 输入：病灶面积、牙位、分级、不确定性、年龄、刷牙频率、含糖饮食、既往龋病史等；
- 融合策略：图像 embedding + 表格特征的轻量级融合；
- 输出：低/中/高风险等级、主要风险因子、复查周期建议。

### 8.5 解释性策略

解释性设计坚持“够用、可信、可演示”的原则：

| 层次 | 面向对象 | 输出 |
|---|---|---|
| 像素级解释 | 医生 | 分割掩膜、高亮叠加图 |
| 特征级解释 | 医生/科研管理员 | SHAP 风险因子贡献 |
| 文本级解释 | 患者 | 通俗化结果说明与复查建议 |

说明：本项目不将“反事实”作为主创新，而将其收敛为“健康行为模拟建议”，用于患者教育展示。

## 九、端到端业务流程

### 9.1 基础筛查流程

1. 医生或筛查员登录平台；
2. 创建或选择患者档案；
3. 录入基础信息与风险因子；
4. 创建病例；
5. 上传口腔影像；
6. 系统自动完成影像质量检查；
7. 质量合格后提交 AI 分析任务；
8. 输出病灶定位、分级和不确定性；
9. 完成多模态风险评估；
10. 生成医生版与患者版报告；
11. 高风险或高不确定性样本进入复核队列；
12. 医生确认或修正后归档病例；
13. 必要时自动生成随访计划。

### 9.2 复诊对比流程

1. 患者复诊并关联历史病例；
2. 上传新影像；
3. 系统重新分析并展示前后结果差异；
4. 平台生成风险变化趋势；
5. 医生复核并更新结论；
6. 修正结果进入回流池候选队列。

### 9.3 数据回流与模型治理流程

1. 医生修正业务结果；
2. 修正记录写入回流日志；
3. 数据管理员按周期导出脱敏候选样本；
4. 形成数据快照版本；
5. 离线训练候选模型；
6. 完成指标评估与错误分析；
7. 专家审核通过后人工批准上线；
8. 更新主模型版本并保留回滚点。

## 十、非功能性设计

### 10.1 可信性

- 结果可追溯到病例、影像、模型版本；
- 高不确定性场景自动提醒人工复核；
- 报告保留版本号和生成时间；
- 医生修正保留原始值与修正值。

### 10.2 稳定性

- AI 分析任务采用 MQ 解耦；
- 分析失败具备重试机制；
- 关键链路记录 traceId；
- AI 服务不可用时支持返回“稍后复核”状态而非直接崩溃。

### 10.3 安全性

- JWT 鉴权与角色权限控制；
- 敏感字段加密存储；
- MinIO 预签名受控访问；
- 审计日志记录关键操作；
- 输入参数校验、防注入与文件白名单校验。

### 10.4 可扩展性

- AI 服务独立部署，便于替换模型；
- 模块化单体便于后续拆分；
- 数据结构支持扩展其他口腔疾病任务；
- API 采用版本化设计。

### 10.5 性能目标

| 指标 | 目标 |
|---|---|
| 单张影像上传 | 小于 3 秒 |
| 单病例 AI 分析 | 小于 15 秒（演示环境） |
| 报告生成 | 小于 5 秒 |
| 首屏加载 | 小于 2 秒 |
| 演示环境并发 | 至少 50 用户 |

### 11.2 模型证据

### 11.5 指标分层策略

|---|---|---|

| 分级 Macro-F1 | ≥ 0.72 | ≥ 0.78 |
| 风险 AUC | ≥ 0.84 | ≥ 0.90 |
| 标注一致性 Dice | ≥ 0.78 | ≥ 0.82 |
| Weighted Kappa | ≥ 0.70 | ≥ 0.75 |

说明：答辩现场优先展示“已达成指标”和“有证据支撑的结果”，而不是只展示冲刺值。

## 十三、答辩策略与风险控制

### 13.1 答辩主叙事

建议答辩时坚持以下主线：

> 我们不是在做一个模型展示网站，而是在做一个面向真实筛查流程的可信辅助决策平台。平台通过牙位感知分割和不确定性感知分级，把 AI 结果真正接入医生复核、报告生成和数据回流闭环中。

### 13.2 60 秒演示脚本建议

1. 选择病例并上传影像；
2. 平台自动完成质量检查与 AI 分析；
3. 展示病灶高亮、牙位、分级与不确定性；
4. 展示医生版和患者版报告；
5. 医生修正一处结果；
6. 系统记录修正并更新报告；
7. 展示该修正进入回流候选池。

### 13.3 需要主动规避的高风险表述

以下表述建议避免在答辩中使用：

- “系统会自动替代医生判断”；
- “模型达标后自动切换主模型”；
- “报告直接给出治疗决策”；
- “全部数据都是真实临床大规模多中心采集”但拿不出证明；
- “我们做了很多 SOTA”但没有消融和工程落地证据。

### 13.4 最大风险与应对策略

| 风险 | 表现 | 应对策略 |
|---|---|---|
| 创新点过多导致主线分散 | 答辩重点模糊 | 固定只讲 2 个核心创新 + 1 个闭环亮点 |
| 数据口径过大但证据不足 | 被质疑真实性 | 采用“已完成 / 计划中”分层口径 |
| 过度工程化 | 被认为炫技 | 坚持精简落地架构 |
| 可信性不足 | 被问医生为何信任系统 | 强调不确定性、复核机制、审计链 |
| 演示不稳定 | 影响整体印象 | 所有链路预演并准备兜底演示素材 |
