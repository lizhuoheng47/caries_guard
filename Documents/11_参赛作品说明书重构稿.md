# 参赛作品说明书重构稿

## 1. 作品名称

**CariesGuard：基于影像分析与检索增强生成的龋病智能辅助筛查与解释平台**

## 2. 作品定位

本作品面向口腔龋病筛查与辅助决策场景，构建“影像分析 + 不确定性复核 + 检索增强解释 + 医生反馈回流”的一体化人工智能实践平台。

作品不将 AI 定位为自动替代医生诊断的黑盒系统，而是定位为一个可追溯、可复核、可解释、可持续迭代的医疗 AI 辅助工具。

## 3. 项目背景

龋病筛查场景中存在以下问题：

1. 人工判读压力大
2. 结果一致性不足
3. 单点 AI 缺少解释、审计和复核能力
4. 患者解释成本高
5. 业务闭环不完整

因此，项目目标不是只做一个识别模型，而是构建一个符合医疗辅助决策特点的人工智能实践闭环系统。

## 4. 总体目标

### 4.1 影像智能分析目标

完成质量检查、病灶检测、区域分割、龋病分级与风险辅助分析，输出结构化结果与可视化产物。

### 4.2 医疗解释目标

基于知识图库和 RAG，为患者与医生提供带引用、带风险提示的解释能力。

### 4.3 安全治理目标

通过 uncertainty、复核建议、签名回调、日志留痕和知识来源管理，保证输出具有可审计性与可解释性。

### 4.4 持续迭代目标

记录医生修正反馈、RAG 检索日志、模型运行日志和版本信息，为后续评估与优化提供数据基础。

## 5. 核心创新点

1. 从单点识别升级为医疗 AI 闭环
2. 引入 uncertainty 驱动的复核机制
3. 采用“通用大模型 + 知识图库 + 结构化病例上下文”路线
4. 强调引用、审计与责任边界
5. 平台化设计支持后续模型替换与扩展

## 6. 技术路线

### 6.1 总体架构

- Java 后端负责用户、病例、影像、分析任务、报告、复核、鉴权与审计
- Python 后端负责影像分析、visual asset、RAG、知识治理与 AI 运行日志
- 两端通过 RabbitMQ 与 HTTP callback 协同

### 6.2 影像分析链路

用户提交影像后，Java 创建任务，Python 消费任务并完成质量检查、检测、分割、分级和风险分析，再通过 callback 将结果回写 Java。

### 6.3 RAG 链路

Java 负责鉴权与病例上下文组织，Python RAG 服务完成知识检索、上下文组装与大模型生成，返回 citations、confidence 与 safetyFlags。

### 6.4 数据治理链路

统一记录 AI 输出、RAG 检索、模型版本、知识版本、回调日志和医生修正反馈。

## 7. 关键模块

### 7.1 影像分析模块

- 质量检查
- 检测
- 分割
- 分级
- uncertainty
- 风险辅助分析
- visual assets

### 7.2 RAG 智能解释模块

- 知识文档入库
- 文档切分与索引重建
- 患者解释
- 医生问答
- 引用展示
- 检索和生成日志

### 7.3 复核与反馈模块

- 高 uncertainty 自动建议复核
- 医生复核提交
- 差异留痕
- 反馈回流

### 7.4 平台支撑模块

- 病例与影像管理
- 分析任务状态推进
- 报告生成
- 操作审计与权限控制

## 8. 系统优势

- 可解释：有 visual assets、引用和结构化证据链
- 可复核：有 uncertainty 触发的人工复核入口
- 可追溯：有模型、知识、日志和反馈留痕
- 可扩展：分析和 RAG 模块均支持替换与升级
- 可落地：已有病例、任务、回调、日志和对象存储工程基础

## 9. 当前完成情况

### 9.1 已完成

1. Java / Python 双后端协同框架
2. AI 分析任务主链路
3. callback 契约与签名校验
4. visual asset 上传与展示
5. 龋病分级与 uncertainty 字段口径
6. RAG 知识入库、索引重建、问答与日志最小闭环
7. Docker 环境联调与 E2E 验证

### 9.2 正在强化

1. 风险融合真实化
2. 医生修正反馈回流与评估闭环
3. 知识图库版本治理
4. RAG 与结构化病例上下文深度整合
5. 医疗 AI 场景下的拒答与安全约束

## 10. 参赛口径下的展示重点

1. 一张影像如何被分析
2. 系统如何判断自己不确定
3. 系统如何解释结果
4. 系统如何通过反馈持续变好

答辩重点不应放在用户管理、菜单管理、配置管理等通用后台能力上。

## 11. 建议演示流程

1. 导入病例与影像
2. 触发 AI 分析
3. 查看 grading、uncertainty、needsReview 和 visual assets
4. 发起患者版解释或医生版问答
5. 提交人工复核与医生反馈

## 12. 应用价值

- 对医生：降低初筛与解释负担
- 对患者：获得更直观的病情解释和建议
- 对机构：沉淀病例、知识、日志和反馈数据
- 对比赛：体现 AI 工程落地、责任边界和持续迭代能力

## 13. 结论

CariesGuard 的核心价值不在于“做了一个识别龋病的模型”，而在于围绕龋病筛查场景，构建了一个具备结构化分析、智能解释、复核控制、反馈回流和工程治理能力的医疗 AI 实践系统。
