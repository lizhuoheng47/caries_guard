## 1. 编制说明

### 1.1 本文档作用

本数据字典面向以下对象共同使用：

- 后端开发工程师；
- 前端开发工程师；
- 产品经理；
- 测试工程师；
- 数据库设计人员；
- AI 协同编码工具（如 Codex、Gemini、Claude Code）。

### 1.2 使用原则

1. 本文档是后续 SQL 设计和代码生成的直接依据；
2. 表名、字段名、状态码一旦冻结，不建议频繁变更；
3. 前后端接口命名应尽量复用本字典术语；
4. 所有枚举型字段应优先映射到字典表；
5. 敏感字段必须参考隐私分级与权限矩阵使用。

## 2. 统一字段设计约定

### 2.1 公共字段约定

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `id` | BIGINT | 主键 |
| `org_id` | BIGINT | 所属机构，用于数据隔离 |
| `status` | VARCHAR(32) | 记录状态/启用状态 |
| `deleted_flag` | CHAR(1) | 逻辑删除标记，0/1 |
| `remark` | VARCHAR(500) | 备注 |
| `created_by` | BIGINT | 创建人 |
| `created_at` | DATETIME | 创建时间 |
| `updated_by` | BIGINT | 更新人 |
| `updated_at` | DATETIME | 更新时间 |

### 2.2 字段设计约定

| 类型 | 设计规范 |
|---|---|
| 编号字段 | 统一使用 `*_no` |
| 类型/状态字段 | 统一使用 `*_code` 或 `status` |
| 布尔字段 | 统一使用 `*_flag` |
| 外键字段 | 统一使用 `xxx_id` |
| 时间字段 | 统一使用 `*_at` 或 `*_date` |
| JSON 字段 | 统一使用 `*_json` |

### 2.3 关于外键

本文档中的“关联某表”表示**逻辑外键关系**。  
物理层是否创建数据库外键，可在 SQL 阶段根据性能、迁移与维护策略决定。

## 4. 系统管理域数据字典

## sys_user（系统用户表）
**所属数据域**：系统管理域  
**表说明**：存储平台登录账号、人员身份、联系方式、账号状态，是统一认证与授权的核心主体表。  

**建模规则**：
- 用户名全局唯一
- 手机号、邮箱建议做脱敏展示与加密存储
- 患者端账号通过业务关联而不是复用患者表主键

**索引建议**：
- UK(username)
- IDX(dept_id,status)
- IDX(org_id,user_type_code)
- IDX(phone)

| 字段 | 类型 | 必填 | 键/索引 | 默认值 | 说明 | 取值/规则 |
|---|---|---|---|---|---|---|
| id | BIGINT | 是 | PK | - | 用户主键 | 全局唯一 |
| dept_id | BIGINT | 否 | IDX | NULL | 所属部门ID | 关联 sys_dept.id |
| user_no | VARCHAR(32) | 是 | UK | - | 用户编号 | 内部业务编号 |
| username | VARCHAR(64) | 是 | UK | - | 登录账号 | 小写英文/数字/下划线 |
| password_hash | VARCHAR(255) | 是 | - | - | 密码哈希 | 禁止明文存储 |
| real_name | VARCHAR(64) | 是 | IDX | - | 真实姓名 | 敏感字段 |
| nick_name | VARCHAR(64) | 否 | - | NULL | 昵称/展示名 | 可空 |
| user_type_code | VARCHAR(32) | 是 | IDX | DOCTOR | 用户类型 | ADMIN/ORG_ADMIN/DOCTOR/SCREENER/RESEARCHER/PATIENT |
| gender_code | VARCHAR(16) | 否 | - | UNKNOWN | 性别代码 | sys_gender |
| phone_enc | VARCHAR(255) | 否 | - | NULL | 手机号密文 | AES加密存储 |
| phone_hash | VARCHAR(128) | 否 | IDX | NULL | 手机号摘要 | 唯一去重与精准搜索 |

| last_login_at | DATETIME | 否 | - | NULL | 最后登录时间 | 登录成功后更新 |
| pwd_updated_at | DATETIME | 否 | - | NULL | 密码更新时间 | 密码策略使用 |
| org_id | BIGINT | 是 | IDX | - | 所属机构ID | 用于行级隔离 |

## 9. 附录：推荐枚举与状态字段落地方式

### 9.1 推荐做法

- 数据库存储：使用英文稳定编码，例如 `ACTIVE`、`CLOSED`、`PASS`；
- 前端展示：从字典接口读取中文标签；
- 统计分析：以编码聚合，以标签展示；
- 导入导出：优先导出“编码 + 标签”双列，避免语义丢失。

### 9.2 不推荐做法

- 在表中直接写中文状态；
- 同一语义在不同表使用不同编码；
- 直接在代码中硬编码大量状态枚举；
- 由前端自行维护一套与后端不同的状态字典。

### 9.3 本文档结论

后续你们一旦进入真正开发阶段，**先以这份数据字典为准冻结字段和状态，再写 SQL，再写 Java 实体、DTO、VO、Mapper、接口与页面**。  
这样才能保证数据库、后端、前端、AI 协同开发使用同一套语言。
