# 02 数据库设计

## 1. 双库设计

CariesGuard 使用物理隔离的双库：

| 数据库 | 职责 | 迁移工具 | 负责人 |
| --- | --- | --- | --- |
| `caries_biz` | Java 业务主库 | Flyway | Java 后端 |
| `caries_ai` | Python AI / RAG 运行库 | Alembic | Python AI/RAG |

两库不跨库迁移、不跨库写主数据、不建跨库外键，只通过业务标识（`task_no` / `case_id` / `case_no` / `image_id` / `attachment_id` / `trace_id`）建立弱引用。

## 2. 归属矩阵

| 数据 | Java `caries_biz` | Python `caries_ai` |
| --- | --- | --- |
| 用户、角色、组织 | Owner | 只通过 API 读取 |
| 患者、就诊、病例 | Owner | Weak reference |
| 原始影像附件 | Owner | 读取 / 下载 |
| visual asset 文件 | Metadata owner | 生成 / 上传 |
| analysis task 业务状态 | Owner | 只消费 `task_no` |
| summary / raw_result_json | Owner | 产出 payload |
| AI 推理运行日志 | —— | Owner |
| RAG 知识库 / 检索日志 | 可选展示 | Owner |
| 通用大模型调用日志 | —— | Owner |
| 模型 / 算法 / 知识版本 | 可选展示 | Owner |

### 3.1 `ana_result_summary` 稳定字段

- `overall_highest_severity`
- `uncertainty_score`
- `review_suggested_flag`
- `summary_json`
- `raw_result_json`（AI 原始证据链，Java 消费扩展字段的主要位置）

### 3.2 附件字段

`med_attachment` 必须包含：

- `bucket_name`
- `object_key`
- `content_type`
- `file_size_bytes`
- `md5`
- `storage_provider_code`
- `visibility_code`

MinIO object 不替代数据库附件主数据。

## 4. caries_ai 核心表

### 4.1 推理运行域

| 表 | 说明 |
| --- | --- |

### 4.2 RAG 域

| 表 | 说明 |
| --- | --- |
| `rag_knowledge_base` | 知识库定义、版本 |
| `rag_document` | 知识文档元数据（指南、规范、科普、报告模板、规则说明） |

| `llm_call_log` | 通用大模型调用日志（provider、model、prompt 摘要、token 估算、响应摘要、错误） |

### 4.3 模型治理域

| 表 | 说明 |
| --- | --- |
| `mdl_model_version` | 模型 / 算法适配器 / 知识版本登记 |

| `mdl_model_approval_record` | 上线审批记录 |

`mdl_model_version` 同时登记：ML 权重模型、HEURISTIC 算法适配器、RAG embedding 模型、通用大模型 provider / model 组合、知识库版本。

| model_type_code | model_code | impl_type |

## 6. 迁移规范

- `caries_biz` 只通过 Flyway 迁移；
- `caries_ai` 只通过 Alembic 迁移；
- 任何服务不得跨库执行 schema 迁移；
- Docker 环境启动时执行迁移（`CG_DB_SCHEMA_BOOTSTRAP_ENABLED=false`、`CG_DB_MIGRATION_ENABLED=true`）。

### 6.1 Alembic 流程

```bash
cd backend-python
alembic revision -m "describe_change"
alembic upgrade head
```

迁移文件命名：`NNNN_动词_对象_细节.py`。`downgrade()` 必须可执行。

### 6.2 JSON 扩展优先原则

若新增字段不影响查询索引和强约束，优先写入 JSON：

- `ai_infer_job.result_json`
- `ai_infer_job_image.result_json`
- `ana_result_summary.raw_result_json`
- `llm_call_log` / `rag_retrieval_log` 的 JSON 扩展字段

只有当字段需要稳定查询、统计、索引或业务状态推进时，才新增列。

### 6.3 禁止事项

- Python 直接写 `ana_task_record` / `ana_result_summary`；
- Java 直接写 `rag_chunk` / `ai_infer_job`；
- 通用大模型回答直接改变病例状态；
- 以微调数据表替代知识库治理。

## 7. 隐私与数据治理

- `caries_ai` 不保存患者明文身份；
- RAG prompt 不包含非必要敏感信息；
- 日志只保存必要摘要；
- MinIO bucket 私有，导出数据有生命周期；
- 所有时间字段采用数据库统一时间策略。

关键质量指标：

- task / callback 成功率；
- visual asset 完整率；
- grading / uncertainty 留痕完整率；
- RAG 检索命中率与引用准确率；
- 医生复核一致率。

4. 运行单元测试；

6. 归档 SQL、日志、callback payload；
