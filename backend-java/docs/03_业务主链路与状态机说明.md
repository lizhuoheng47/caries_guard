# 业务主链路与状态机说明

本文档描述当前项目真正实现的业务流程，而不是目标蓝图。

## 1. 总体链路

当前后端已经落地的业务主链路如下：

1. 用户登录
2. 创建患者
3. 创建就诊
4. 创建病例
5. 上传附件
6. 绑定病例图像
7. 做图像质检
8. 创建 AI 分析任务
9. Python / AI 服务回调分析结果
10. 生成报告
11. 根据风险或复核建议触发随访
12. 在看板查询统计数据

## 2. 登录与权限

### 2.1 登录链路

入口：
- `AuthController.login`
- `AuthAppService.login`

行为：
- 校验用户名密码
- 生成 JWT
- 写入 `sys_login_log`
- 更新最后登录时间

### 2.2 鉴权链路

入口：
- `SecurityConfig`
- `JwtAuthenticationFilter`

行为：
- 除白名单接口外全部需要认证
- JWT 解析后，会通过 `UserDetailsService` 加载用户
- 用户上下文在业务层通过 `SecurityContextUtils.currentUser()` 获取

### 2.3 权限判断

入口：
- `RequirePermissionAspect.around`

当前规则：
- 角色含 `ADMIN` 或 `SYS_ADMIN` 时，可绕过细粒度权限校验
- 普通用户需要依赖权限仓储判断是否拥有指定权限码

当前事实：
- 因为默认库没有菜单种子，所以非管理员权限场景当前不完整

## 3. 患者、就诊、病例链路

### 3.1 创建患者

入口：
- `PatientController.createPatient`
- `PatientCommandAppService.createPatient`

行为：
- 生成患者编号 `PATyyyyMMdd######`
- 保存患者主档
- 保存监护人信息
- 对姓名、电话、证件号、出生日期等敏感字段做加密/哈希/脱敏处理
- 如证件号重复，会拒绝创建

### 3.2 创建就诊

入口：
- `VisitController.createVisit`
- `VisitCommandAppService.createVisit`

行为：
- 生成就诊编号 `VISyyyyMMdd######`
- 挂接到患者
- 记录就诊时间、就诊类型、来源渠道等

### 3.3 创建病例

入口：
- `CaseController.createCase`
- `CaseCommandAppService.createCase`

行为：
- 绑定患者和就诊
- 初始状态为 `CREATED`
- 记录病例状态日志

### 3.4 保存诊断与牙位记录

入口：
- `CaseController.saveDiagnoses`
- `CaseController.saveToothRecords`
- `CaseClinicalRecordAppService.saveDiagnoses`
- `CaseClinicalRecordAppService.saveToothRecords`

行为：
- 按病例整组替换原有记录
- 牙位记录会做重复项校验

## 4. 图像链路

### 4.1 上传附件

入口：
- `FileController.upload`
- `AttachmentAppService.upload`

行为：
- 接收 multipart 文件
- 计算 MD5
- 做附件去重
- 通过 `ObjectStorageService` 持久化
- 默认 `MINIO` 实现类为 `MinioObjectStorageService`，`LOCAL_FS` 实现类为 `LocalObjectStorageService`

### 4.2 病例绑定图像

入口：
- `CaseImageController.create`
- `CaseImageAppService.createCaseImage`

行为：
- 校验病例、就诊、患者、附件之间的归属关系
- 同步维护是否主图
- 创建第一张图像时，若病例仍在 `CREATED`，会推进到 `QC_PENDING`

### 4.3 图像质检

入口：
- `ImageQualityCheckController.save`
- `CaseImageAppService.saveQualityCheck`

行为：
- 保存质检结果
- 通过 `current_flag` 标记最新质检
- 后续分析任务只挑选 `PASS` 图像

## 5. 分析链路

### 5.1 创建分析任务

入口：
- `AnalysisTaskController.createAnalysisTask`
- `CaseAnalysisAliasController.createAnalysisFromCase`
- `AnalysisTaskAppService.createTask`

前置条件：
- 病例必须处于可分析状态
- 命令中的 `patientId` 必须与病例 patientId 一致
- 默认不允许同一病例同时存在运行中任务
- 必须存在至少一张 `PASS` 图像

行为：
- 生成 `taskId` 和 `taskNo`
- 构造 `AiAnalysisRequestDTO`
- 写入 `ana_task_record`
- 推动病例进入 `ANALYZING`
- 发布 `analysis.requested` 事件

### 5.2 重试分析任务

入口：
- `AnalysisTaskController.retryAnalysisTask`
- `AnalysisTaskAppService.retryTask`

规则：
- 只有 `FAILED` 任务允许重试
- 重试时会创建新任务，不覆盖原任务
- 新任务通过 `retry_from_task_id` 指向原失败任务

### 5.3 Python / AI 回调

入口：
- `AnalysisCallbackController.receiveAnalysisResultCallback`
- `AnalysisCallbackAppService.handleResultCallback`

当前支持状态：
- `PROCESSING`
- `SUCCESS`
- `FAILED`

处理规则：
- 先验签
- 再按 `taskNo` 找任务
- 如果是重复终态回调，直接 ACK
- 如果该任务已经被重试，晚到回调只 ACK，不再写库

### 5.4 成功回调后的行为

当 `taskStatusCode = SUCCESS` 时：
- 更新 `ana_task_record` 状态和时间
- 写 `ana_result_summary`
- 替换写 `ana_visual_asset`
- 如存在风险评估，则写 `med_risk_assessment_record`
- 推动病例状态为 `REVIEW_PENDING`
- 发布 `analysis.completed` 事件

### 5.5 失败回调后的行为

当 `taskStatusCode = FAILED` 时：
- 更新 `ana_task_record`
- 记录错误信息
- 推动病例回退到 `QC_PENDING`
- 发布 `analysis.failed` 事件

### 5.6 纠偏反馈链路

入口：
- `CorrectionFeedbackController.submitCorrectionFeedback`
- `CaseAnalysisAliasController.submitCorrectionFromCase`
- `CorrectionFeedbackAppService.submit`

行为：
- 保存原始推断 JSON
- 保存医生修正后的真值 JSON
- 用于业务留痕和后续训练数据准备

## 6. 病例状态机

状态机入口：
- `CaseStatusMachine.ensureTransitionAllowed`

当前允许的状态流转：

| 当前状态 | 可转目标状态 |
| --- | --- |
| `CREATED` | `QC_PENDING` `CANCELLED` |
| `QC_PENDING` | `ANALYZING` `CREATED` `CANCELLED` |
| `ANALYZING` | `REVIEW_PENDING` `QC_PENDING` `CANCELLED` |
| `REVIEW_PENDING` | `REPORT_READY` `FOLLOWUP_REQUIRED` `CANCELLED` |
| `REPORT_READY` | `FOLLOWUP_REQUIRED` `CLOSED` |
| `FOLLOWUP_REQUIRED` | `CLOSED` `CANCELLED` |
| `CLOSED` | 终态，不可流转 |
| `CANCELLED` | 终态，不可流转 |

关键原则：
- 终态不可再流转
- 任何模块要改病例状态，都应该通过 `CaseCommandAppService`
- 分析模块和随访模块也不能旁路直接改 `med_case`

## 7. 报告链路

### 7.1 生成报告

入口：
- `ReportController.generateReport`
- `ReportAppService.generateReport`

前置条件：
- 病例状态必须是 `REVIEW_PENDING` 或 `REPORT_READY`
- 必须已有分析摘要

生成过程：
1. 读取病例、分析摘要、病例图像、风险评估、纠偏记录
2. 解析模板内容，优先数据库模板，否则默认模板
3. 渲染文本
4. 用 `ReportPdfService` 生成 PDF 二进制
5. 存储 PDF 为附件
6. 建立 `rpt_record`
7. 必要时推动病例到 `REPORT_READY`
8. 触发随访判断

### 7.2 当前 PDF 能力边界

- 只支持 ASCII 输出
- 非 ASCII 字符会变成 `?`
- 行数上限约 35
- 每行长度有限
- 当前更适合作为流程闭环证明，不适合作为正式生产排版

### 7.3 报告导出

入口：
- `ReportController.exportReport`
- `ReportAppService.exportReport`

当前真实行为：
- 校验报告存在
- 校验附件存在
- 写 `rpt_export_log`
- 返回 `exported=true`

当前未实现的行为：
- 不直接输出 PDF 文件流
- 不直接推送到邮件、微信、OSS 外链

## 8. 随访链路

### 8.1 手工创建计划/任务

入口：
- `FollowupPlanController.createPlan`
- `FollowupTaskController.createTask`
- 对应 AppService 的 `createPlan`、`createTask`

说明：
- 允许手工管理随访计划和任务
- 手工创建计划时会派生首个任务

### 8.2 自动触发随访

入口：
- `FollowupTriggerService.triggerFromReport`

触发条件：
- 风险等级为 `HIGH`
- 或 `reviewSuggestedFlag = 1`

触发行为：
- 根据风险和建议解析触发源
- 做幂等判断，避免同一报告重复生成计划
- 创建计划
- 创建首个任务
- 写消息提醒留痕
- 尝试推动病例进入 `FOLLOWUP_REQUIRED`

### 8.3 随访记录推动任务完成

入口：
- `FollowupRecordController.addRecord`
- `FollowupRecordAppService.addRecord`

行为：
- 新增一条随访记录
- 当前任务通常会被标记完成
- 如果 `followNext=true`，可能继续派生下一任务
- 否则根据计划状态自动收口

## 9. 看板链路

入口：
- `DashboardController`
- `DashboardOpsController`
- `DashboardStatsRepository`

当前统计口径：
- 总览：患者数、病例数、已分析病例数、已生成报告数、需随访病例数、已关闭病例数
- 病例状态分布：按 `med_case.case_status_code`
- 风险分布：按最新 `med_risk_assessment_record`
- 随访任务汇总：按 `fup_task.task_status_code`
- 积压汇总：待审核病例、待处理随访、逾期随访、高风险未闭环病例
- 模型运行：最近 30 天任务总量、成功量、失败量、成功率
- 趋势：按日统计新增病例、分析完成、报告生成、随访触发

## 10. 当前最重要的业务边界

1. Python 如果不消费 Rabbit，分析链路不会自动继续。
2. 回调成功才会有报告和随访后续链路。
3. 本地默认库没有业务数据，看板多数返回 0 是正常现象。
4. 非管理员角色场景受菜单种子缺失影响，默认不可完整演示。
5. 业务模块的数据权限主要还是 `org_id` 校验，不是完整的数据权限平台式实现。
