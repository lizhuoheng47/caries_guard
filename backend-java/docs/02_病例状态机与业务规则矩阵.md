# 病例状态机与业务规则矩阵

## 1. 当前代码来源

本文完全依据 `caries-patient/app/CaseStatusMachine.java`、`CaseCommandAppService.java`、`CaseImageAppService.java`、`AnalysisCallbackAppService.java`、`ReportAppService.java`、`FollowupTriggerService.java` 重写。

## 2. 主状态集合

`med_case.case_status_code` 当前主状态为：

- `CREATED`
- `QC_PENDING`
- `ANALYZING`
- `REVIEW_PENDING`
- `REPORT_READY`
- `FOLLOWUP_REQUIRED`
- `CLOSED`
- `CANCELLED`

终态：

- `CLOSED`
- `CANCELLED`

终态后不允许继续流转。

## 3. 当前允许迁移矩阵

- `CREATED -> QC_PENDING | CANCELLED`
- `QC_PENDING -> ANALYZING | CREATED | CANCELLED`
- `ANALYZING -> REVIEW_PENDING | QC_PENDING | CANCELLED`
- `REVIEW_PENDING -> REPORT_READY | FOLLOWUP_REQUIRED | CANCELLED`
- `REPORT_READY -> FOLLOWUP_REQUIRED | CLOSED`
- `FOLLOWUP_REQUIRED -> CLOSED | CANCELLED`

## 4. 各迁移触发源

### 4.1 创建病例

- 触发：`POST /api/v1/cases`
- 初始状态：`CREATED`
- 日志：`change_reason_code = CASE_CREATED`

### 4.2 首张影像入库

- 触发：`POST /api/v1/cases/{caseId}/images`
- 条件：病例当前状态为 `CREATED`
- 迁移：`CREATED -> QC_PENDING`
- 日志原因码：`IMAGE_UPLOADED`

### 4.3 创建 AI 任务

- 触发：`POST /api/v1/analysis/tasks` 或 `POST /api/v1/cases/{caseId}/analysis`
- 前置条件：
  - 病例状态必须允许分析
  - 至少存在一张 `quality_status_code = PASS` 的影像
  - 同病例没有正在运行的任务，或显式允许重试
- 迁移：`QC_PENDING -> ANALYZING`
- 日志原因码：`QC_PASSED`

### 4.4 AI 成功回调

- 触发：`POST /api/v1/internal/ai/callbacks/analysis-result`
- 迁移：`ANALYZING -> REVIEW_PENDING`
- 日志原因码：`AI_CALLBACK_SUCCESS`

### 4.5 AI 失败回调

- 触发：同上
- 迁移：`ANALYZING -> QC_PENDING`
- 日志原因码：`AI_CALLBACK_FAILED`

### 4.6 生成报告

- 触发：`POST /api/v1/cases/{caseId}/reports`
- 当前实现仅在病例状态为 `REVIEW_PENDING` 时自动迁移
- 迁移：`REVIEW_PENDING -> REPORT_READY`
- 日志原因码：`DOCTOR_CONFIRMED`
- 副作用：`report_ready_flag = 1`

### 4.7 报告触发随访

- 触发：`FollowupTriggerService.triggerFromReport`
- 条件：高风险或建议复查
- 迁移：`REPORT_READY -> FOLLOWUP_REQUIRED`
- 日志原因码：`FOLLOWUP_TRIGGERED`
- 副作用：`followup_required_flag = 1`

## 5. 应用层前置校验

### 5.1 迁移到 `ANALYZING`

必须满足：

- 病例存在有效影像
- 至少一张影像质检通过

### 5.2 迁移到 `REVIEW_PENDING`

必须满足：

- 已存在 AI 摘要数据

### 5.3 迁移到 `REPORT_READY` / `FOLLOWUP_REQUIRED` / `CLOSED`

当前只有以下角色可做关键终态相关迁移：

- `DOCTOR`
- `ADMIN`
- `SYS_ADMIN`

系统内部服务使用 `transitionStatusAsSystem` 进行自动迁移。

## 6. 状态日志要求

每次迁移都会写入 `med_case_status_log`，字段包括：

- `case_id`
- `from_status_code`
- `to_status_code`
- `changed_by`
- `change_reason_code`
- `change_reason`
- `changed_at`
- `org_id`

因此，状态迁移在当前实现中已经具备审计留痕能力。

## 7. 与其他子状态的关系

本文只定义病例主状态，不替代以下子状态：

- 影像质检状态：`med_image_quality_check.check_result_code`
- AI 任务状态：`ana_task_record.task_status_code`
- 报告状态：`rpt_record.report_status_code`
- 随访任务状态：`fup_task.task_status_code`

## 8. 当前未覆盖点

- 没有“报告撤回后退回 REVIEW_PENDING”之类的回退规则
- `FOLLOWUP_REQUIRED` 之后当前没有自动迁移到 `CLOSED` 的后台作业，需依赖显式操作
- 代码未引入 BPM/工作流引擎，状态机完全在应用层维护
