# 病例状态机与业务规则矩阵

> 文档定位：病例主状态冻结说明 + 状态迁移规则矩阵  
> 目标：把“状态变化必须可审计”从口头要求落到可执行规则

---

# 1. 适用范围

本文只定义：

- `med_case.case_status_code` 的主状态机；
- 状态迁移触发规则；
- 必须写入的日志规则；
- 状态变化与子流程的关系。

本文不替代：
- 影像质检状态；
- AI 任务状态；
- 报告状态；
- 随访任务状态。

这些状态独立存在于对应子表。

---

# 2. 状态机设计总原则

1. 病例主状态只表达**病例主流程位置**；
2. 子流程细节不得全部塞进 `med_case.case_status_code`；
3. 所有主状态变化必须写入 `med_case_status_log`；
4. 每次变化必须记录操作人、原因、时间；
5. 不允许跳过非法状态；
6. 所有状态变化 API 必须经过应用服务统一流转，不允许 Controller 直接改表。

---

# 3. 病例主状态冻结列表

| 状态码 | 中文含义 | 说明 |
|---|---|---|
| CREATED | 已创建 | 病例已建立，尚未进入质检/分析 |
| QC_PENDING | 待质检 | 已上传影像，等待影像质检 |
| ANALYZING | AI 分析中 | 已发起 AI 任务 |
| REVIEW_PENDING | 待医生复核 | AI 结果已回写，等待医生确认/修正 |
| REPORT_READY | 报告已就绪 | 报告已可查看或已生成 |
| FOLLOWUP_REQUIRED | 需随访 | 已明确需要随访 |
| CLOSED | 已关闭 | 流程结束 |
| CANCELLED | 已取消 | 病例作废、撤销或终止 |

---

# 4. 合法状态迁移矩阵

## 4.1 总矩阵

| 当前状态 | 可迁移到 | 说明 |
|---|---|---|
| CREATED | QC_PENDING, CANCELLED | 上传影像后进入质检；或人工取消 |
| QC_PENDING | ANALYZING, CREATED, CANCELLED | 质检通过进入分析；重置可回 CREATED；作废可取消 |
| ANALYZING | REVIEW_PENDING, QC_PENDING, CANCELLED | AI 成功后待复核；失败重试可退回 QC_PENDING；也可人工取消 |
| REVIEW_PENDING | REPORT_READY, FOLLOWUP_REQUIRED, CANCELLED | 医生确认后可生成报告或直接判定需随访 |
| REPORT_READY | FOLLOWUP_REQUIRED, CLOSED | 需要随访则进入随访；否则可关闭 |
| FOLLOWUP_REQUIRED | CLOSED, CANCELLED | 完成随访后关闭；作废可取消 |
| CLOSED | CLOSED | 终态，不允许普通业务再流转 |
| CANCELLED | CANCELLED | 终态，不允许普通业务再流转 |

## 4.2 特殊说明
- `CREATED -> ANALYZING` 不允许直接发生；
- `ANALYZING -> REPORT_READY` 不允许跳过医生复核；
- `REVIEW_PENDING -> CLOSED` 原则上不允许，除非项目负责人特别批准并保留审计备注；
- `CLOSED` 和 `CANCELLED` 为终态，禁止常规恢复。

---

# 5. 状态迁移触发事件映射

| 业务动作 | 原状态 | 目标状态 | 触发方 |
|---|---|---|---|
| 创建病例 | - | CREATED | 医生 / 筛查员 |
| 上传首张有效影像 | CREATED | QC_PENDING | image 模块 |
| 质检通过并创建分析任务 | QC_PENDING | ANALYZING | analysis 模块 |
| AI 成功并写回摘要 | ANALYZING | REVIEW_PENDING | analysis 模块 |
| 医生确认并生成报告 | REVIEW_PENDING | REPORT_READY | 医生 |
| 医生确认需随访 | REVIEW_PENDING / REPORT_READY | FOLLOWUP_REQUIRED | 医生 / 规则引擎 |
| 随访完成 | FOLLOWUP_REQUIRED | CLOSED | followup 模块 |
| 人工取消 | 非终态 | CANCELLED | 有权限用户 |

# 6. med_case_status_log 落表规则

## 6.1 必须落表字段
每次状态变化必须写入：

- `case_id`
- `from_status_code`
- `to_status_code`
- `changed_by`
- `change_reason_code`
- `change_reason`
- `changed_at`
- `org_id`

说明：
若当前表暂未含 `change_reason_code`，建议通过增量迁移补齐；若暂不补字段，也必须在应用层标准化 `change_reason` 内容格式。

## 6.2 必须记录的原因结构
推荐原因采用：

- 标准原因码；
- 可选自由备注。

示例：

| reason_code | 含义 |
|---|---|
| CASE_CREATED | 病例创建 |
| IMAGE_UPLOADED | 已上传影像 |
| QC_PASSED | 质检通过 |
| QC_FAILED_RETRY | 质检失败后重提 |
| AI_RESULT_RECEIVED | AI 结果已回写 |
| DOCTOR_CONFIRMED | 医生确认 |
| DOCTOR_CORRECTED | 医生修正 |
| REPORT_GENERATED | 报告生成 |
| FOLLOWUP_TRIGGERED | 触发随访 |
| CASE_CLOSED | 病例关闭 |
| CASE_CANCELLED | 病例取消 |

# 7. 子流程状态分层规则

## 7.1 影像质检状态
承载表：
- `med_image_file.quality_status_code`
- `med_image_quality_check.check_result_code`

推荐值：
- PENDING
- PASS
- REJECT
- REVIEW

## 7.2 AI 任务状态
承载表：
- `ana_task_record.task_status_code`

推荐值：
- QUEUEING
- PROCESSING
- SUCCESS
- FAILED
- CANCELLED

## 7.3 报告状态
承载表：
- `rpt_record.report_status_code`

推荐值：
- DRAFT
- FINAL
- ARCHIVED

## 7.4 随访任务状态
承载表：
- `fup_task.task_status_code`

推荐值：
- TODO
- IN_PROGRESS
- DONE
- OVERDUE
- CANCELLED

说明：
子流程状态不得直接覆盖病例主状态语义。

# 8. 状态迁移校验规则

## 8.1 应用服务强校验
所有状态变化都必须通过统一领域服务，例如：

- `CaseDomainService.transition(caseId, targetStatus, reason, operator)`

禁止：
- Controller 直接更新 `med_case`
- Mapper 直接在外层随意改 `case_status_code`

## 8.2 必校验项
迁移前必须校验：

1. 当前状态是否合法；
2. 目标状态是否合法；
3. 当前状态是否允许转到目标状态；
4. 当前操作人是否有权限触发；
5. 必填上下文是否完整。

示例：
- `QC_PENDING -> ANALYZING` 必须存在可分析影像；
- `ANALYZING -> REVIEW_PENDING` 必须存在成功 AI 摘要结果；
- `REVIEW_PENDING -> REPORT_READY` 必须有医生确认动作；
- `REPORT_READY -> FOLLOWUP_REQUIRED` 必须有规则判定或人工确认。

---

# 9. 异常规则

## 9.1 必须抛业务异常的情况
- 非法状态跳转；
- 终态再次流转；
- 缺少操作原因；
- 缺少必要前置数据；
- 越权操作。

## 9.2 推荐异常码
- CASE_STATUS_INVALID
- CASE_TRANSITION_NOT_ALLOWED
- CASE_ALREADY_TERMINAL
- CASE_TRANSITION_REASON_REQUIRED
- CASE_REVIEW_REQUIRED
- CASE_REPORT_NOT_READY
- CASE_FOLLOWUP_PRECONDITION_FAILED

---

# 10. 测试要求

## 10.1 单元测试必须覆盖
- 合法状态跳转成功；
- 非法状态跳转失败；
- 终态不可流转；
- 每次状态变化会写日志；
- reason 为空时失败。

## 10.2 集成测试必须覆盖
- 创建病例后状态为 CREATED；
- 上传影像后变为 QC_PENDING；
- 触发分析后变为 ANALYZING；
- AI 回写后变为 REVIEW_PENDING；
- 医生确认后变为 REPORT_READY。

---

# 11. 输出结论

病例状态机不是“说明性设计”，而是整个后端主链路的业务骨架。

本文件正式冻结以下结论：

1. 病例主状态只保留 8 个状态；
2. 所有状态变化必须审计；
3. 子流程状态与主状态分层管理；
4. 非法跳转必须被业务层阻断；
5. 后续 patient / image / analysis / report / followup 模块必须按本状态机协作。
