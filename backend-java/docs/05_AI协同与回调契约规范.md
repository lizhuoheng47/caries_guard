# AI 协同与回调契约规范

> 文档定位：Java 后端与 Python AI 服务的协同协议  
> 目标：避免 analysis 模块开发时接口反复返工  
> 适用范围：image / analysis / risk / report / correction feedback

---

# 1. 设计原则

1. Java 负责业务编排，不重复实现模型推理；
2. Python 负责模型推理、结果生成、可视化输出；
3. Java 与 Python 边界清晰，避免耦合内部实现；
4. 大任务优先异步；
5. 所有任务必须可追踪、可重试、可审计；
6. 回调必须幂等。

---

# 2. V1 AI 协同表正式角色

## 2.1 ana_task_record
用于：
- 创建 AI 任务；
- 跟踪状态；
- 记录耗时、异常、请求体。

## 2.2 ana_result_summary
用于：
- 承载业务侧可直接消费的 AI 摘要结果；
- 支撑医生页面快速展示；
- 作为报告与复核输入之一。

## 2.3 ana_visual_asset
用于：
- 持有热力图、mask、overlay 等资产引用；
- 与对象存储附件表关联。

## 2.4 ana_correction_feedback
用于：
- 记录医生修正；
- 形成候选回流池；
- 作为后续训练治理的业务侧出口。

## 2.5 med_risk_assessment_record
用于：
- 记录风险评估结果；
- 支撑趋势与随访触发；
- 为报告提供风险输入。

---

# 3. 协同链路总流程

## 3.1 主流程
1. 影像上传完成；
2. Java 创建 `ana_task_record`；
3. Java 发布 MQ 事件或调用 AI 接口；
4. Python 拉取或接收任务；
5. Python 完成推理；
6. Python 通过回调接口把结果发回 Java；
7. Java 幂等写入 `ana_result_summary` / `ana_visual_asset`；
8. Java 更新病例状态为 `REVIEW_PENDING`；
9. 后续由医生确认或修正；
10. 必要时写入 `ana_correction_feedback`。

# 4. Java -> AI 请求契约

## 4.1 推荐方式
- 任务创建：Java 本地写表 + MQ 发布
- 轻量诊断：可保留 HTTP 同步查询能力
- 真实推理：异步为主

## 4.2 推荐请求载荷
```json
{
  "taskNo": "TASK202604110001",
  "taskTypeCode": "INFERENCE",
  "caseId": 3001,
  "patientId": 1001,
  "orgId": 10,
  "modelVersion": "caries-v1",
  "images": [
    {
      "imageId": 5001,
      "attachmentId": 4001,
      "imageTypeCode": "PANORAMIC",
      "objectKey": "case-image/2026/04/11/CASE202604110001/original_01.jpg"
    }
  ],
  "patientProfile": {
    "age": 12,
    "genderCode": "MALE",
    "brushingFreqPerDay": 1,
    "sugarDietLevelCode": "HIGH"
  }
}
```

## 4.3 请求原则
- 不传不必要明文敏感信息；
- 只传 AI 所需字段；
- 结构要版本稳定；
- `taskNo` 必须全局唯一；
- `modelVersion` 必须可追溯。

# 5. AI -> Java 回调契约

## 5.1 回调接口
建议：
`POST /api/v1/internal/ai/callbacks/analysis-result`

## 5.2 回调安全
必须至少满足一种：
- 网关白名单；
- HMAC 签名；
- 内网访问控制；
- 双向认证。

V1 建议：
- 网关白名单 + HMAC 签名。

# 6. 回调幂等规则

## 6.1 正式要求
回调接口必须幂等。

## 6.2 实现建议
以 `taskNo` 为幂等键：

- 若任务已成功处理完成，再次收到相同成功回调，直接返回成功但不重复写入；
- 若任务第一次失败，后续允许在受控场景重试；
- 若收到冲突状态，进入异常日志并人工排查。

## 6.3 推荐幂等顺序
1. 查 `ana_task_record`；
2. 校验 taskNo 是否存在；
3. 判断当前 task_status_code；
4. 若已终态且已完成写回，直接幂等返回；
5. 否则事务写入摘要、资产、风险结果并更新任务状态。

# 7. Java 写回规则

## 7.1 成功回调时
必须完成：
- 更新 `ana_task_record.task_status_code=SUCCESS`
- 写入 `ana_result_summary`
- 写入 `ana_visual_asset`
- 写入 `med_risk_assessment_record`
- 更新病例状态为 `REVIEW_PENDING`
- 记录操作日志 / 事件日志

## 7.2 失败回调时
必须完成：
- 更新 `ana_task_record.task_status_code=FAILED`
- 记录 `error_message`
- 视业务决定是否回退病例状态到 `QC_PENDING` 或保留人工处理入口
- 写异常日志

---

# 8. 医生修正反馈规则

## 8.1 使用时机
当医生对以下结果做修正时，必须写入：
- 分级结果；
- 牙位判定；
- 分割边界；
- 诊断结论。

## 8.2 正式字段要求
`ana_correction_feedback` 至少应保留：
- case_id
- diagnosis_id（可选）
- source_image_id
- doctor_user_id
- original_inference_json
- corrected_truth_json
- feedback_type_code
- is_exported_for_train
- org_id

## 8.3 业务含义
这张表是：
- 业务侧修正留痕；
- 不是训练平台主表；
- 但要作为未来脱敏导出训练样本的桥接出口。

# 9. 风险评估协同

## 9.1 风险评估来源
可以来自：
- AI 回调直接输出；
- Java 聚合 patient profile + AI 结果后再生成；
- 或两者混合。

V1 建议：
- AI 提供风险建议；
- Java 负责落库与业务触发。

## 9.2 风险评估结果用途
- 驱动患者报告；
- 驱动医生报告；
- 触发随访计划；
- 支撑 dashboard 趋势。

---

# 10. 测试要求

## 10.1 analysis 模块最小测试
必须覆盖：
- 创建任务成功；
- MQ/事件发布成功；
- 成功回调写回成功；
- 失败回调写回成功；
- 重复回调幂等；
- 写回后病例状态正确变化。

## 10.2 联调验证清单
- AI 是否能识别 `taskNo`
- Java 是否能根据 `taskNo` 正确落库
- 资产 attachment 是否可访问
- 风险结果是否进入报告模块
- 高不确定性是否进入待复核

---

# 11. 输出结论

AI 协同不是“调个接口”这么简单，而是完整的：

> 任务管理 + 回调协议 + 幂等处理 + 状态变化 + 业务留痕 + 医生修正桥接

本文件冻结这些规则后，analysis 模块即可进入真实开发。
