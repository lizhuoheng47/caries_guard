# 模块说明与代码入口索引

本文档按模块说明“代码现在在哪里、入口是什么、核心类有哪些、每层职责是什么”。

## 1. 总体分层

当前项目主要采用如下分层：
- `controller` / `interfaces`：HTTP 接口层
- `app`：应用服务层，负责组装业务流程
- `domain`：领域模型、领域服务、仓储接口
- `infrastructure`：仓储实现、消息、存储、外部适配
- `interfaces.command / dto / vo / query`：接口入参与出参

原则上：
- 控制器不直接写数据库
- 跨模块状态变更尽量通过对应模块的 AppService 完成
- `med_case` 状态变化统一走病例模块，不允许旁路直接改表

## 2. 启动与公共模块

### 2.1 `caries-boot`

职责：
- Spring Boot 应用入口
- 聚合所有业务模块
- 提供配置装配、组件扫描、接口启动

关键类：
- `com.cariesguard.boot.CariesBootApplication`

### 2.2 `caries-common`

职责：
- 通用响应结构
- 异常码与业务异常
- 基础工具类

调用特点：
- 各模块 Controller 基本都返回统一 `ApiResponse<T>`
- 业务错误通过 `BusinessException` 抛出

### 2.3 `caries-framework`

职责：
- JWT 鉴权
- Spring Security 配置
- 权限切面
- 敏感数据保护
- 当前登录用户上下文

关键类：

| 类名 | 作用 |
| --- | --- |
| `SecurityConfig` | 配置公开接口和鉴权规则 |
| `JwtAuthenticationFilter` | 从 JWT 解析用户并放入安全上下文 |
| `RequirePermissionAspect` | 处理 `@RequirePermission` 注解 |
| `SensitiveDataFacade` | 统一封装加密、哈希、脱敏 |

## 6. 分析模块 `caries-analysis`

职责：
- 组织 AI 分析请求
- 创建任务、重试任务
- 接收 Python / AI 回调
- 落库摘要、可视化资产、风险评估
- 发布 requested/completed/failed 事件
- 保存医生纠偏反馈

### 6.1 控制器

| 控制器 | 路径前缀 | 说明 |
| --- | --- | --- |
| `AnalysisTaskController` | `/api/v1/analysis/tasks` | 创建任务、重试、详情、分页 |
| `AnalysisCallbackController` | `/api/v1/internal/ai/callbacks` | 接收 AI 回调 |
| `CaseAnalysisAliasController` | `/api/v1/cases/{caseId}` | 从病例路径触发分析与纠偏 |
| `CorrectionFeedbackController` | `/api/v1/analysis/corrections` | 直接提交纠偏 |

### 6.2 应用服务

| 类名 | 主要方法 | 作用 |
| --- | --- | --- |
| `AnalysisTaskAppService` | `createTask` `retryTask` | 创建/重试分析任务 |
| `AnalysisQueryAppService` | `getTaskDetail` `pageTasks` | 任务查询 |
| `AnalysisCallbackAppService` | `handleResultCallback` | 验签、写回、状态流转、事件发布 |
| `CorrectionFeedbackAppService` | `submit` | 保存纠偏反馈 |

### 6.3 领域服务与关键实现

| 类名 | 作用 |
| --- | --- |
| `AnalysisTaskDomainService` | 校验病例是否可分析、校验图像是否可分析、生成任务编号 |
| `AnalysisIdempotencyDomainService` | 去重终态回调、限制重试、处理晚到回调 |
| `AnalysisCallbackDomainService` | 校验回调状态、解析摘要聚合字段、决定病例目标状态 |
| `LoggingAnalysisTaskEventPublisher` | `logging` 模式下只打日志 |
| `RabbitAnalysisTaskEventPublisher` | `rabbit` 模式下发 RabbitMQ |
| `AiCallbackSignatureVerifier` | 回调签名校验 |

### 6.4 当前实现要点

- 只会挑选 `qualityStatusCode = PASS` 的图像参与分析
- 同一病例存在运行中任务时，默认不允许再次创建任务
- 重试不会覆盖原任务，而是新建任务并回填 `retry_from_task_id`
- 分析成功会写：
  - `ana_task_record`
  - `ana_result_summary`
  - `ana_visual_asset`
  - `med_risk_assessment_record`
- 分析成功后病例状态转为 `REVIEW_PENDING`
- 分析失败后病例状态回退为 `QC_PENDING`

### 8.2 应用服务

### 8.3 当前自动触发规则

- 风险等级为 `HIGH` 时触发
- 或 `reviewSuggestedFlag = 1` 时触发
- 使用 `case_id + trigger_source_code + trigger_ref_id` 做幂等保护
- 触发后自动创建：
  - `fup_plan`
  - 首个 `fup_task`
  - 一条 `msg_notify_record`
- 尝试推动病例进入 `FOLLOWUP_REQUIRED`

## 10. Python 对接时最关键的代码入口

如果 Python 开发只关心 AI 协同，请重点看这些类：

| 类名 | 原因 |
| --- | --- |
| `AnalysisTaskAppService` | 负责构造发送给 Python 的任务请求载荷 |
| `AiAnalysisRequestDTO` | Java 发给 Python 的请求结构定义 |
| `RabbitAnalysisTaskEventPublisher` | Rabbit 消息头和 routing key 的真实来源 |
| `AiCallbackSignatureVerifier` | Python 回调验签必须对齐的算法来源 |
| `AiAnalysisCallbackDTO` | Python 回调请求体定义 |
| `AnalysisCallbackAppService` | Java 对回调的真实处理逻辑 |
| `FollowupTriggerService` | 风险结果如何转成随访计划 |
| `ReportAppService` | 分析成功之后如何生成报告和触发随访 |
