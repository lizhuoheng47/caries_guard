# 当前项目对齐后的开发文档

## 1. 对齐结论

经过本轮按代码、数据库和测试重写后，当前文档体系应被理解为：

- `00 ~ 05`：当前实现基线与规则说明
- `06 ~ 07`：测试证据与剩余任务
- `09 ~ 17`：模块级完成情况、交付收口与差异说明
- `architecture / development-plan / schema-gap-analysis`：架构、计划、残余差距

## 2. 当前仓库真实定位

当前仓库不是“方案设计稿”，而是：

- 已有接口实现
- 已有数据库迁移
- 已有真实数据库测试
- 仍有若干工程化缺口

因此，所有文档都应同时区分：

- 已实现能力
- 当前库内容
- 尚未补齐的工程化项

## 3. 当前统一口径

### 3.1 已实现

- 登录、JWT、安全过滤
- 患者/就诊/病例主数据
- 影像上传与签名访问
- AI 分析任务与回调
- 医生修正反馈
- 报告生成与导出审计
- 随访自动触发与记录
- dashboard 统计接口

### 3.2 已入库但默认无样例数据

- 绝大多数业务表
- 报告模板
- 权限菜单
- 数据权限规则

### 3.3 未完成

- 完整普通角色权限演示数据
- 全业务数据权限规则下沉
- 生产级 PDF 与外部对象存储落地
- Python AI 消费执行器

## 4. 当前默认演示路径

当前最稳妥的演示路径是：

- 使用 `admin / 123456`
- 通过主链路 API 新建业务数据
- 依赖 boot 真库测试同款链路进行验证

不建议把 `cg` 当前库误认为“已经灌好可展示业务样例”的演示库。

如果要向老师、评审或前端说明当前后端：

- 先看 `README`
- 再看 `00/01/03/15/16/17`
- 模块细节分别看 `09 ~ 14` 与 `P6/P7` 文档
