## 4. 系统域表

### 4.1 `sys_user`

作用：系统用户主表。

当前默认管理员关键字段：
- `id = 100001`
- `user_no = U100001`
- `username = admin`
- `nick_name = Admin`
- `user_type_code = ADMIN`
- `status = ACTIVE`
- `org_id = 100001`

说明：
- 认证时最终是否拥有管理员能力，主要看角色，不只看 `user_type_code`

### 4.2 `sys_role`

作用：系统角色表。

当前种子角色关键字段：
- `id = 100001`
- `role_code = SYS_ADMIN`
- `role_name = System Administrator`
- `status = ACTIVE`
- `org_id = 100001`

### 4.3 `sys_user_role`

作用：用户与角色关联表。

当前只有一条记录，把 `admin` 绑定到 `SYS_ADMIN`。

### 4.4 `sys_menu`

作用：菜单、按钮、权限点定义。

当前状态：
- 表存在
- 当前数据为 0

直接影响：
- 执行 Flyway V014 后可演示患者、就诊、病例、影像、分析、报告、随访、看板菜单树
- 执行 Flyway V014 后可演示 ORG_ADMIN、DOCTOR、SCREENER 的基础菜单权限链路

### 4.5 `sys_data_permission_rule`

作用：角色在模块维度上的数据权限规则。

当前状态：
- 表结构存在
- 可通过接口维护
- 默认数据为空
- 业务模块尚未做到全量下推使用

### 4.6 `sys_dict_type` / `sys_dict_item`

作用：系统字典类型与字典项。

当前状态：
- 已有种子数据
- 用于性别、状态、业务类型等枚举值支撑

## 5. 患者病例域表

### 5.1 `pat_patient`

作用：患者主档。

职责字段类别：
- 主身份：`patient_no`、姓名、性别、出生日期
- 隐私保护：明文加密列、哈希列、脱敏列
- 业务属性：来源、首诊日期、隐私等级、状态
- 组织维度：`org_id`

说明：
- 当前创建患者时会同步维护监护人信息
- 身份证号有重复校验

### 5.2 `pat_guardian`

作用：监护人信息。

说明：
- 与患者关联
- 同样涉及证件与电话类敏感信息保护

### 5.3 `pat_profile`

作用：患者扩展画像。

当前业务使用程度：
- 表已存在
- 当前核心主链路使用有限

### 5.4 `med_visit`

作用：就诊记录。

说明：
- 一个患者可以有多次就诊
- 病例创建时需要明确挂到某次就诊下

### 5.5 `med_case`

作用：病例主表，是整个主链路核心主表。

关键职责：
- 串联患者、就诊、图像、分析、报告、随访
- 承担主状态机字段 `case_status_code`
- 维护是否需要随访等标记

### 5.6 `med_case_status_log`

作用：病例状态流转日志。

说明：
- 每次通过病例模块做状态变更，都会留下日志
- 这是追踪链路是否符合规则的重要证据表

### 5.7 `med_case_diagnosis`

作用：病例诊断结论。

说明：
- 接口保存方式是按病例整组替换
- 不做单条 patch 更新

### 5.8 `med_case_tooth_record`

作用：牙位级别结构化记录。

说明：
- 也是替换式保存
- 会校验同一个病例内牙位记录重复问题

## 6. 图像与附件域表

### 6.1 `med_attachment`

作用：附件元数据表。

关键字段语义：
- 文件原始名
- bucket / object_key
- 内容类型、后缀、文件大小、MD5
- provider code
- 访问级别

说明：
- 当前不仅图像附件走这里，报告 PDF 附件也会走这里

### 6.2 `med_image_file`

作用：病例图像业务表。

关键语义：
- 与病例、就诊、患者三者绑定
- 关联附件 ID
- 图像类型、来源、拍摄时间、是否主图
- 当前质检状态

### 6.3 `med_image_quality_check`

作用：图像质检结果。

当前结构特征：
- 记录质检类型、结果、多个评分字段
- 记录问题码列表和建议文本
- 通过 `current_flag` 表示当前有效质检结果

说明：
- 分析任务只会挑选 `PASS` 图像

## 7. 分析域表

### 7.1 `ana_task_record`

作用：AI 分析任务主表。

关键字段：
- `task_no`
- `case_id`
- `patient_id`
- `model_version`
- `task_type_code`
- `task_status_code`
- `payload_json`
- `started_at`
- `completed_at`
- `retry_from_task_id`

说明：
- 当前重试机制通过 `retry_from_task_id` 保留链路，而不是覆盖原任务

### 7.2 `ana_result_summary`

作用：分析结果摘要。

当前直接落库的聚合字段：
- `overall_highest_severity`
- `uncertainty_score`
- `review_suggested_flag`

这些字段来源：
- 优先从 AI 回调的 `summary` 结构提取
- 用于报告、列表、统计、答辩展示时直接查询

### 7.3 `ana_visual_asset`

作用：分析可视化资产。

说明：
- 保存 AI 标注图、热力图等附件引用
- 当前只记录附件引用，不负责生成附件本身

### 7.4 `ana_correction_feedback`

作用：医生纠偏反馈。

说明：
- 用于保存原推断结果和修正后的真值 JSON
- 当前不自动回流训练，仅做业务留痕

### 7.5 `med_risk_assessment_record`

作用：病例风险评估结果。

关键语义：
- `overall_risk_level_code`
- `assessment_report_json`
- `recommended_cycle_days`

说明：
- 随访触发直接依赖这个表中的风险等级和推荐周期

## 8. 报告域表

### 8.1 `rpt_template`

作用：报告模板表。

说明：
- 可维护不同 `reportTypeCode` 的模板内容
- 当前如果库里没有激活模板，会使用代码内置默认模板

### 8.2 `rpt_record`

作用：报告记录主表。

关键语义：
- 报告编号、病例、患者、报告类型、版本号
- 生成状态
- 摘要文本
- 归档附件 ID
- 生成时间

说明：
- 当前生成流程是先草稿、后归档、再置最终状态

### 8.3 `rpt_export_log`

作用：报告导出审计表。

说明：
- `exportReport` 接口当前只会插入此表
- 并不会直接把 PDF 文件流回传给调用方

## 9. 随访域表

### 9.1 `fup_plan`

作用：随访计划主表。

关键语义：
- 计划编号
- 病例、患者
- 计划类型、状态、负责人
- 下次应随访日期、间隔天数
- `trigger_source_code`
- `trigger_ref_id`

说明：
- `trigger_source_code + trigger_ref_id + case_id` 用于幂等判断

### 9.2 `fup_task`

作用：随访任务表。

关键语义：
- 任务编号
- 计划 ID、病例 ID、患者 ID
- 任务类型、状态、指派人、截止日期

当前状态码：
- `TODO`
- `IN_PROGRESS`
- `DONE`
- `OVERDUE`
- `CANCELLED`

### 9.3 `fup_record`

作用：随访执行记录。

关键语义：
- 联系方式
- 联系结果
- 是否跟进下一次
- 下次间隔天数
- 结果摘要、医生备注

说明：
- 写入记录后，任务通常会被推进为完成

### 9.4 `msg_notify_record`

作用：站内消息/提醒留痕。

说明：
- 自动触发随访或任务逾期时会尝试写消息记录
- 当前是留痕，不等同于真实短信/微信网关已接入

## 10. 当前种子与缺口总结

当前系统可直接依赖的种子：
- 管理员用户
- 管理员角色
- 管理员与角色关联
- 部门根节点
- 基础字典

当前缺失的默认业务数据：
- 患者
- 就诊
- 病例
- 图像
- 分析结果
- 报告
- 随访

当前缺失的默认系统数据：
- 菜单
- 角色菜单关联
- 数据权限规则样例

如果要做前后端演示或 Python 联调，通常需要先补：
1. 业务样例数据
2. 菜单与角色菜单数据
3. Rabbit 消费端或 logging 模式假联调
