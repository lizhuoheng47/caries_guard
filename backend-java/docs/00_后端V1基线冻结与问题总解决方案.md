> 文档定位：后端 V1 冻结总纲 + 问题总解决方案  
> 适用对象：项目负责人、后端开发、数据库设计、AI 联调、测试

# 1. 文档目标

本文用于一次性解决当前后端推进中的核心问题：

1. 文档、Flyway、Java 代码三者口径不完全一致；
2. 部分规则只写在说明中，尚未真正落到数据库与代码约束；
3. 当前工程已具备基础底座，但尚未形成可演示的业务闭环；
4. 若继续无冻结开发，后期极易产生字段漂移、状态混乱、重复返工。

本文目标不是“补充说明”，而是正式冻结：

- V1 正式业务范围；
- 数据库与迁移基线；
- 状态机与核心业务约束；
- 权限、安全、AI 协同口径；
- 模块开发顺序与验收标准。

# 3. V1 正式范围冻结

## 3.1 V1 架构冻结

当前后端架构正式冻结为：

> **模块化单体 + 独立 AI 服务 + RabbitMQ 异步解耦**

说明：

- 当前不拆微服务；
- 当前不引入复杂服务治理体系；
- 所有业务模块在一个 Java 工程内按领域分模块实现；
- Python AI 服务作为独立外部服务边界存在；
- Java 负责业务编排、任务调度、结果承载、状态流转、审计与报告随访闭环。

# 4. 当前存在的核心问题总览

## 4.1 P0 级问题（必须先解决）

### 问题 1：AI 协同表口径不统一
现状：
- 部分总文档曾将 AI 协同表视为“边界外”；
- 当前 Flyway 已经存在 `ana_task_record`、`ana_result_summary`、`ana_correction_feedback`、`ana_visual_asset`；
- 业务闭环又明确需要它们。

解决：
- 正式冻结为 V1 业务侧 AI 协同表；
- 文档、代码、迁移统一口径；
- 不再使用“临时表/过渡表”说法。

### 问题 2：`med_image_quality_check` 缺失或未正式落库
现状：
- 业务设计明确要求一张图可多次质检；
- 当前部分迁移未落此表。

解决：
- 在 image 模块实施前补充 Flyway 迁移；
- 明确“最新有效质检记录”查询策略；
- 不允许把所有质检语义挤进 `med_image_file` 单表。

### 问题 3：主图唯一规则未工程化
现状：
- 规则要求同病例、同影像类型仅允许一条 `is_primary=1`；
- 当前多为说明约束，未必有强校验。

解决：
- 数据库 + 事务服务双重约束；
- 保存主图前先清旧主图；
- 并发场景引入悲观锁或幂等保护；
- 落地模块级集成测试。

### 问题 4：敏感数据方案只到字段层，未到实现层
现状：
- 已设计 `*_enc / *_hash / *_masked`；
- 尚未在 Java 代码实现统一加密/哈希/脱敏服务。

解决：
- 在 P1/P2 前冻结加密方案；
- 提供统一 TypeHandler / CryptoService / MaskingService；
- 严禁业务代码直接处理明文落库。

### 问题 5：data scope 已建模但未生效
现状：
- `sys_data_permission_rule` 已存在；
- `org_id` 也广泛存在；
- 但查询权限控制尚未在系统模块真正落地。

解决：
- 先完成 `system` 模块；
- 固化 org-aware 查询模式；
- 对 patient/case/image/report 等模块统一透传数据范围上下文。

# 5. V1 开发总原则

## 5.1 主原则
1. **先冻结，再开发**；
2. **先主链路，再增强功能**；
3. **先约束落地，再写页面功能**；
4. **数据库规则与服务规则双保险**；
5. **所有状态变化必须审计可追溯**；
6. **任何跨模块开发不得绕过命名规范与字段冻结字典**。

## 5.2 单一真相来源
V1 中各类事实来源统一如下：

- 字段命名与表字段语义：以《完整数据字典》为主；
- 当前真实数据库基线：以仓库内 Flyway 为主；
- 若二者不一致：必须通过增量迁移纠正，并同步修正文档；
- 禁止“代码先改，文档以后再补”。

# 6. 推荐开发顺序

## P0 基线对齐
目标：
- 统一文档、Flyway、代码基线；
- 明确 V1 表范围；
- 补齐缺失迁移项。

## P1 system 模块完成
目标：
- 把 system 变成真正的权限与机构隔离底座；
- 完成 RBAC 基础管理、字典、配置、日志、数据范围雏形。

## P2 patient + case 核心链路
目标：
- 打通患者 -> 就诊 -> 病例；
- 实现病例状态机与状态流转日志。

## P3 image 模块
目标：
- 完成附件、影像、质检、主图规则、对象存储接入。

## P4 analysis 模块
目标：
- 完成 AI 任务、回调、结果写回、修正反馈入口。

## P5 report 模块
目标：
- 完成医生版/患者版报告生成、版本归档、导出审计。

## P6 followup 模块
目标：
- 形成风险/复核结果驱动的随访闭环。

## P7 dashboard 模块
目标：
- 为答辩与演示提供聚合展示能力。

## P8 测试与交付
目标：
- 用测试、脚本、文档把全链路真正封口。

# 8. 国一标准下的真实验收口径

V1 是否能支撑国一级别后端答辩，主要看以下问题能否回答“是”：

1. 不是只会登录，而是真的有权限体系；
2. 不是只会建表，而是真的能跑患者到报告主链路；
3. 不是只写文档规则，而是规则已经写进 migration + service + test；
4. 不是只讲 AI，而是 Java 侧业务编排、结果承载、审计、修正已闭环；
5. 不是只会做接口，而是可以稳定演示 1 条完整病例流程。

如果以上 5 条都能满足，后端部分就会从“方案好看”跃迁为“工程可信”。
