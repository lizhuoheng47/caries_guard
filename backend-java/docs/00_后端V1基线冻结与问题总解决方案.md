# 后端 V1 基线冻结与问题总解决方案

## 1. 文档定位

本文不再描述“目标态方案”，而是冻结 `2026-04-14` 时点的真实后端基线，作为代码、数据库、测试和答辩口径的统一入口。

## 2. 当前冻结范围

当前仓库已经冻结并可对外说明的范围：

- 多模块单体架构与模块边界
- `cg` / `cg_e2e` 两套 MySQL schema 与 Flyway 迁移
- system / patient / image / analysis / report / followup / dashboard 七个业务域的接口层
- 病例状态机、AI 回调验签、报告生成、随访触发和看板统计口径
- boot 层真库 E2E 与集成测试

## 3. 当前系统结论

结论不是“骨架完成”，而是“闭环代码已实现，业务样例数据默认未灌入”。

当前真实情况：

- 代码层：核心主链路已经完整串起
- 数据库层：Flyway 已执行到 `V013`
- 本地库内容：默认只有系统种子数据，没有预置业务病例/报告/随访样例
- 测试层：仓库内已有多条真实数据库测试链路

## 4. 已冻结的核心业务闭环

当前可确认的主链路是：

1. 登录
2. 查询当前用户
3. 创建患者
4. 创建就诊
5. 创建病例
6. 上传附件并关联影像
7. 影像质检通过
8. 创建 AI 分析任务
9. AI 回调写回摘要/可视化/风险
10. 医生提交修正反馈
11. 生成报告并归档附件
12. 自动触发随访计划/任务
13. 录入随访记录并收口计划
14. dashboard 聚合校验

## 5. 数据库冻结结论

当前数据库基线不是“待整改”，而是：

- 系统管理表、病例主表、影像表、分析表、报告表、随访表均已存在
- `med_image_quality_check`、`ana_result_summary` 聚合列、`ana_task_record.retry_from_task_id`、`fup_task/fup_record` 对齐字段均已落到迁移中
- 本地 `cg` 库共有 `38` 张表（含 `flyway_schema_history`）

当前默认种子数据：

- `SYS_ADMIN` 角色 1 条
- `admin` 用户 1 条
- 用户角色绑定 1 条
- 字典类型 4 条
- 字典项 18 条

## 6. 安全与权限冻结结论

当前已经落地：

- JWT 登录与鉴权过滤器
- `@RequirePermission` 权限切面
- 敏感字段加密、哈希、脱敏
- 文件签名访问
- AI 回调验签
- 业务服务中的 `org_id` 机构隔离检查

当前不能宣称已经完成：

- 全量 RBAC 菜单初始化
- 全业务模块的数据权限规则下推
- 细粒度列级脱敏策略执行

## 7. 外部依赖冻结结论

当前项目对外依赖的真实状态：

- MySQL：已真实接入
- RabbitMQ：本地 profile 配置为真实发布；E2E profile 切换为 logging publisher
- Redis：配置存在，但业务代码未形成关键链路依赖
- 对象存储：接口抽象已完成，当前实现为本地文件系统
- Python AI：仓库内没有服务实现，只有消息/回调契约

## 8. 当前主要风险

- 普通角色如果未补 `sys_menu` 与权限菜单数据，很多受保护接口无法直接使用
- 报告导出目前是“审计留痕”，不是单独的下载流接口
- PDF 生成器为极简 ASCII 实现，不适合作为生产版正式报告输出
- dashboard 直接查业务表，适合当前规模，不适合高并发大屏场景

## 9. 后续所有文档口径

从本次更新开始，所有文档统一遵守以下规则：

- “已实现”只写代码和数据库中真实存在的能力
- “当前库内容”只写 `cg` 中真实可见的数据
- “剩余工作”单独列出，不混入“已完成”描述
- 阶段号 `P*` 仅表示开发阶段来源，不再代表功能尚未实现
