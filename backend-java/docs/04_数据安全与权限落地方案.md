# 数据安全与权限落地方案

> 文档定位：敏感数据保护、RBAC、org 隔离、data scope 落地方案  
> 目标：把“安全设计”变成真实工程能力

---

# 1. 目标与范围

本文解决 4 类问题：

1. 敏感字段如何真正落地加密与脱敏；
2. RBAC 如何在 Java 工程中真正生效；
3. `org_id` 如何变成真实的数据隔离锚点；
4. `sys_data_permission_rule` 如何从表设计变成可执行逻辑。

---

# 2. 安全设计总原则

1. 不在数据库裸存 PII 明文；
2. 不在日志输出敏感明文；
3. 不通过前端传参控制权限；
4. 不允许服务层绕过 org/data-scope 检查；
5. 不允许测试代码使用“先跳过安全，后面再补”的方式长期存在。

---

# 3. 敏感数据字段分级

## 3.1 高敏字段
必须加密：
- 患者姓名
- 手机号
- 身份证/证件号
- 监护人姓名
- 监护人手机号
- 用户真实姓名
- 用户手机号
- 用户证件号

## 3.2 中敏字段
建议脱敏或权限控制：
- 出生日期
- 临床备注
- 诊断描述
- 报告摘要
- 慢性病描述
- 过敏信息

## 3.3 普通字段
可按常规处理：
- 枚举状态
- 类型字段
- 非敏感编号
- 通用时间字段

# 4. enc / hash / masked 三段式正式实现

## 4.1 字段含义
- `*_enc`：用于安全存储；
- `*_hash`：用于等值检索、去重、唯一校验；
- `*_masked`：用于前端展示脱敏摘要。

## 4.2 V1 实现方案
### 加密
- 推荐：AES/GCM
- 使用统一 `CryptoService`
- 密钥通过环境变量或密钥管理配置注入
- 禁止业务代码散落自行加密

### 哈希
- 推荐：HMAC-SHA256 + 项目盐值
- 使用统一 `HashService`
- 保证同值输入输出稳定，便于检索

### 脱敏
- 使用统一 `MaskingService`
- 在 DTO / VO 输出时处理
- 不建议让前端自行脱敏

## 4.3 Java 落地建议
- DO 层字段保留数据库形态；
- 应用层封装明文输入 -> enc/hash/masked 生成；
- 或对部分字段用 MyBatis TypeHandler，但 hash/masked 仍建议显式服务计算。

---

# 5. 推荐组件设计

## 5.1 CryptoService
职责：
- encrypt(String plain)
- decrypt(String cipher)

## 5.2 HashService
职责：
- hmacSha256(String value)
- normalizeThenHash(String value)

## 5.3 MaskingService
职责：
- maskName(String name)
- maskPhone(String phone)
- maskIdCard(String idCard)
- maskBirthDate(String birthDate)

## 5.4 SensitiveDataFacade
职责：
- 对 patient / guardian / user 等场景做统一转换
- 避免业务代码到处复制加密逻辑

# 6. RBAC 权限落地方案

## 6.1 功能权限模型
基础关系：
- 用户 -> 角色
- 角色 -> 菜单/按钮/API
- 用户 -> 权限集合

## 6.2 Java 生效方式
登录后：
1. 解析 JWT；
2. 加载当前用户；
3. 聚合角色与权限；
4. 存入 SecurityContext；
5. Controller 通过注解或鉴权器校验。

## 6.3 推荐注解
可自定义：
- `@RequirePermission("patient:create")`
- `@RequirePermission("case:review")`

说明：
若暂未实现自定义注解，至少要在 Service 层显式校验权限点。

# 7. org_id 行级隔离方案

## 7.1 正式规则
所有主业务表都必须有 `org_id`，并且：

> 默认查询必须带当前登录用户的 org 范围约束。

## 7.2 落地方式
推荐两层：

### 第一层：Repository 查询参数注入
分页查询、列表查询默认带：
- currentOrgId
- dataScope 条件

### 第二层：Service 层业务校验
当用户通过 ID 访问记录时：
- 必须校验目标记录属于当前可访问 org 范围；
- 不允许因为“主键命中”就绕过机构隔离。

## 7.3 超级管理员
`ADMIN` 可允许跨 org 查看，但必须在 system 模块内特殊定义，不得默认所有角色越权。

# 8. data scope 落地方案

## 8.1 数据范围类型
- ALL
- ORG
- DEPT
- SELF
- CUSTOM

## 8.2 推荐实现策略
在登录后构造 `DataScopeContext`：

```text
userId
orgId
deptId
roleCodes
scopeType
customDeptIds
```

查询业务数据时统一透传。

## 8.3 各范围含义

### ALL
不加额外组织限制（仅限高权限角色）

### ORG
只允许访问当前 org_id

### DEPT
只允许访问当前部门或部门树

### SELF
只允许访问本人相关数据  
例如：
- 自己创建的病例
- 自己负责的病例
- 自己上传的影像

### CUSTOM
从 `sys_data_permission_rule.dept_ids_json` 动态取范围

# 9. 模块落地要求

## 9.1 system 模块
必须实现：
- 登录鉴权；
- 用户角色权限聚合；
- 字典与配置查询；
- 登录日志；
- 操作日志；
- data scope 基础能力。

## 9.2 patient 模块
必须实现：
- 创建/更新患者时的敏感字段加密；
- 查询患者时按角色脱敏返回；
- 患者列表按 org/data-scope 查询。

## 9.3 case / image / report 模块
必须实现：
- 所有查询默认 org-aware；
- 详情查询做归属校验；
- 报告导出记审计日志。

---

# 10. 日志安全规范

## 10.1 禁止打印内容
严禁在日志中打印：
- 明文姓名
- 明文手机号
- 明文证件号
- 明文 JWT
- 明文报告全文
- 原始临床敏感备注

## 10.2 允许打印内容
- traceId
- userId
- orgId
- caseId
- reportId
- taskNo
- 状态码
- 错误码

---

# 11. 测试要求

## 11.1 安全测试
必须覆盖：
- 未登录拒绝访问；
- 无权限拒绝访问；
- 跨 org 访问失败；
- 敏感字段落库后非明文；
- 查询返回字段已脱敏。

## 11.2 权限测试
至少覆盖：
- ADMIN 可访问系统管理；
- DOCTOR 可访问病例与报告；
- SCREENER 可创建基础病例但不能签发报告；
- RESEARCHER 不可直接读业务高敏明文。

---

# 12. 输出结论

本文件正式冻结以下要求：

1. V1 必须真实实现敏感字段保护；
2. V1 必须真实实现 RBAC 与 org 隔离；
3. `sys_data_permission_rule` 不能只建表不用；
4. 所有业务模块都必须依赖 system 提供的安全上下文。
