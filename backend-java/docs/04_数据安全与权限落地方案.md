# 数据安全与权限落地方案

## 1. 当前已落地能力

### 1.1 认证

- 登录接口：`POST /api/v1/auth/login`
- 鉴权方式：JWT Bearer Token
- 过滤器链：`JwtAuthenticationFilter`
- 未登录访问受保护接口时统一走 `RestAuthenticationEntryPoint`

### 1.2 权限校验

- 注解：`@RequirePermission`
- 切面：`RequirePermissionAspect`
- 权限来源：`sys_menu.permission_code`
- 当前特权角色白名单：`ADMIN`、`SYS_ADMIN`

### 1.3 敏感字段处理

当前已在 system/patient 场景中落地：

- 加密：`DefaultCryptoService`，算法 `AES/GCM/NoPadding`
- 哈希：`DefaultHashService`，`HmacSHA256`
- 脱敏：`DefaultMaskingService`

典型字段：

- 用户真实姓名、手机号、邮箱、证件号
- 患者姓名、出生日期、手机号、证件号
- 监护人姓名、手机号、证件号

### 1.4 文件安全

- 附件内容下载入口不要求 JWT
- 但必须带 `expireAt + signature`
- 签名算法为 `HmacSHA256`
- URL 有效期由 `caries.image.storage.access-url-expire-seconds` 控制

### 1.5 AI 回调安全

- Header：`X-AI-Timestamp`
- Header：`X-AI-Signature`
- 签名内容：`timestamp + "." + rawBody`
- 时钟偏移窗口默认 `300` 秒

## 2. 当前权限模型的真实状态

### 2.1 已完成部分

- 登录成功后可查询当前用户和当前权限集
- 系统后台管理接口统一挂了权限码
- `SYS_ADMIN` 角色可绕过细项权限码校验
- 各业务服务普遍补了 `org_id` 机构访问校验

### 2.2 当前数据库事实

本地 `cg` 库当前：

- `sys_role` 中已有 1 条 `SYS_ADMIN`
- `sys_user_role` 中已有 `admin -> SYS_ADMIN`
- `sys_menu` 当前为 0 条
- `sys_data_permission_rule` 当前为 0 条

因此：

- 本地库能直接用 `admin` 跑通，因为它是 `SYS_ADMIN`
- 普通角色 + 菜单权限场景在当前 `cg` 库中默认无法直接演示

## 3. org 隔离的当前做法

当前 patient / image / analysis / report / followup 业务主要通过应用服务中的 `ensureOrgAccess` 控制：

- `ADMIN` / `SYS_ADMIN` 可跨机构
- 其他角色只能访问 `record.org_id == currentUser.orgId` 的数据

这是当前仓库最真实、最稳定的隔离方式。

## 4. 数据权限规则的当前落地程度

当前仓库中已经存在：

- `sys_data_permission_rule` 表
- `SystemDataScopeService`
- `SystemDataScopeRuleRepository`
- `DataScopeContext`

但当前真正全面落地的只有：

- system 模块相关查询/管理支撑
- 文档/表结构/服务骨架

未全面落地的部分：

- patient/image/report/followup/dashboard 并未统一按数据权限规则改写查询 SQL
- 当前多业务查询仍主要依赖 `org_id` 边界，而非 `ALL/ORG/DEPT/SELF/CUSTOM` 全矩阵

## 5. 操作审计现状

### 5.1 登录日志

已落地表：`sys_login_log`

由 `LoginAuditService` 写入，覆盖：

- 用户名
- 登录状态
- userId / orgId
- 客户端 IP
- user-agent
- 失败原因
- 登录时间

### 5.2 操作日志

已落地表：`sys_oper_log`

当前通过 `@SystemOperationLog` + AOP 记录，但覆盖面主要集中在：

- auth
- system 管理接口

注意：不是所有 patient/image/report/followup 接口都已经补全操作日志注解。

## 6. 当前剩余安全缺口

- 菜单与权限码初始化数据不足，普通角色不可演示
- 数据权限规则没有全面下沉到所有模块查询 SQL
- 操作日志并未覆盖所有业务写接口
- 文件签名访问当前不与用户身份绑定，只与签名串绑定
- 默认本地密钥仍使用示例值，部署时必须通过环境变量替换
