## 4. 主链路证据点

### 4.1 登录与系统侧

证据：

- 登录接口成功
- `/me` 返回当前用户
- `sys_login_log` 成功落库

答辩价值：

- 说明系统有基础认证闭环
- 说明日志留痕不是口头描述，而是实际落库

### 4.2 患者与病例

证据：

- 患者建档成功
- 敏感字段加密 / 哈希 / 脱敏落库
- 患者详情返回脱敏视图
- 就诊创建成功
- 病例创建成功
- `med_case_status_log` 记录 `CASE_CREATED`

答辩价值：

- 说明数据安全不是后补
- 说明病例状态机从建档开始就有审计链

### 4.3 影像与质检

证据：

- 文件上传成功
- 影像关联成功
- 质检记录成功写入
- `CREATED -> QC_PENDING`
- 状态日志有 `IMAGE_UPLOADED`

答辩价值：

- 说明病例不是直接跳分析，而是经过影像入库和前置质检

### 4.4 AI 分析

证据：

- 分析任务创建成功
- AI 成功回调写回成功
- `ana_result_summary`
- `ana_visual_asset`
- `med_risk_assessment_record`
- `QC_PENDING -> ANALYZING -> REVIEW_PENDING`

答辩价值：

- 说明 AI 输出不是一句话，而是有任务、摘要、可视化资产、风险结果和状态机联动

### 4.5 医生修正

证据：

- 修正反馈接口成功
- `ana_correction_feedback` 落库成功

答辩价值：

- 说明系统具备人机协同闭环，不是纯黑盒推理

### 4.6 报告与审计

证据：

- 医生版报告生成成功
- 报告 PDF 已归档
- 导出成功
- `rpt_export_log` 留痕成功
- `REVIEW_PENDING -> REPORT_READY`

答辩价值：

- 说明报告不是一次性页面数据，而是带版本与导出审计的正式产物

### 4.7 随访闭环

证据：

- 自动触发 `fup_plan`
- 自动生成 `fup_task`
- `msg_notify_record` 留痕
- 写入 `fup_record`
- 首任务 `DONE`
- 计划自动收口 `DONE`
- `REPORT_READY -> FOLLOWUP_REQUIRED`

答辩价值：

- 说明系统不是“出报告就结束”，而是具备高风险病例管理闭环

### 4.8 dashboard

证据：

- overview 聚合正确
- 风险分布正确
- 随访任务统计正确
- backlog 正确
- model runtime 正确
- trend 按天补零正确

答辩价值：

- 说明平台已经具备管理视角、演示视角和运营观察视角
