# 文档说明与项目总览

本文档按 `2026-04-15` 的工作区、源码和本地数据库 `cg` 现状整理。

## 1. 文档目标

本目录只保留“当前项目已经实现了什么”的最终参考文档，不再保留开发计划、阶段检查、过程记录、差异追踪、里程碑拆解之类的过程性材料。

目标读者：
- Java 维护开发
- Python / AI 服务开发
- 联调、测试、答辩、交接人员

使用原则：
- 以当前代码为准

## 2. 当前项目组成

项目是一个基于 Spring Boot 的口腔龋病业务后端，采用 Maven 多模块结构。

模块清单：

| 模块 | 作用 |
| --- | --- |
| `caries-boot` | 启动模块，整合所有业务模块并提供应用入口 |
| `caries-common` | 通用返回、异常、工具、基础常量 |
| `caries-framework` | 安全、JWT、权限切面、敏感数据加解密、Web 配置 |
| `caries-system` | 登录认证、用户/角色/菜单/字典/配置、数据权限规则 |
| `caries-patient` | 患者、监护人、就诊、病例、病例诊断、牙位记录、状态流转 |
| `caries-image` | 附件上传、对象存储、本地文件访问、病例图像、图像质检 |
| `caries-analysis` | AI 分析任务创建、重试、回调写回、纠偏反馈、分析事件发布 |
| `caries-report` | 报告模板、报告渲染、PDF 生成、报告导出审计 |
| `caries-followup` | 随访计划、随访任务、随访记录、报告触发随访 |
| `caries-dashboard` | 看板统计、趋势查询、模型运行指标 |
| `caries-integration` | 预留的集成层，当前业务承载较少 |

启动类：
- `caries-boot/src/main/java/com/cariesguard/boot/CariesBootApplication.java`

## 3. 当前运行基线

本地 profile 以 `caries-boot/src/main/resources/application-local.yml` 为准。

关键配置：
- HTTP 端口：`8080`
- MySQL：`jdbc:mysql://127.0.0.1:3306/cg`
- Redis：`127.0.0.1:6379`
- RabbitMQ：`127.0.0.1:5672`
- Flyway：启用
- 分析消息模式：`caries.analysis.messaging.mode: rabbit`
- 图像/报告对象存储：已实现 `MINIO` 与 `LOCAL_FS` 两种 provider，local 默认 `MINIO`，E2E 使用 `LOCAL_FS`
- 本地对象存储根目录：`${user.dir}/var/image-storage`

补充说明：
- `provider-code=MINIO` 对应 `MinioObjectStorageService`，`provider-code=LOCAL_FS` 对应 `LocalObjectStorageService`
- 因此“存储提供商代码”和“真实存储方式”当前并不一致，文档必须按实现解释
- `application-e2e.yml` 会把分析消息模式切成 `logging`，用于端到端测试

## 4. 当前数据库真实状态

当前本地库：`cg`

真实数据现状：
- `sys_user = 1`
- `sys_role = 1`
- `sys_user_role = 1`
- `sys_menu` 由 Flyway `V014__14_minio_model_governance_and_permissions.sql` 初始化业务菜单
- `pat_patient = 0`
- `med_visit = 0`
- `med_case = 0`
- `med_image_file = 0`
- `ana_task_record = 0`
- `ana_result_summary = 0`
- `rpt_record = 0`
- `fup_plan = 0`
- `fup_task = 0`
- `fup_record = 0`

这意味着：
- 当前默认本地库只有系统种子数据
- 没有病例、影像、分析、报告、随访的演示数据
- 所有业务链路可以通过接口跑通，但需要先手工造业务数据

默认管理员种子：
- 用户名：`admin`
- 用户 ID：`100001`
- 用户编号：`U100001`
- 角色：`SYS_ADMIN`
- 机构 ID：`100001`

当前库里没有菜单种子，因此：
- `sys_menu` 由 Flyway V014 初始化业务菜单；已有库需执行迁移后生效
- 非管理员角色的菜单授权和前端菜单回显需先执行到 Flyway V014；V014 已内置 ORG_ADMIN、DOCTOR、SCREENER 及业务菜单授权

## 5. 当前已经实现的主链路

已实现的后端主链路：
1. 登录获取 JWT
2. 创建患者
3. 创建就诊
4. 创建病例
5. 上传附件
6. 绑定病例图像
7. 保存图像质检
8. 创建 AI 分析任务
9. AI 回调写回分析结果
10. 生成报告
11. 满足条件时自动触发随访
12. 查询看板统计

对应的病例状态大致流向：
- `CREATED`
- `QC_PENDING`
- `ANALYZING`
- `REVIEW_PENDING`
- `REPORT_READY`
- `FOLLOWUP_REQUIRED`
- `CLOSED`

失败/撤销分支：
- `QC_PENDING -> CREATED`
- `ANALYZING -> QC_PENDING`
- 各阶段可进入 `CANCELLED`

## 6. 当前实现边界与限制

这些限制是“当前事实”，不是待办备注：

1. 图像对象存储是本地文件系统
- Python 服务如果要直接读取对象，需要和 Java 服务共享文件系统路径，或者额外增加取图接口适配。

2. 分析请求消息默认走 RabbitMQ
- 本地 `local` profile 默认不是 `logging`。
- 如果 Python 消费端未启动，分析任务只会入队，不会自动完成。

3. AI 回调只开放一个内部接口
- `POST /api/v1/internal/ai/callbacks/analysis-result`
- 必须带签名头

4. 报告导出接口只记录导出日志
- `POST /api/v1/reports/{reportId}/export` 当前不会直接返回文件字节流
- 其真实作用是写 `rpt_export_log`

5. PDF 生成器是极简实现
- 类为 `ReportPdfService`
- 仅支持 ASCII
- 非 ASCII 字符会被替换为 `?`
- 行数和行宽有限制，不适合作为正式生产排版方案

6. 数据权限框架存在，但未完整下推到所有业务模块
- 系统模块已有角色、权限、数据权限规则能力
- 业务模块多数仍以 `org_id` 校验为主

7. 默认库没有业务样例数据
- 所有统计接口返回值是否有意义，取决于你是否先跑完业务链路

## 7. Python 开发最需要知道的结论

如果 Python 团队只看一页，需要记住下面几件事：

1. 当前 Java 后端已经把“病例到分析”的主链路搭好了。
2. Java 侧创建分析任务后，会发布 `analysis.requested` 事件。
3. Python 侧要么消费 RabbitMQ，要么让 Java 切到 `logging` 模式做假联调。
4. Python 回调必须调用 `/api/v1/internal/ai/callbacks/analysis-result`。
5. 回调签名算法是 `Base64URL(HMAC_SHA256(secret, timestamp + "." + rawBody))`。
6. 回调成功后，Java 会负责写分析摘要、风险评估、可视化资产，并驱动病例状态流转。
7. 报告和随访都依赖分析回写结果，Python 不需要直接改数据库。
