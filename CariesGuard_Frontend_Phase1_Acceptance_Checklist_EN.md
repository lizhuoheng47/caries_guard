# CariesGuard Frontend Phase 1 Acceptance Checklist

This checklist is intended to evaluate the first-phase frontend delivery for CariesGuard. It is designed for reviewing output generated by Gemini or by a frontend team.

This checklist is based on three fixed constraints:

1. **Architecture and API boundary**: the frontend must call only Java BFF APIs, exposed through `/api/v1/**`. Java is the business entry point and state authority. The frontend must not call Python services directly.  
2. **UI design specification**: the product must look like a **medical AI command console**, not a generic admin dashboard. The visual style is deep-space dark theme, holographic HUD, glassmorphism, fine glow effects, and strong semantic color usage.  
3. **Implementation priority**: in Phase 1, only the first batch of five pages is required. The second batch depends on additional Java BFF view APIs and must not be treated as mandatory for this phase.

---

## 1. Acceptance Scope

### Required in this phase
- Login page `/login`
- Analysis task queue `/analysis`
- Analysis detail page `/analysis/:taskId`
- RAG console `/rag`
- Knowledge repository `/knowledge`

### Not required in this phase
- AI dashboard `/dashboard/ai`
- Review workbench `/review/:taskId`
- Case and image portal `/cases`

These three belong to Phase 2 and require additional Java BFF view APIs first.

---

## 2. Overall Pass Criteria

### Pass
The delivery is considered **passed** only if all of the following are true:
- Architecture boundary is correct
- All first-batch pages are implemented and runnable
- Visual style is highly consistent with the UI design specification
- The product does not look like a generic admin template
- There is no direct frontend-to-Python integration
- There is no direct page-level parsing of Python internal structures
- Phase 2 pages are not falsely presented as fully completed

### Partial Pass
The delivery is considered **partially passed** if one or more of the following applies:
- Pages run, but the visual design is noticeably off-spec
- Architecture is broadly correct, but data flow layering is messy
- One or two first-batch pages are still static shells
- Mock mode or mock/real switching is incomplete

### Fail
The delivery is considered **failed** if one or more of the following applies:
- The product looks like a standard admin dashboard
- Default Element / Ant Design styling is used as the visual base
- The frontend calls Python `/ai/v1/**` directly
- Phase 2 pages are falsely marked as completed
- Pages directly consume raw Python structures instead of Java view objects
- The analysis detail page lacks the core HUD scanner experience and closed-loop timeline

## 11. Recommended Review Process

When reviewing Gemini output or frontend delivery, use this sequence:

### Step 1 — Check project structure
Verify that the project contains:
- `api`
- `dto`
- `vm`
- `adapters`
- `mock`
- `views`
- shared components

### Step 2 — Check the first five pages
If the five first-batch pages are not all present, reject the delivery.

### Step 3 — Inspect Page 02 first
If the analysis detail page is weak, the whole frontend is weak. This is the visual centerpiece of the system.

### Step 4 — Check red-line failures
Especially:
- direct Python calls
- admin-template look
- missing citations
- missing timeline
- missing mock support

---

## 12. Final Review Position

This checklist is intended for **Phase 1 frontend acceptance**, not final project closure.

A delivery that passes this checklist is considered:
- structurally correct
- visually aligned enough to continue
- safe to move into the next implementation stage

A delivery that fails this checklist should not be treated as acceptable just because "the pages can run".
